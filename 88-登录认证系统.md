# 登录认证系统架构设计 — 从协议到工程实践

---

## 第一章：认证与授权基础

### 1.1 认证（Authentication）vs 授权（Authorization）

先用一个生活类比开场：

> 你去公司上班。刷工卡进大门——这是**认证**（Authentication），门禁系统确认"你是张三"。  
> 进了大门后，你能进哪些会议室、能访问哪些机密文件——这是**授权**（Authorization），取决于你的角色和权限。

两者的核心区别：

```
+----------------+----------------------------------+----------------------------------+
|     维度       |   认证 (Authentication, AuthN)    |   授权 (Authorization, AuthZ)     |
+----------------+----------------------------------+----------------------------------+
|   回答的问题    |   "你是谁？"                      |   "你能做什么？"                   |
+----------------+----------------------------------+----------------------------------+
|   发生时机      |   先发生                          |   后发生（认证通过才谈授权）        |
+----------------+----------------------------------+----------------------------------+
|   依据         |   凭据（密码/指纹/证书）           |   策略（角色/权限/规则）            |
+----------------+----------------------------------+----------------------------------+
|   结果         |   身份确认（是/否）                |   访问决策（允许/拒绝）             |
+----------------+----------------------------------+----------------------------------+
|   举例         |   输入用户名密码登录               |   普通员工无法查看薪资管理页面       |
+----------------+----------------------------------+----------------------------------+
|   类比         |   出示身份证                       |   凭VIP票进VIP包间                |
+----------------+----------------------------------+----------------------------------+
|   常见技术      |   SSO, OIDC, LDAP, MFA           |   RBAC, ABAC, ACL, OAuth 2.0     |
+----------------+----------------------------------+----------------------------------+
```

**一个容易犯的错误**：很多人把 OAuth 2.0 当作"登录协议"。实际上 OAuth 2.0 是**授权框架**，它解决的是"我授权你代我访问我的数据"，而不是"你是谁"。要回答"你是谁"，需要在 OAuth 2.0 之上加一层身份层，这就是 OpenID Connect（后面会详细讲）。

### 1.2 为什么需要 SSO（单点登录）

假设一家公司有 5 个内部系统：OA、CRM、财务、HR、项目管理。如果没有 SSO，每个系统各管各的登录，会带来**4 大问题**：

**问题一：用户体验差**

```
张三的一天：
  08:30  登录 OA 系统         → 输入用户名密码
  08:35  打开 CRM 查客户信息   → 又输入用户名密码
  09:00  提交报销到财务系统     → 再输入用户名密码
  09:30  去 HR 系统请假        → 还要输入用户名密码
  10:00  打开项目管理看需求     → 第5次输入用户名密码...
  
  张三内心：我是不是在做密码输入练习？
```

**问题二：安全风险高**

- 5 个系统 = 5 个地方存储密码 = 5 倍被拖库的风险
- 用户为了方便，很可能 5 个系统都用同一个密码
- 一个系统被攻破 → 所有系统全部沦陷

**问题三：维护成本大**

- 每个系统都要自己实现登录逻辑、密码找回、账号管理
- 员工入职要在 5 个系统分别建账号，离职要分别注销
- 密码策略不统一（有的要求 8 位，有的要求特殊字符...）

**问题四：审计困难**

- 想知道"张三今天登录了哪些系统" → 要去 5 个系统分别查日志
- 安全事件溯源极其困难

**SSO 的目标**：用户只需登录一次，即可访问所有互信的系统。

### 1.3 SSO 三种模式演进

#### 模式一：传统各自登录

```
  ┌──────────┐     ┌──────────┐     ┌──────────┐
  │  系统 A   │     │  系统 B   │     │  系统 C   │
  │          │     │          │     │          │
  │ 用户表    │     │ 用户表    │     │ 用户表    │
  │ 登录逻辑  │     │ 登录逻辑  │     │ 登录逻辑  │
  │ Session  │     │ Session  │     │ Session  │
  └──────────┘     └──────────┘     └──────────┘
       ▲                ▲                ▲
       │                │                │
       │   每个系统独立    │   用户需要      │
       │   管理认证       │   多次登录      │
       └────────────────┴────────────────┘
                        │
                     用户张三
```

**特点**：每个系统有自己的用户表和登录逻辑，用户需要在每个系统分别登录。

---

#### 模式二：同域 SSO（共享根域 Cookie）

```
  所有系统都在 *.company.com 下：
  
    a.company.com    b.company.com    c.company.com
         │                │                │
         ▼                ▼                ▼
    ┌─────────────────────────────────────────┐
    │    共享根域 Cookie (.company.com)         │
    │    Set-Cookie: token=xxx;               │
    │    Domain=.company.com                  │
    └─────────────────────────────────────────┘
    
  原理：Cookie 的 Domain 设置为 .company.com
       则所有 *.company.com 的子域都能读到这个 Cookie
```

**优点**：实现简单，登录一次，同域下的所有系统自动带上 Cookie。

**致命局限**：只能在**同一个根域名**下工作。如果系统分布在 `company.com` 和 `partner.com` 两个域名下，Cookie 无法共享，方案失效。

---

#### 模式三：跨域 SSO（独立认证中心）— 主流方案

```
                    ┌─────────────────────┐
                    │   SSO 认证中心        │
                    │   (sso.company.com)  │
                    │                     │
                    │  ┌───────────────┐  │
                    │  │ 统一用户数据库  │  │
                    │  │ 统一登录页面    │  │
                    │  │ 统一会话管理    │  │
                    │  └───────────────┘  │
                    └──────┬──────┬───────┘
                     ┌─────┘      └─────┐
                     ▼                  ▼
              ┌─────────────┐    ┌─────────────┐
              │  a.foo.com  │    │  b.bar.com  │
              │  (系统 A)    │    │  (系统 B)    │
              └─────────────┘    └─────────────┘

  流程：
  1. 用户访问 a.foo.com → 未登录 → 跳转 sso.company.com
  2. 用户在 SSO 认证中心登录 → SSO 种下自己域的 Cookie (TGC)
  3. SSO 生成授权码 → 重定向回 a.foo.com
  4. a.foo.com 用授权码换 Token → 登录成功
  5. 用户再访问 b.bar.com → 未登录 → 跳转 sso.company.com
  6. SSO 发现已有 TGC Cookie → 直接生成授权码 → 重定向回 b.bar.com
  7. b.bar.com 换 Token → 自动登录（用户无感知）
```

**核心思想**：把登录这件事从各个业务系统中"提取"出来，交给一个独立的认证中心统一管理。各业务系统只需信任认证中心即可。

**这就是现代 SSO 的主流架构，后面章节的 CAS、OIDC、mtcas 都是这个模式的具体实现。**

## 第二章：核心协议基础

### 2.1 OAuth 2.0

#### 2.1.1 OAuth 2.0 是什么

OAuth 2.0（RFC 6749）是一个**授权框架**，注意不是认证协议。

生活类比：

> 你住酒店，需要让清洁工进你的房间打扫。你不会把房卡给清洁工（那等于给了所有权限），  
> 而是通知前台："给清洁工发一张临时工作卡，只能在 10:00-12:00 进入我的房间，不能打开保险箱。"  
>  
> 这就是 OAuth 2.0 的核心思想：**不直接给密码，而是发放受限的临时令牌。**

#### 2.1.2 四大角色

```
  ┌─────────────────┐                    ┌─────────────────┐
  │  Resource Owner  │                    │     Client      │
  │    (资源所有者)   │                    │     (应用)       │
  │                 │                    │                 │
  │  就是用户本人     │───── 授权给 ──────▶│  第三方应用       │
  │  比如：张三       │                    │  比如：某记账App  │
  └─────────────────┘                    └────────┬────────┘
                                                  │
                                          请求Token │  用Token访问资源
                                                  │
  ┌─────────────────┐                    ┌────────▼────────┐
  │ Resource Server  │◀── 携带Token访问 ──│  Authorization   │
  │   (资源服务器)    │                    │    Server        │
  │                 │                    │  (授权服务器)      │
  │  存放用户数据     │                    │                 │
  │  比如：微信API    │                    │  颁发和管理Token  │
  │                 │                    │  比如：微信OAuth  │
  └─────────────────┘                    └─────────────────┘
```

用一个具体场景串联四个角色：

> **场景**：某记账 App 想读取你的微信支付账单来自动记账。
> 
> - **Resource Owner**：你（张三），微信支付账单是你的数据
> - **Client**：记账 App，想访问你的数据
> - **Authorization Server**：微信的 OAuth 服务，负责弹出授权页面、发放 Token
> - **Resource Server**：微信的支付 API，存储你的账单数据

#### 2.1.3 四种授权类型

**1. Authorization Code（授权码模式）— 最安全最常用**

```
适用场景：有后端服务器的 Web 应用
安全等级：★★★★★

流程：用户 → 授权页面 → 获得授权码(code) → 后端用 code 换 Token
关键：Token 的交换在后端服务器完成，不会暴露给浏览器
```

举例：你在某网站点击"用微信登录"，跳转微信授权页，点击同意后跳回网站并附带一个 code，网站后端拿这个 code 去微信换取 Token。

**2. Implicit（隐式模式）— 已废弃**

```
适用场景：早期的纯前端 SPA 应用（OAuth 2.0 发布时 CORS 还不普及）
安全等级：★★☆☆☆

流程：用户 → 授权页面 → 直接在 URL fragment 中返回 Token
问题：Token 直接暴露在浏览器中，容易被截获

  ⚠️  OAuth 2.1 已正式废弃此模式
  ⚠️  现代 SPA 应使用 Authorization Code + PKCE
```

**3. Client Credentials（客户端凭证模式）— 机器对机器**

```
适用场景：没有用户参与的服务端到服务端调用（M2M, Machine-to-Machine）
安全等级：★★★★☆

流程：应用直接用 client_id + client_secret 换 Token
特点：没有用户参与，Token 代表的是应用本身而非某个用户
```

举例：你的后端定时任务每天凌晨调用天气 API 获取数据。这里没有"用户"的概念，是服务器自己去调接口，用 Client Credentials 即可。

**4. Password（资源所有者密码凭证模式）— 不推荐**

```
适用场景：高度信任的第一方应用（如公司内部 App 对接自己的认证服务）
安全等级：★★☆☆☆

流程：用户把用户名密码直接给 Client → Client 拿去换 Token
问题：Client 能拿到用户的明文密码，违背了 OAuth 的初衷

  ⚠️  OAuth 2.1 已正式废弃此模式
  ⚠️  即使是第一方应用也建议用 Authorization Code + PKCE
```

#### 2.1.4 Token 类型

```
+-------------------+---------------------------+-----------------------------+
|      属性          |   Access Token            |   Refresh Token             |
+-------------------+---------------------------+-----------------------------+
|   作用             |   访问受保护资源（API）      |   获取新的 Access Token      |
+-------------------+---------------------------+-----------------------------+
|   有效期           |   短期（分钟~小时级）        |   长期（天~月级）             |
+-------------------+---------------------------+-----------------------------+
|   使用场景         |   放在 HTTP Header 中       |   只在后端与授权服务器间使用   |
|                   |   Authorization: Bearer xxx|                             |
+-------------------+---------------------------+-----------------------------+
|   暴露风险         |   相对较高（随请求传递）      |   低（只在换 Token 时传递）   |
+-------------------+---------------------------+-----------------------------+
|   被盗影响         |   影响有限（很快过期）        |   影响大（能持续获取新Token） |
+-------------------+---------------------------+-----------------------------+
|   举例             |   演唱会门票（当场有效）      |   年卡会员（可以反复领门票）   |
+-------------------+---------------------------+-----------------------------+
```

为什么要分成两种 Token？

> 类比：你去游乐园，年卡（Refresh Token）放在家里保险柜，每次去之前用年卡在 App 上领一张当日门票（Access Token）。  
> 即使当日门票丢了，小偷也只能玩一天；而年卡在家里安全着呢。

#### 2.1.5 Token vs 密码

```
+-------------------+---------------------------+-----------------------------+
|      属性          |   密码 (Password)          |   令牌 (Token)               |
+-------------------+---------------------------+-----------------------------+
|   有效期           |   长期（用户不改就一直有效）  |   短期（自动过期）             |
+-------------------+---------------------------+-----------------------------+
|   可撤销性         |   必须用户主动修改           |   服务端可随时撤销             |
+-------------------+---------------------------+-----------------------------+
|   权限范围         |   全部权限（等于用户本人）    |   可限定范围（scope）          |
+-------------------+---------------------------+-----------------------------+
|   使用场景         |   登录时一次性使用           |   每次 API 调用都携带          |
+-------------------+---------------------------+-----------------------------+
|   泄露影响         |   灾难级（全部权限被盗）     |   有限（权限受限+快速过期）     |
+-------------------+---------------------------+-----------------------------+
|   可审计性         |   只知道登录时间             |   每次使用都有记录             |
+-------------------+---------------------------+-----------------------------+
```

**一句话总结：密码是万能钥匙，令牌是临时通行证。**

#### 2.1.6 安全注意事项

**1. CSRF 攻击 — 用 state 参数防御**

```
攻击场景：
  1. 攻击者自己走完 OAuth 流程，拿到一个 code
  2. 构造一个链接：https://app.com/callback?code=攻击者的code
  3. 诱导受害者点击这个链接
  4. 受害者的账号绑定了攻击者的第三方账号！

防御：
  1. 应用在发起 OAuth 请求时生成随机 state 值，存在 Session 中
  2. OAuth 回调时检查返回的 state 是否与 Session 中一致
  3. 攻击者无法猜到受害者的 state 值，攻击失败
```

**2. Code 泄露 — 用 PKCE + 短有效期防御**

```
攻击场景：
  授权码(code)在 URL 中传递，可能被浏览器历史、Referer 头、日志等泄露

防御：
  - code 有效期极短（通常 10 秒）
  - code 只能使用一次
  - 使用 PKCE（下一节详讲）：即使 code 被截获，
    没有 code_verifier 也换不到 Token
```

**3. Token 存储 — SPA 不应存 localStorage**

```
  ❌ localStorage：任何 JS 代码都能读取，XSS 攻击一锅端
  ❌ sessionStorage：同上，只是关闭标签页会清除
  ⚠️  内存变量：安全但刷新页面就没了
  ✅ HttpOnly Cookie：JS 无法读取，相对安全
  ✅ BFF (Backend For Frontend)：Token 存后端，前端只有 Session Cookie
```

### 2.2 PKCE（Proof Key for Code Exchange）

#### 2.2.1 为什么需要 PKCE

在标准的 Authorization Code Flow 中，第 8 步用 code 换 Token 时需要提供 `client_secret`。但有两类客户端**无法安全保存 client_secret**：

```
  移动 App：
    - APK/IPA 可以反编译
    - 攻击者可以从应用包中提取 client_secret
    
  SPA（单页应用）：
    - 所有 JavaScript 代码在浏览器中完全暴露
    - F12 打开控制台就能看到源码
    - 任何"隐藏"方式都只是"藏起来"而非"保密"
```

**没有 client_secret，谁截获了 code 就能换到 Token。** PKCE 就是解决这个问题的。

#### 2.2.2 核心机制

PKCE 引入了两个概念：

```
  code_verifier（验证器）
    - 客户端生成的随机高熵字符串
    - 长度：43 ~ 128 个字符
    - 字符集：[A-Z] [a-z] [0-9] - . _ ~
    - 示例：dBjftJeZ4CVP-mB92K27uhbUJU1p1r_wW1gFWFOEjXk

  code_challenge（挑战码）
    - 由 code_verifier 经过哈希计算得来
    - 公式：code_challenge = BASE64URL( SHA256( code_verifier ) )
    - 示例：E9Melhoa2OwvFrEMTJguCHaoeK1t8URWbuGJSstw-cM
```

**关键安全属性：SHA256 是单向哈希函数，从 code_challenge 无法反推 code_verifier。**

#### 2.2.3 完整流程

```
  ┌──────────┐              ┌──────────────────┐
  │  Client   │              │ Authorization    │
  │ (App/SPA) │              │ Server           │
  └────┬──────┘              └────────┬─────────┘
       │                              │
       │  1. 生成 code_verifier        │
       │     (随机字符串)               │
       │                              │
       │  2. 计算 code_challenge       │
       │     = BASE64URL(SHA256(       │
       │       code_verifier))         │
       │                              │
       │  3. 发起授权请求               │
       │     ?code_challenge=E9Mel...  │
       │     &code_challenge_method    │
       │       =S256                   │
       │─────────────────────────────▶│
       │                              │
       │                              │  4. 服务端保存
       │                              │     code_challenge
       │                              │     (与 code 绑定)
       │                              │
       │  5. 返回 code                 │
       │◀─────────────────────────────│
       │                              │
       │  6. 用 code + code_verifier   │
       │     换 Token                  │
       │     (注意：发送的是原文，      │
       │      不是哈希值)              │
       │─────────────────────────────▶│
       │                              │
       │                              │  7. 服务端验证：
       │                              │  BASE64URL(SHA256(
       │                              │    code_verifier))
       │                              │  == 之前存的
       │                              │    code_challenge ?
       │                              │
       │  8. 验证通过，返回 Token       │
       │◀─────────────────────────────│
       │                              │
```

#### 2.2.4 为什么安全

```
  攻击者能截获到的：
    ✅ code_challenge （在授权请求 URL 中，公开的）
    ✅ code           （在回调 URL 中，可能被截获）
    
  攻击者需要但拿不到的：
    ❌ code_verifier  （只在客户端内存中，从未在网络上传输过
                        ——直到最后换 Token 时才发送）
    
  即使攻击者同时拿到了 code 和 code_challenge：
    - 换 Token 需要 code_verifier（原文）
    - 从 code_challenge 反推 code_verifier？ SHA256 不可逆，做不到
    - 攻击者卡在最后一步，无法完成攻击

  简单总结：服务端存哈希，客户端发送原文，哈希不可逆。
```

### 2.3 OpenID Connect (OIDC)

#### 2.3.1 OIDC = OAuth 2.0 + 身份层

```
  OAuth 2.0 能告诉你：
    "用户授权了这个应用访问他的照片和联系人"
    → 但不能告诉你这个用户到底是谁
  
  OIDC 在 OAuth 2.0 基础上增加了：
    "这个用户是张三，他的邮箱是 zhangsan@company.com，
     他的唯一标识是 user_12345"

  关系：
    ┌─────────────────────────────┐
    │         OIDC                │
    │  ┌───────────────────────┐  │
    │  │     OAuth 2.0         │  │
    │  │                       │  │
    │  │   授权框架             │  │
    │  │   (你能访问什么)       │  │
    │  └───────────────────────┘  │
    │  + 身份层 (你是谁)          │
    │  + ID Token (JWT 格式)      │
    │  + UserInfo Endpoint        │
    │  + 标准 Claims              │
    └─────────────────────────────┘
```

#### 2.3.2 ID Token — JWT 格式

OIDC 引入了一种新的 Token 类型：**ID Token**，采用 JWT（JSON Web Token）格式。

JWT 由三部分组成，用 `.` 分隔：

```
  eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9
  .
  eyJpc3MiOiJodHRwczovL3Nzby5jb21wYW55LmNvbSIsInN1YiI6InVzZXJfMTIzNDUiLC
  JhdWQiOiJteV9hcHBfY2xpZW50X2lkIiwiZXhwIjoxNzAwMDAwMDAwLCJpYXQiOjE2MDkz
  NTk2MDAsIm5vbmNlIjoiYWJjMTIzIiwibmFtZSI6IuW8oOS4iSIsImVtYWlsIjoiemhhbm
  dzYW5AY29tcGFueS5jb20ifQ
  .
  SIGNATURE_BYTES
  
  ┌──────────────────────────────────────────────────┐
  │  Header (头部) — Base64URL 编码                    │
  │  {                                               │
  │    "alg": "RS256",   // 签名算法                   │
  │    "typ": "JWT"      // Token 类型                 │
  │  }                                               │
  ├──────────────────────────────────────────────────┤
  │  Payload (载荷) — Base64URL 编码                   │
  │  {                                               │
  │    "iss": "https://sso.company.com",  // 发行者    │
  │    "sub": "user_12345",               // 用户ID    │
  │    "aud": "my_app_client_id",         // 受众      │
  │    "exp": 1700000000,                 // 过期时间   │
  │    "iat": 1609359600,                 // 签发时间   │
  │    "nonce": "abc123",                 // 防重放     │
  │    "name": "张三",                     // 姓名      │
  │    "email": "zhangsan@company.com"    // 邮箱      │
  │  }                                               │
  ├──────────────────────────────────────────────────┤
  │  Signature (签名)                                 │
  │  RS256(                                          │
  │    base64UrlEncode(header) + "." +               │
  │    base64UrlEncode(payload),                     │
  │    privateKey                                    │
  │  )                                               │
  └──────────────────────────────────────────────────┘
```

#### 2.3.3 标准 Claims（声明）

```
+-----------+------------------+------------------------------------------+
|   Claim   |   含义           |   示例值                                  |
+-----------+------------------+------------------------------------------+
|   iss     |   发行者          |   "https://sso.company.com"              |
|           |   (Issuer)       |   标识谁签发了这个 Token                   |
+-----------+------------------+------------------------------------------+
|   sub     |   用户唯一标识     |   "user_12345"                           |
|           |   (Subject)      |   在该 Issuer 下唯一                      |
+-----------+------------------+------------------------------------------+
|   aud     |   受众            |   "my_app_client_id"                     |
|           |   (Audience)     |   即 client_id，Token 是给谁用的           |
+-----------+------------------+------------------------------------------+
|   exp     |   过期时间         |   1700000000 (Unix 时间戳)               |
|           |   (Expiration)   |   Token 在此时间后无效                    |
+-----------+------------------+------------------------------------------+
|   iat     |   签发时间         |   1609359600 (Unix 时间戳)               |
|           |   (Issued At)    |   Token 签发的时间                        |
+-----------+------------------+------------------------------------------+
|   nonce   |   防重放随机数     |   "abc123"                               |
|           |                  |   客户端生成，防止 Token 被重放攻击          |
+-----------+------------------+------------------------------------------+
|   name    |   用户姓名         |   "张三"                                 |
+-----------+------------------+------------------------------------------+
|   email   |   用户邮箱         |   "zhangsan@company.com"                 |
+-----------+------------------+------------------------------------------+
```

#### 2.3.4 JWT vs 不透明 Token

```
+--------------+----------------------------+----------------------------+
|    维度       |   JWT (自包含Token)         |   不透明Token (Opaque)      |
+--------------+----------------------------+----------------------------+
|  验证方式     |  本地验证（用公钥验签名）     |  远程调用（请求授权服务器）   |
+--------------+----------------------------+----------------------------+
|  性能        |  快（无网络调用）             |  慢（每次都要调接口）        |
+--------------+----------------------------+----------------------------+
|  撤销        |  困难（签发后无法主动失效，     |  简单（服务端直接删除即可）   |
|              |  除非引入黑名单机制）         |                            |
+--------------+----------------------------+----------------------------+
|  大小        |  较大（几百字节到几KB）        |  小（通常几十字节的随机串）   |
+--------------+----------------------------+----------------------------+
|  自包含      |  是（包含用户信息和权限）      |  否（只是一个引用/指针）     |
+--------------+----------------------------+----------------------------+
|  适用场景     |  微服务间调用、无状态API      |  需要即时撤销的敏感场景      |
+--------------+----------------------------+----------------------------+
|  服务端存储   |  不需要                      |  需要（存储Token元数据）     |
+--------------+----------------------------+----------------------------+
```

#### 2.3.5 混合策略：取长补短

实际工程中，通常采用混合策略：

```
  ┌─────────────────────────────────────────────────────┐
  │                   混合策略                           │
  │                                                     │
  │   1. 短生命周期 JWT（如 15 分钟）作为 Access Token    │
  │      → 大部分请求用 JWT 本地验证，性能好              │
  │      → 即使无法即时撤销，15分钟后自动失效              │
  │                                                     │
  │   2. Refresh Token 轮转                             │
  │      → 每次刷新后旧 Refresh Token 立即失效            │
  │      → 如果旧 Token 被使用，说明可能被盗，             │
  │        立即撤销该用户所有 Token                       │
  │                                                     │
  │   3. 关键操作二次验证                                 │
  │      → 转账、修改密码等敏感操作                       │
  │      → 即使 Token 有效，也要输入密码或短信验证          │
  └─────────────────────────────────────────────────────┘
```

4. 提供了标准的 UserInfo Endpoint

### 2.4 CIBA（Client-Initiated Backchannel Authentication）

#### 2.4.1 与传统 OIDC 的区别

传统 OIDC 流程需要浏览器跳转：用户 → 浏览器 → 授权页面 → 回调。但很多场景没有浏览器：

```
  传统 OIDC：
    用户在浏览器上操作，一切通过前端页面跳转完成
    
  CIBA：
    去掉浏览器跳转，认证在"后台通道"(Backchannel) 进行
    用户在另一个设备（如手机）上完成认证
```

#### 2.4.2 适用场景

- **呼叫中心**：客服在电脑上操作，客户在手机上确认身份
- **IoT 设备**：智能音箱没有屏幕和浏览器，用户在手机 App 上确认
- **CLI 工具**：命令行终端没有浏览器，用户在手机上确认登录
- **无头环境**：服务器、后台任务等没有 UI 的场景

#### 2.4.3 双设备模型

```
  ┌────────────────────────┐       ┌────────────────────────┐
  │  Consumption Device     │       │  Authentication Device  │
  │  (消费设备)             │       │  (认证设备)              │
  │                        │       │                        │
  │  发起业务请求的设备       │       │  完成身份认证的设备       │
  │                        │       │                        │
  │  例如：                 │       │  例如：                 │
  │  - 收银台终端            │       │  - 用户的手机 App       │
  │  - CLI 终端             │       │  - 推送通知             │
  │  - IoT 设备             │       │  - 短信验证             │
  │  - 呼叫中心电脑          │       │  - MOA 应用             │
  │                        │       │                        │
  │  特点：                 │       │  特点：                 │
  │  没有浏览器/无法跳转      │       │  有用户交互能力          │
  │  但需要获取用户授权       │       │  能展示确认页面          │
  └───────────┬────────────┘       └────────────┬───────────┘
              │                                  │
              │      通过后台通道协调               │
              └──────────┬───────────────────────┘
                         │
                         ▼
              ┌─────────────────────┐
              │  Authorization       │
              │  Server              │
              │  (认证服务器)         │
              │                     │
              │  协调两个设备之间     │
              │  的认证流程           │
              └─────────────────────┘
```

> 类比：你在银行柜台（消费设备）办理大额转账，银行给你手机（认证设备）发了一条短信验证码，你输入验证码后柜台才继续操作。两个设备各司其职。

#### 2.4.4 三种模式

```
+-----------+------------------+------------------------------------------+
|   模式     |   名称           |   说明                                    |
+-----------+------------------+------------------------------------------+
|   Poll    |   轮询模式        |   Client 定期查询认证结果                  |
|           |   (最通用)        |   优点：实现简单，无需 Client 暴露端点      |
|           |                  |   缺点：存在轮询间隔，不够实时              |
+-----------+------------------+------------------------------------------+
|   Ping    |   回调通知模式     |   认证完成后 AS 主动回调 Client 的端点      |
|           |                  |   优点：实时性好                           |
|           |                  |   缺点：Client 需暴露回调 URL              |
+-----------+------------------+------------------------------------------+
|   Push    |   直接推送模式     |   认证完成后 AS 直接推送 Token 到 Client    |
|           |                  |   优点：最简单直接                         |
|           |                  |   缺点：安全性要求更高                     |
+-----------+------------------+------------------------------------------+
```

## 第三章：SSO 核心实现

### 3.1 CAS：业界经典 SSO 方案

#### 3.1.1 CAS 是什么

CAS（Central Authentication Service，中心认证服务）是耶鲁大学开源的单点登录项目，是 SSO 领域最经典的实现方案之一。

```
  CAS 架构三要素：
  
  ┌──────────────────┐
  │   CAS-Server      │  ← 认证中心，负责用户认证、签发票据
  │  (认证服务器)      │
  └────────┬─────────┘
           │
     ┌─────┴──────┐
     ▼            ▼
  ┌──────────┐  ┌──────────┐
  │CAS-Client│  │CAS-Client│  ← 嵌入在各业务系统中的客户端库
  │(过滤器)   │  │(过滤器)   │     负责拦截请求、验证票据
  ├──────────┤  ├──────────┤
  │ Biz-App  │  │ Biz-App  │  ← 业务应用本身
  │ (业务A)   │  │ (业务B)   │
  └──────────┘  └──────────┘
```

#### 3.1.2 核心概念：三种票据

```
+-----------+-----------------------------+-------------------------------------+
|   票据     |   全称                       |   说明                               |
+-----------+-----------------------------+-------------------------------------+
|   TGT     |   Ticket Granting Ticket     |   登录票据，存在 CAS-Server 端        |
|           |   (票据授权票据)              |   代表"用户已在 SSO 中心登录"          |
|           |                             |   类比：你的身份信息存在公安系统里       |
+-----------+-----------------------------+-------------------------------------+
|   TGC     |   Ticket Granting Cookie     |   包含 TGT ID 的 Cookie              |
|           |   (票据授权Cookie)           |   存在浏览器中，域名为 CAS-Server      |
|           |                             |   有效期 14 天                        |
|           |                             |   类比：你的身份证（随身携带的凭证）      |
+-----------+-----------------------------+-------------------------------------+
|   ST      |   Service Ticket             |   访问某个业务系统的一次性票据           |
|           |   (服务票据)                  |   由 TGT 签发，使用后即失效             |
|           |                             |   类比：用身份证买的火车票（一次性的）     |
+-----------+-----------------------------+-------------------------------------+
```

**TGT 与 ST 的关系**：

```
  TGT 是"根"，ST 是"叶"
  
  用户登录后，CAS-Server 创建 TGT
  每次用户访问一个新的业务系统时，CAS-Server 用 TGT 签发一张新的 ST
  
  TGT ──签发──▶ ST_for_系统A (用完即毁)
       ──签发──▶ ST_for_系统B (用完即毁)
       ──签发──▶ ST_for_系统C (用完即毁)
       
  好比你的身份证（TGT）可以反复用来买火车票（ST），
  但每张火车票只能用一次。
```

#### 3.1.3 CAS 标准流程

**场景一：首次登录完整流程**

```
  ┌──────┐        ┌──────────┐       ┌──────────────┐
  │ 用户  │        │  系统 A   │       │  CAS-Server  │
  │      │        │(CAS-Client)│      │  (认证中心)   │
  └──┬───┘        └────┬──────┘       └──────┬───────┘
     │                 │                     │
     │  1. 访问系统 A   │                     │
     │────────────────▶│                     │
     │                 │                     │
     │                 │  2. 检查：没有有效    │
     │                 │     Session          │
     │                 │                     │
     │  3. 302 重定向到 CAS-Server            │
     │  https://cas.company.com/login         │
     │  ?service=https://a.company.com        │
     │◀────────────────│                     │
     │                 │                     │
     │  4. 浏览器跳转到 CAS-Server             │
     │  (此时浏览器携带 CAS 域的 Cookie)        │
     │────────────────────────────────────────▶
     │                 │                     │
     │                 │              5. 检查 TGC Cookie
     │                 │                 → 不存在
     │                 │                 → 展示登录页面
     │                 │                     │
     │  6. 展示登录页面  │                     │
     │◀────────────────────────────────────────
     │                 │                     │
     │  7. 输入用户名密码 │                     │
     │────────────────────────────────────────▶
     │                 │                     │
     │                 │              8. 验证成功
     │                 │                 → 创建 TGT
     │                 │                 → 生成 ST
     │                 │                 → 设置 TGC Cookie
     │                 │                     │
     │  9. 302 重定向回系统 A                  │
     │  https://a.company.com?ticket=ST-xxx   │
     │  Set-Cookie: TGC=TGT-yyy;             │
     │    Domain=cas.company.com; HttpOnly    │
     │◀────────────────────────────────────────
     │                 │                     │
     │  10. 携带 ST 访问系统 A                 │
     │────────────────▶│                     │
     │                 │                     │
     │                 │  11. 后台验证 ST      │
     │                 │─────────────────────▶
     │                 │                     │
     │                 │  12. ST 有效，返回用户信息
     │                 │◀─────────────────────
     │                 │                     │
     │                 │  13. 创建本地 Session │
     │                 │     ST 标记为已使用   │
     │                 │                     │
     │  14. 登录成功     │                     │
     │◀────────────────│                     │
     │                 │                     │
```

**场景二：访问系统 B — 免登录**

```
  ┌──────┐        ┌──────────┐       ┌──────────────┐
  │ 用户  │        │  系统 B   │       │  CAS-Server  │
  │      │        │(CAS-Client)│      │  (认证中心)   │
  └──┬───┘        └────┬──────┘       └──────┬───────┘
     │                 │                     │
     │  1. 访问系统 B   │                     │
     │────────────────▶│                     │
     │                 │                     │
     │                 │  2. 检查：没有有效    │
     │                 │     Session          │
     │                 │                     │
     │  3. 302 重定向到 CAS-Server            │
     │◀────────────────│                     │
     │                 │                     │
     │  4. 浏览器跳转到 CAS-Server             │
     │  (此时浏览器自动携带 TGC Cookie！)       │
     │────────────────────────────────────────▶
     │                 │                     │
     │                 │              5. 检查 TGC Cookie
     │                 │                 → TGC 有效！
     │                 │                 → 找到对应 TGT
     │                 │                 → 直接签发新 ST
     │                 │                 → 无需再次登录 ⭐
     │                 │                     │
     │  6. 302 重定向回系统 B (带新的 ST)       │
     │◀────────────────────────────────────────
     │                 │                     │
     │  7. 携带 ST 访问系统 B                  │
     │────────────────▶│                     │
     │                 │                     │
     │                 │  8. 验证 ST           │
     │                 │─────────────────────▶
     │                 │                     │
     │                 │  9. ST 有效           │
     │                 │◀─────────────────────
     │                 │                     │
     │  10. 登录成功    │                     │
     │  (用户全程无感知) │                     │
     │◀────────────────│                     │
     │                 │                     │
```

**关键点**：用户访问系统 B 时全程无需输入密码。浏览器跳转到 CAS-Server 时自动带上了之前种下的 TGC Cookie，CAS-Server 验证 TGC 有效后直接签发新的 ST 并跳转回来。

#### 3.1.4 CAS 代理模式

有时候，系统 A 需要代替用户去访问系统 B 的数据（典型场景：前端门户聚合多个后端系统的数据）。CAS 引入了**代理模式**来解决这个问题：

```
  PGT (Proxy Granting Ticket) — 代理授权票据
    → 系统 A 拿 ST 向 CAS-Server 换取 PGT
    → PGT 代表"系统 A 有权代理用户访问其他系统"
    
  PT  (Proxy Ticket) — 代理票据
    → 系统 A 用 PGT 向 CAS-Server 申请访问系统 B 的 PT
    → 系统 A 拿 PT 去访问系统 B
    
  整个链路：
    用户 → 系统A(用ST认证) → 系统A用PGT申请PT → 系统A拿PT访问系统B
```

这种代理模式虽然解决了问题，但链路较复杂。后面会看到 mtcas 如何用更简洁的方式（Token Exchange）解决同样的问题。

### 3.2 mtcas 实现

#### 3.2.1 mtcas 是什么

mtcas 是基于 OIDC 标准协议扩展的统一 SSO 系统。它不是另起炉灶，而是站在标准协议的基础上，结合实际业务需求进行了工程化扩展。

#### 3.2.2 核心特点

```
  ┌──────────────────────────────────────────────────────┐
  │  mtcas 特点                                          │
  │                                                      │
  │  1. 基于 OIDC Authorization Code Flow + PKCE         │
  │     → 标准协议，安全性有保障                            │
  │                                                      │
  │  2. 支持 CIBA（无浏览器场景）                           │
  │     → CLI 工具、IoT 设备也能认证                       │
  │                                                      │
  │  3. 支持 Token Exchange（跨系统凭证转换）               │
  │     → 微服务间调用不再需要复杂的代理模式                  │
  │                                                      │
  │  4. 以 misId（员工MIS号）作为用户唯一标识               │
  │     → 统一身份标识，全司通用                            │
  │                                                      │
  │  5. 与 MOA 深度集成                                   │
  │     → 支持 MOA 无感登录（后面详述）                            │
  └──────────────────────────────────────────────────────┘
```

#### 3.2.3 与标准协议对应关系

```
+-------------------------------+--------------------------------------+
|   标准协议概念                  |   mtcas 对应实现                      |
+-------------------------------+--------------------------------------+
|   Authorization Server        |   mtcas（SSO 认证中心）               |
+-------------------------------+--------------------------------------+
|   OIDC Auth Code Flow         |   mtcas OIDC 登录流程                 |
+-------------------------------+--------------------------------------+
|   CIBA Poll Mode              |   mtcas bc-authorize 接口             |
+-------------------------------+--------------------------------------+
|   Token Exchange (RFC 8693)   |   /websso/auth/token-exchange 接口    |
+-------------------------------+--------------------------------------+
|   login_hint                  |   misId（员工 MIS 号）                |
+-------------------------------+--------------------------------------+
|   client_id                   |   业务系统在开放平台注册的 ID           |
+-------------------------------+--------------------------------------+
```

#### 3.2.4 核心概念详解

**1. clientId — 应用标识**

```
  什么是 clientId：
    - 每个接入 SSO 的业务系统都有一个唯一的 clientId
    - 由开放平台申请颁发
    - 永久有效
    
  类比：
    clientId 就像公司给每个部门发的"部门编号"
    人事部是 HR001，财务部是 FIN002
    这个编号一旦分配就不会变
    
  用途：
    - 标识"是哪个应用在请求认证"
    - 所有 SSO 相关请求都要带 clientId
    - code、ssoid 都与 clientId 绑定
```

**2. code — 授权码**

```
  什么是 code：
    - 用户在 SSO 登录成功后，SSO 生成并返回给业务系统的临时凭证
    - 有效期：10 秒
    - 一次性使用（用过即失效）
    - 与 clientId 和 redirect_uri 一一对应
    
  类比：
    code 就像银行取号机吐出来的号码牌
    - 有效期极短（过号不候）
    - 只能用一次（叫到号就要销毁）
    - 和你本人绑定（别人不能用你的号）
    
  为什么这么短？
    code 在 URL 中传递（https://app.com/callback?code=xxx），
    可能被浏览器历史、日志等记录。
    有效期越短，被滥用的窗口就越小。
```

**3. TGC — Ticket Granting Cookie**

```
  什么是 TGC：
    - 存在 SSO 登录中心域名下的 Cookie
    - 有效期：14 天
    - 是实现"单点登录"的核心——浏览器访问 SSO 时自动携带
    
  TGC 包含两个 Cookie：
  
  ┌─────────────────────────────────────────────┐
  │  TGCN：兼容老版 SSO 的 Cookie                │
  │  TGCX：新版 SSO 的 Cookie                    │
  │                                             │
  │  有任意一个有效即可完成单点登录                  │
  │  (向后兼容策略)                               │
  └─────────────────────────────────────────────┘
  
  失效条件：
    - 用户主动登出（点击"退出登录"）
    - 14 天内没有任何登录操作
    
  续期机制：
    - 任何一次单点登录都会刷新 TGC 的有效期
    - 也就是说，只要你 14 天内用过一次 SSO，TGC 就会续期
    - 频繁使用的用户基本永远不会过期
    
  类比：
    TGC 就像一张"月票"——14 天有效，每次刷卡都自动续期。
    如果你天天坐地铁（天天用 SSO），这张月票永远不会过期。
    但如果你出差两周没用，回来就要重新买一张（重新登录）。
```

**4. ssoid — 业务域名下的访问令牌**

```
  什么是 ssoid：
    - 本质上就是 accessToken
    - 存在业务域名下的 Cookie
    - Cookie 的 key 为 ${clientId}_ssoid
      例如：myapp_ssoid=eyJhbGciOiJSUzI1NiIs...
    - 有效期：3 天
    
  种植过程：
    sso-sdk 在回调业务域名的 /sso/callback 时自动种下 Cookie
    (无需业务系统手动处理)
    
  失效条件：
    - 用户主动登出
    - 3 天后自动过期
    
  自动续期机制：
    - 当 TGC 有效时，用户访问业务系统
    - 业务系统发现 ssoid 过期 → 重定向到 SSO
    - SSO 发现 TGC 有效 → 静默签发新 code → 回调业务系统
    - 业务系统用 code 换取新 ssoid → 重新种下 Cookie
    - 整个过程对用户透明（多次 302 跳转在瞬间完成）
    
  ssoid vs TGC 对比：
  
  ┌──────────────┬─────────────────────┬─────────────────────┐
  │              │   TGC               │   ssoid             │
  ├──────────────┼─────────────────────┼─────────────────────┤
  │  存储域名     │  SSO 登录中心域名    │  业务系统域名         │
  │  有效期       │  14 天              │  3 天               │
  │  作用        │  证明"已在SSO登录"   │  证明"已在本系统登录" │
  │  谁种的       │  SSO 服务器          │  sso-sdk 自动种      │
  │  谁验证       │  SSO 服务器          │  业务系统            │
  └──────────────┴─────────────────────┴─────────────────────┘
```

#### 3.2.5 mtcas 完整认证流程

```
  ┌──────┐      ┌──────────┐     ┌────────────┐     ┌──────────────┐
  │ 用户  │      │ 业务系统  │     │  sso-sdk   │     │   mtcas      │
  │      │      │  (前端)   │     │ (回调处理)  │     │  (SSO中心)    │
  └──┬───┘      └────┬─────┘     └─────┬──────┘     └──────┬───────┘
     │               │                 │                   │
     │  1. 访问业务系统│                 │                   │
     │───────────────▶│                 │                   │
     │               │                 │                   │
     │               │  2. 检查 Cookie   │                   │
     │               │  ${clientId}_ssoid                   │
     │               │  → 不存在或已过期  │                   │
     │               │                 │                   │
     │  3. 302 重定向到 mtcas                               │
     │  /authorize?client_id=xxx                            │
     │  &redirect_uri=https://biz.com/sso/callback          │
     │  &response_type=code                                 │
     │  &scope=openid&code_challenge=...                    │
     │◀───────────────│                 │                   │
     │               │                 │                   │
     │  4. 浏览器跳转到 mtcas                                │
     │  (自动携带 SSO 域名下的 TGC Cookie)                    │
     │──────────────────────────────────────────────────────▶│
     │               │                 │                   │
     │               │                 │          5. 验证 TGC
     │               │                 │                   │
     │               │                 │       ┌─── TGC 有效
     │               │                 │       │   → 直接生成 code
     │               │                 │       │   → 刷新 TGC (续期!)
     │               │                 │       │
     │               │                 │       └─── TGC 无效/不存在
     │               │                 │           → 展示登录页
     │               │                 │           → 用户登录后
     │               │                 │             创建新 TGC
     │               │                 │             生成 code
     │               │                 │                   │
     │  6. 302 重定向回业务系统                               │
     │  https://biz.com/sso/callback?code=xxx               │
     │  Set-Cookie: TGCX=...; TGCN=...                      │
     │  (续期 TGC)                                          │
     │◀──────────────────────────────────────────────────────│
     │               │                 │                   │
     │  7. 请求到达 /sso/callback       │                   │
     │───────────────────────────────▶│                   │
     │               │                 │                   │
     │               │                 │  8. 用 code +      │
     │               │                 │  code_verifier     │
     │               │                 │  换 Token          │
     │               │                 │──────────────────▶│
     │               │                 │                   │
     │               │                 │  9. 返回           │
     │               │                 │  access_token      │
     │               │                 │  (即 ssoid)        │
     │               │                 │◀──────────────────│
     │               │                 │                   │
     │               │          10. 种 Cookie               │
     │               │          ${clientId}_ssoid=...       │
     │               │          有效期 3 天                  │
     │               │                 │                   │
     │  11. 302 重定向到原始请求地址      │                   │
     │  Set-Cookie: ${clientId}_ssoid=...                   │
     │◀─────────────────────────────────│                   │
     │               │                 │                   │
     │  12. 携带 ssoid 访问业务系统      │                   │
     │───────────────▶│                 │                   │
     │               │                 │                   │
     │  13. 验证 ssoid│                 │                   │
     │  → 有效        │                 │                   │
     │  → 返回数据    │                 │                   │
     │◀───────────────│                 │                   │
     │               │                 │                   │
```

#### 3.2.6 mtcas vs CAS 差异对比

```
+-------------------+-----------------------------+-------------------------------+
|      维度          |   传统 CAS                  |   mtcas                        |
+-------------------+-----------------------------+-------------------------------+
|   协议基础         |   自有 CAS 协议              |   OAuth2 + OIDC 扩展            |
+-------------------+-----------------------------+-------------------------------+
|   用户凭证存储      |   服务端 Session             |   accessToken Cookie (ssoid)   |
+-------------------+-----------------------------+-------------------------------+
|   跨应用访问       |   代理模式 (PGT/PT)          |   直接携带 accessToken          |
|                   |   链路复杂                    |   或 Token Exchange            |
+-------------------+-----------------------------+-------------------------------+
|   适用客户端       |   主要 Web                   |   全场景 (Web/App/CLI/IoT)      |
+-------------------+-----------------------------+-------------------------------+
|   服务端到服务端    |   PGT/PT                    |   Token Exchange               |
+-------------------+-----------------------------+-------------------------------+
|   无浏览器场景     |   不原生支持                  |   CIBA 模式支持                 |
+-------------------+-----------------------------+-------------------------------+
|   协议标准化       |   CAS 私有协议               |   基于 RFC 标准                 |
+-------------------+-----------------------------+-------------------------------+
```

**为什么 mtcas 用 accessToken 而非纯 CAS 模式**：

```
  1. 适用场景更广泛
     - accessToken 不依赖 Cookie
     - 移动 App、CLI 工具、IoT 设备都能使用
     - 纯 CAS 模式强依赖浏览器 Cookie
     
  2. 跨应用访问更简单
     - 携带 accessToken 通过 HTTP Header (Authorization: Bearer xxx)
     - 不需要复杂的 PGT/PT 代理链路
     - 任何能发 HTTP 请求的地方都能用
```

### 3.3 跨应用信任机制

#### 3.3.1 什么是跨应用信任

每个业务系统接入 SSO 后都有自己的 clientId，每个系统的 ssoid 默认只在自己的系统内有效：

```
  默认情况：
  
  应用 A (clientId_A)
    → 登录后获得 ssoid_A
    → ssoid_A 只在应用 A 内有效
    
  应用 B (clientId_B)
    → 登录后获得 ssoid_B
    → ssoid_B 只在应用 B 内有效
    
  如果用 ssoid_A 去访问应用 B → ❌ 拒绝
  （因为 ssoid 中的 clientId 与应用 B 的 clientId 不匹配）
```

#### 3.3.2 配置信任关系

```
  配置"B 信任 A"后：
  
  ┌──────────┐    ssoid_A    ┌──────────┐
  │  应用 A   │─────────────▶│  应用 B   │
  │clientId_A │              │clientId_B │
  └──────────┘              └──────────┘
                               │
                               ▼
                         B 检查信任列表
                         → clientId_A 在列表中
                         → ✅ 接受 ssoid_A
```

#### 3.3.3 常见场景

```
  1. 前后端分离
     前端页面 (clientId_A) 和后端 API (clientId_B) 分属不同应用
     → 配置"后端信任前端"，前端的 ssoid 可以直接调后端 API
     
  2. 多系统聚合平台
     门户系统 (clientId_Portal) 聚合了 OA、CRM、ERP 的数据
     → 配置 OA/CRM/ERP 都信任 Portal
     → Portal 的 ssoid 可以访问所有子系统
     
  3. 服务端调用链
     服务 A → 服务 B → 服务 C
     → 逐级配置信任关系，或使用 Token Exchange
```

#### 3.3.4 安全考量

```
  信任关系是单向的："B 信任 A" 不代表 "A 信任 B"
  
  安全检验要点：
    1. 业务系统必须验证 ssoid 中的 clientId
    2. 只接受信任列表中的 clientId 对应的 ssoid
    3. 信任列表应尽可能短（最小权限原则）
    4. 定期审计信任关系，清理不再需要的配置
```

### 3.4 多因子认证（MFA）

多因子认证（Multi-Factor Authentication）通过要求用户提供多种身份验证方式，大幅提升安全性。即使一种因子被攻破，攻击者仍然需要突破其他因子。

#### 3.4.1 双因子登录认证机制

```
  认证流程：
  
  ┌─────────────────────────────────────────────────────┐
  │                                                     │
  │  第一因子（可选其一）                                  │
  │  ┌─────────────────┐  ┌────────────────────┐        │
  │  │  手机短信认证     │  │  账号密码认证       │        │
  │  │  (发送验证码)     │  │  (输入用户名密码)   │        │
  │  └────────┬────────┘  └─────────┬──────────┘        │
  │           │                     │                   │
  │           └──────────┬──────────┘                   │
  │                      │                              │
  │                      ▼                              │
  │             第一因子通过                              │
  │                      │                              │
  │                      ▼                              │
  │  第二因子（可选其一）                                  │
  │  ┌──────────┐  ┌──────────┐  ┌──────────────┐      │
  │  │ 可信设备   │  │ 手机短信  │  │  人脸识别     │      │
  │  │ 认证      │  │ 认证     │  │  认证         │      │
  │  └──────────┘  └──────────┘  └──────────────┘      │
  │                                                     │
  │  第二因子通过 → 登录成功 ✅                            │
  └─────────────────────────────────────────────────────┘
```

#### 3.4.2 因子组合规则

```
  规则一：手机短信作为第一因子时
    → 第二因子可选：可信设备 或 人脸识别
    → 注意：不能再选手机短信（同一因子不能重复使用）
    
  规则二：账号密码作为第一因子时
    → 第二因子可选：可信设备 或 手机短信 或 人脸识别
    → 三种方式都可以
    
  规则三：第二因子三种方式同时存在时
    → 用户可以自主选择使用哪种方式
    → 系统展示所有可用选项，用户挑一个
    
  规则四：第二因子指定仅人脸识别时
    → 用户只能使用人脸识别
    → 没有其他选择
```

#### 3.4.3 举例说明

```
  场景一：普通办公系统登录
    第一因子：账号密码（zhangsan / ********）
    第二因子：可信设备（公司电脑已标记为可信，自动通过）
    → 用户感知：输入密码，直接进去了
    
  场景二：高安全级别系统（如财务系统）
    第一因子：账号密码
    第二因子：仅人脸识别（系统管理员强制要求）
    → 用户感知：输入密码后，摄像头打开，刷脸确认
    
  场景三：移动端登录
    第一因子：手机短信验证码
    第二因子：人脸识别
    → 用户感知：输入验证码后，刷脸确认
    
  场景四：日常使用（可信设备已认证）
    第一因子：账号密码
    第二因子：可信设备认证（设备指纹匹配，自动通过）
    → 用户感知：输入密码后直接进入，几乎无感
```

### 3.5 Token Exchange (RFC 8693)

#### 3.5.1 什么是 Token Exchange

Token Exchange 是一种凭证转换机制——将一种 Token 换成另一种 Token。

```
  核心思想：
    "我手里有一个 Token_A，我需要用它换一个 Token_B"
    
  类比：
    你拿着人民币（Token_A）去银行换美元（Token_B）
    银行验证你的人民币是真的，然后按汇率给你美元
    你不需要重新"挣"美元，只需要"换"
```

#### 3.5.2 使用场景

```
  场景一：微服务间调用
  
    用户登录前端 → 获得 Token_前端
    前端调用 服务A → 需要 Token_服务A
    服务A 调用 服务B → 需要 Token_服务B
    
    ┌──────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
    │ 前端  │────▶│  服务 A   │────▶│  服务 B   │────▶│  服务 C   │
    │      │     │          │     │          │     │          │
    │Token │     │Token     │     │Token     │     │Token     │
    │前端   │     │服务A      │     │服务B      │     │服务C      │
    └──────┘     └──────────┘     └──────────┘     └──────────┘
    
    每一步都通过 Token Exchange 将上游 Token 换成下游 Token
    
  场景二：工具代理
  
    AI Agent 通过 CIBA 获得用户授权 Token
    → 需要用这个 Token 去访问目标系统
    → 通过 Token Exchange 换成目标系统的专属 Token
```

#### 3.5.3 Token Exchange 流程

```
  ┌──────────┐         ┌──────────────┐        ┌───────────────┐
  │ 服务 A    │         │   mtcas       │        │   服务 B       │
  │(持有      │         │ Token        │        │  (目标服务)     │
  │ Token_A)  │         │ Exchange     │        │               │
  └────┬──────┘         └──────┬───────┘        └───────┬───────┘
       │                       │                        │
       │  1. 请求 Token Exchange │                        │
       │  POST /websso/auth/    │                        │
       │       token-exchange   │                        │
       │  {                     │                        │
       │   grant_type:          │                        │
       │    "urn:ietf:params:   │                        │
       │     oauth:grant-type:  │                        │
       │     token-exchange",   │                        │
       │   subject_token:       │                        │
       │    Token_A,            │                        │
       │   subject_token_type:  │                        │
       │    "access_token",     │                        │
       │   audience:            │                        │
       │    "服务B的clientId"    │                        │
       │  }                     │                        │
       │───────────────────────▶│                        │
       │                       │                        │
       │                       │  2. 验证 Token_A         │
       │                       │     检查权限             │
       │                       │     签发 Token_B         │
       │                       │                        │
       │  3. 返回 Token_B       │                        │
       │◀───────────────────────│                        │
       │                       │                        │
       │  4. 用 Token_B 访问服务 B                        │
       │───────────────────────────────────────────────▶│
       │                       │                        │
       │  5. 返回数据            │                        │
       │◀───────────────────────────────────────────────│
       │                       │                        │
```

### 3.7 三路认证策略

在 CLI 工具和无浏览器场景中，需要一套完善的认证策略来覆盖各种情况。

#### 3.7.1 认证优先级链

```
  ┌──────────────────────────────────────────────────────────┐
  │              CLI / 工具场景认证优先级链                     │
  │                                                          │
  │   ①  MOA 无感登录（最高优先级）                             │
  │   │   探测 wss://localhost.moa.sankuai.com:16161/         │
  │   │   连通 → 直接换票，用户无感知                           │
  │   │                                                      │
  │   │   ❌ 不连通（MOA 未运行/未安装）                        │
  │   ▼                                                      │
  │   ②  OIDC 标准流程                                       │
  │   │   打开系统浏览器 → 跳转到 SSO 登录页面                   │
  │   │   用户登录后回调 → 获取 Token                          │
  │   │                                                      │
  │   │   ❌ 无法打开浏览器（SSH 远程/容器环境）                  │
  │   ▼                                                      │
  │   ③  Token Exchange                                      │
  │   │   如果已有其他系统的有效 Token                          │
  │   │   → 通过 Token Exchange 换取当前系统的 Token            │
  │   │                                                      │
  │   │   ❌ 没有任何有效 Token                                │
  │   ▼                                                      │
  │   ④  CIBA（最低优先级/最终兜底）                            │
  │       发送认证请求 → 用户手机 App 收到通知 → 用户确认         │
  │       → 轮询获取 Token                                    │
  │                                                          │
  └──────────────────────────────────────────────────────────┘
```

#### 3.7.2 每条路径实现思路

```
  路径一：MOA 无感
    1. 尝试建立 WSS 连接到 localhost:16161
    2. 设置短超时（如 2 秒），连不上立即降级
    3. 连通后调用换票 API
    4. 获取 code → 换 Token
    
  路径二：OIDC
    1. 启动本地 HTTP Server 监听随机端口（作为 redirect_uri）
    2. 用系统默认浏览器打开 SSO 授权 URL
    3. 用户在浏览器中完成登录
    4. SSO 回调本地 HTTP Server
    5. 从回调中提取 code → 换 Token
    
  路径三：Token Exchange
    1. 检查本地是否缓存了其他系统的有效 Token
    2. 调用 /websso/auth/token-exchange
    3. 用已有 Token 换取目标系统 Token
    
  路径四：CIBA
    1. 调用 /bc-authorize，提供 login_hint（misId）
    2. 获取 auth_req_id 和轮询间隔
    3. 终端显示"请在手机上确认..."
    4. 按间隔轮询 /token 接口
    5. 用户确认后获取 Token
```

#### 3.7.3 降级策略

```
  优先使用缓存：
    ┌─────────────────────────────────────────┐
    │  1. 检查本地缓存的 Token                  │
    │     → 有效？直接使用                      │
    │                                         │
    │  2. Token 过期但有 refresh_token？        │
    │     → 用 refresh_token 刷新              │
    │     → 刷新成功？使用新 Token               │
    │                                         │
    │  3. refresh_token 也过期了？              │
    │     → 走完整认证流程                      │
    │     → 按 MOA → OIDC → Exchange → CIBA   │
    │       的优先级尝试                        │
    └─────────────────────────────────────────┘
    
  核心原则：
    - 缓存优先 → refresh_token 刷新 → 完整认证
    - 每一步失败后自动降级到下一步
    - 全程无需人工干预选择认证方式
```

## 第四章：认证体系的授权层

认证（Authentication）解决的是"你是谁"的问题，而授权（Authorization）解决的是"你能做什么"的问题。认证通过后，还需要一套授权体系来控制用户的访问权限。

### 4.1 UAC（用户访问控制）

#### 4.1.1 UAC 是什么

UAC（User Access Control，用户访问控制）是通用应用权限管理框架，负责管理菜单、角色、资源、接口等权限，采用 RBAC（Role-Based Access Control，基于角色的访问控制）模型。

```
  UAC 管理的权限维度：
  
  ┌─────────────────────────────────────────────────┐
  │  UAC 权限管理                                    │
  │                                                 │
  │  菜单权限：用户能看到哪些菜单项                    │
  │  按钮权限：用户能操作哪些按钮                      │
  │  接口权限：用户能调用哪些 API 接口                  │
  │  数据权限：用户能看到哪些数据范围                   │
  │                                                 │
  │  RBAC 模型：                                     │
  │  用户 ──▶ 角色 ──▶ 权限                           │
  │                                                 │
  │  例如：                                          │
  │  张三 ──▶ 运营角色 ──▶ [查看报表, 导出数据]        │
  │  李四 ──▶ 管理员角色 ──▶ [查看报表, 导出数据,       │
  │                          修改配置, 用户管理]       │
  └─────────────────────────────────────────────────┘
```

#### 4.1.2 接入路径

```
  一个业务系统接入 UAC 的标准流程：
  
  1. 注册应用    → 在平台上注册你的业务系统
  2. 申请 SSO   → 接入 SSO 单点登录
  3. 申请 UAC   → 开通 UAC 权限管理
  4. 配置权限    → 在 UAC 管理后台配置菜单树、角色、权限
  5. 分配角色    → 管理员给员工分配角色
```

#### 4.1.3 两种授权身份

```
+----------------+----------------------------+------------------------------+
|     维度        |   用户身份授权              |   应用身份授权                 |
+----------------+----------------------------+------------------------------+
|   授权对象      |   某个员工（misId）         |   某个应用（clientId）         |
+----------------+----------------------------+------------------------------+
|   代表含义      |   "这个人"能做什么          |   "这个应用/机器"能做什么       |
+----------------+----------------------------+------------------------------+
|   典型场景      |   张三登录后台管理系统       |   AI Agent 代理调用 API       |
|                |   → 拥有"运营"角色权限      |   → 拥有"自动化工具"权限       |
+----------------+----------------------------+------------------------------+
|   凭证类型      |   用户的 ssoid              |   应用的 client_credentials   |
+----------------+----------------------------+------------------------------+
|   使用方式      |   用户在浏览器中操作         |   程序通过 API 调用            |
+----------------+----------------------------+------------------------------+
```

#### 4.1.4 综合场景

```
  场景：张三通过 AI 代理访问报表系统
  
  ┌──────┐      ┌──────────┐      ┌──────────────┐
  │ 张三  │─────▶│ AI Agent │─────▶│  报表系统     │
  │(misId)│      │(clientId)│      │              │
  └──────┘      └──────────┘      └──────────────┘
  
  报表系统需要同时检查两个维度的权限：
  
  ✅ 检查一：张三（用户身份）是否有权访问报表系统？
     → UAC 查询张三的角色和权限
     
  ✅ 检查二：AI Agent（应用身份）是否被授权调用报表 API？
     → UAC 查询 AI Agent 的应用权限
     
  只有两者都通过，请求才会被放行。
```

#### 4.1.5 UAC 与 SSO 的关系

```
  UAC 依赖 SSO 来获取用户身份信息。在请求处理链中：
  
  请求进来
    │
    ▼
  ┌──────────────┐
  │  SSO Filter  │  ← 先运行：验证 ssoid，解析出用户身份（misId）
  │  (认证)       │
  └──────┬───────┘
         │ 传递用户身份
         ▼
  ┌──────────────┐
  │  UAC Filter  │  ← 后运行：根据用户身份检查权限
  │  (授权)       │
  └──────┬───────┘
         │ 权限通过
         ▼
  ┌──────────────┐
  │  业务逻辑     │
  └──────────────┘
  
  UAC Filter 必须在 SSO Filter 之后运行，
  因为它需要 SSO 解析出的用户身份信息来做权限判断。
```

### 4.2 IAM（身份与访问管理）

#### 4.2.1 IAM 是什么

IAM（Identity and Access Management，身份与访问管理）管理的是**系统入口级别的一级权限**——即"你能不能进这个系统"。

#### 4.2.2 IAM vs UAC

```
  IAM 和 UAC 的关系，就像小区门禁和房间门锁的关系：
  
  IAM（一级权限）：你能不能进这个小区？
    → 有门禁卡 → 进入小区 ✅
    → 没门禁卡 → 被拦在小区门口 ❌
    
  UAC（二级权限）：你能进小区里的哪些房间？
    → 你有 1 号楼 101 的钥匙 → 能进 101 ✅
    → 你没有 2 号楼 201 的钥匙 → 不能进 201 ❌
```

#### 4.2.3 层级关系

```
  请求处理流程：
  
  用户请求 ──▶ ┌─────────────┐
              │  IAM 拦截    │  "你有没有权限访问这个系统？"
              │  (一级权限)   │
              └──────┬──────┘
                     │
               ┌─────┴─────┐
               │            │
             放行 ✅      拒绝 ❌
               │          (到此为止，
               │           看不到任何页面)
               ▼
        ┌─────────────┐
        │  UAC 鉴权    │  "你能使用这个系统的哪些功能？"
        │  (二级权限)   │
        └──────┬──────┘
               │
         ┌─────┴─────┐
         │            │
       有权限 ✅    无权限 ❌
         │          (能进系统，
         │           但看不到某些菜单)
         ▼
   ┌──────────────┐
   │  业务逻辑     │
   └──────────────┘
```

#### 4.2.4 2025 年 10 月后的变化

```
  变化内容：
    IAM 管理后台的一级权限管理功能 → 迁移到 UAC 的「一级授权」模块
    
  但是：
    - 员工申请权限的入口仍然是 IAM
    - 用户视角不变，管理员视角变了
    
  ┌─────────────────────────────────────────────────────┐
  │  迁移前：                                            │
  │    管理员在 IAM 后台 配置一级权限                       │
  │    员工在 IAM 入口 申请权限                             │
  │                                                     │
  │  迁移后：                                            │
  │    管理员在 UAC「一级授权」模块 配置一级权限              │
  │    员工仍在 IAM 入口 申请权限 (入口不变)                 │
  └─────────────────────────────────────────────────────┘
```

#### 4.2.5 IAM 的其他职责

```
  除了一级权限管理外，IAM 还承担：
  
  1. 用户账号数据源
     → 全司员工信息的权威来源
     → 其他系统（如 UAC）从 IAM 获取员工数据
     
  2. 权限申请入口
     → 员工申请系统访问权限的统一入口
     → 走审批流程
     
  3. 自动填充跳转
     → 用户访问未授权系统时，自动跳转到 IAM 申请页面
     → 申请通过后自动跳回原系统
```

### 4.3 将军令

#### 4.3.1 将军令是什么

将军令是**数据资产访问权限管理**系统，管理的是对 BI 报表、数据 API、库表等数据资源的访问权限。

```
  将军令管理的数据资产类型：
  
  ┌───────────────────────────────────────┐
  │  将军令 — 数据级权限                    │
  │                                       │
  │  📊 BI 报表：谁能看哪些报表             │
  │  🔌 数据 API：谁能调用哪些数据接口       │
  │  🗄️ 库表：谁能查询哪些数据库表           │
  │                                       │
  └───────────────────────────────────────┘
```

#### 4.3.2 与 UAC/IAM 的区别

```
  三者的关系：从粗到细的三层过滤
  
  IAM  = 入口级：你能不能进这个系统？
  UAC  = 功能级：你能用系统里的哪些功能？
  将军令 = 数据级：你能看到哪些具体数据？
  
  举例：
  张三要查看"华北区 Q3 销售报表"
  
  ① IAM 检查：张三有权访问 BI 系统吗？ → ✅ 通过
  ② UAC 检查：张三有"查看报表"功能的权限吗？ → ✅ 通过
  ③ 将军令检查：张三有权查看"华北区 Q3 销售报表"这个具体报表吗？ → ✅ 通过
  
  三层都通过，张三才能看到数据。
```

#### 4.4.2 一个请求的完整生命周期

```
  用户发起请求：GET /api/report/north-china-q3-sales
  
  ┌──────────────────────────────────────────────────────────────┐
  │  Step 1: SSO Filter 验证 ssoid                               │
  │    → 从 Cookie 中取出 ${clientId}_ssoid                       │
  │    → 验证签名、有效期                                         │
  │    → 解析出 misId = "zhangsan"                               │
  │    → ❌ 无效 → 重定向到 SSO 登录                              │
  │    → ✅ 有效 → 继续                                          │
  ├──────────────────────────────────────────────────────────────┤
  │  Step 2: IAM 一级拦截                                        │
  │    → 检查 zhangsan 是否有 BI 系统的访问权限                     │
  │    → ❌ 无权限 → 返回 403，引导去 IAM 申请                     │
  │    → ✅ 有权限 → 继续                                        │
  ├──────────────────────────────────────────────────────────────┤
  │  Step 3: UAC 二级鉴权                                        │
  │    → 检查 zhangsan 的角色是否包含"查看报表"权限                  │
  │    → ❌ 无权限 → 返回 403，"您没有查看报表的权限"                │
  │    → ✅ 有权限 → 继续                                        │
  ├──────────────────────────────────────────────────────────────┤
  │  Step 4: 业务逻辑                                            │
  │    → 处理报表查询请求                                         │
  │    → 准备返回数据                                             │
  ├──────────────────────────────────────────────────────────────┤
  │  Step 5: 将军令数据权限检查                                    │
  │    → 检查 zhangsan 是否有"华北区 Q3 销售报表"的数据权限          │
  │    → ❌ 无权限 → 返回"您没有该报表的查看权限，请申请"             │
  │    → ✅ 有权限 → 返回报表数据                                  │
  └──────────────────────────────────────────────────────────────┘
```

#### 4.4.3 易混淆点梳理

```
  ┌──────────────────────────────────────────────────────────────┐
  │  易混淆点 1：MOA 无感 ≠ CIBA                                 │
  │                                                              │
  │  MOA 无感：本地 WSS 通信，用户完全无感知，毫秒级               │
  │  CIBA：    后台通道认证，用户需要在手机上确认，秒级到分钟级      │
  │                                                              │
  │  虽然都是"不需要浏览器"的认证方式，但机制完全不同               │
  ├──────────────────────────────────────────────────────────────┤
  │  易混淆点 2：IAM ≠ UAC ≠ 将军令（三层从粗到细）                │
  │                                                              │
  │  IAM：   能不能进系统？         (入口级，粗粒度)               │
  │  UAC：   能用系统里的哪些功能？  (功能级，中粒度)               │
  │  将军令：  能看哪些具体数据？     (数据级，细粒度)               │
  │                                                              │
  │  不是替代关系，而是层层递进的关系                               │
  ├──────────────────────────────────────────────────────────────┤
  │  易混淆点 3：ssoid ≠ TGC                                     │
  │                                                              │
  │  TGC：   存在 SSO 域名下，有效期 14 天                        │
  │          代表"用户已在 SSO 中心登录"                           │
  │  ssoid： 存在业务域名下，有效期 3 天                           │
  │          代表"用户已在本业务系统登录"                           │
  │                                                              │
  │  TGC 是"通行证"，ssoid 是"入场券"                              │
  │  TGC 可以签发多张 ssoid（一次登录，到处使用）                   │
  ├──────────────────────────────────────────────────────────────┤
  │  易混淆点 4：员工 SSO ≠ 商家 SSO（EPassport）                  │
  │                                                              │
  │  员工 SSO（mtcas）：内部员工使用，misId 作为标识                │
  │  商家 SSO（EPassport）：外部商家使用，完全独立的另一套体系        │
  │                                                              │
  │  两套系统互不相通，不要混用                                    │
  ├──────────────────────────────────────────────────────────────┤
  │  易混淆点 5：UAC 应用身份授权 ≠ SSO client_id                  │
  │                                                              │
  │  SSO client_id：标识"这是哪个应用"（身份标识）                  │
  │  UAC 应用身份授权：控制"这个应用能做什么"（权限控制）             │
  │                                                              │
  │  client_id 是名片，UAC 应用身份授权是门禁卡                     │
  │  有名片不代表有门禁卡                                          │
  └──────────────────────────────────────────────────────────────┘
```

### 5.1 接入流程总览

把一个业务系统接入 SSO 认证体系，需要经过以下步骤：

```
  ┌─────────────────────────────────────────────────────────────────────────┐
  │                       SSO 接入完整流程                                  │
  │                                                                         │
  │   ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐         │
  │   │ Step 1   │    │ Step 2   │    │ Step 3   │    │ Step 4   │         │
  │   │ 注册应用  │───→│ 申请SSO  │───→│ 申请UAC  │───→│ 集成SDK  │         │
  │   │          │    │ client   │    │ 权限     │    │          │         │
  │   └──────────┘    └──────────┘    └──────────┘    └──────────┘         │
  │        │                                               │               │
  │        │         去 SSO 管理后台                         │               │
  │        │         获取 client_id                         │               │
  │        │         和 secret                              ▼               │
  │        │                                         ┌──────────┐         │
  │        │                                         │ Step 5   │         │
  │        ▼                                         │ 配置属性  │         │
  │   在公司的服务                                    │ yml配置  │         │
  │   管理平台注册                                    └──────────┘         │
  │   你的应用                                              │               │
  │                                                         ▼               │
  │                                                  ┌──────────┐         │
  │                                                  │ Step 6   │         │
  │                                                  │ 测试验证  │         │
  │                                                  │          │         │
  │                                                  └──────────┘         │
  └─────────────────────────────────────────────────────────────────────────┘
```

每一步具体做什么：

```
+----------+---------------------------+-------------------------------------------+
|   步骤   |         做什么             |              备注                          |
+----------+---------------------------+-------------------------------------------+
| Step 1   | 在服务管理平台注册应用      | 获取你的 appkey，这是应用的"身份证号"       |
+----------+---------------------------+-------------------------------------------+
| Step 2   | 去 SSO 管理后台申请        | 提供 appkey，申请 client_id 和 secret      |
|          | SSO 接入                  | secret 必须存到 KMS，不能写在代码里！       |
+----------+---------------------------+-------------------------------------------+
| Step 3   | 申请 UAC 权限              | 如果你的系统需要功能级权限控制，才需要这步   |
|          | （可选）                   | 纯内部工具可以先跳过                        |
+----------+---------------------------+-------------------------------------------+
| Step 4   | 集成 SDK 到项目            | 加 Maven 依赖，配 yml，几行代码搞定        |
+----------+---------------------------+-------------------------------------------+
| Step 5   | 配置属性                   | client-id、secret、环境、拦截路径等         |
+----------+---------------------------+-------------------------------------------+
| Step 6   | 测试验证                   | 先在 test 环境验证，再上线 prod             |
+----------+---------------------------+-------------------------------------------+
```

> 生活类比：这就像开一家新店要接入支付宝——先注册商户→申请接口权限→集成 SDK→配参数→测试→上线。

### 5.2 Java SDK 接入（sso-xframe-boot-starter）

#### 5.2.1 环境要求

在接入之前，确认你的项目满足以下条件：

```
+------------------+----------------------------------+
|       条件       |           要求                    |
+------------------+----------------------------------+
| Java 版本        | Java 8 及以上                     |
+------------------+----------------------------------+
| XFrame 版本      | >= 2.4.0-RC1                     |
+------------------+----------------------------------+
| Spring Boot 版本 | 2.x（暂不支持 3.x）              |
+------------------+----------------------------------+
```

> 为什么要 XFrame >= 2.4.0-RC1？因为从这个版本开始，SDK 才支持自动 filter 注入和 SSO 回调路径的自动注册。低版本需要手动配置大量 Bean，非常麻烦。

#### 5.2.2 Maven 依赖

在你的 `pom.xml` 中添加：

```xml
<dependency>
    <groupId>com.sankuai.meituan.sso</groupId>
    <artifactId>sso-xframe-boot-starter</artifactId>
    <!-- 版本号跟随 XFrame BOM 管理，通常不需要手动指定 -->
</dependency>
```

就这一个依赖，它会自动引入：
- SSO 核心认证逻辑
- Filter 自动注册（拦截请求、检查登录态）
- 回调 Controller（处理 SSO 重定向回来的 code）
- Token 管理（ssoid 的签发和校验）

#### 5.2.3 application.yml 配置

```yaml
sso:
  enabled: true                          # 是否启用 SSO 认证
  client-id: your-client-id              # 必填！去 SSO 后台申请的 client_id
  secretKmsKey: your-kms-key-name        # 必填！client_secret 在 KMS 中的 key 名称
  accessEnv: prod                        # 环境：prod（线上）或 test（测试）
  excludedUriList:                       # 不需要登录就能访问的路径
    - /api/health
    - /api/public/**
  includedUriList:                       # 需要登录才能访问的路径（优先级高于 excluded）
    - /api/admin/**
  ssoListener: com.example.MySSOListener # 自定义登录回调监听器（可选）
  rewriteLocation: https://your-domain.com  # 重写回调地址（多域名部署时用）
```

#### 5.2.4 配置属性详解

```
+-------------------+--------+------------------------------------------------------------+
|      属性名       | 必填？ |                         说明                                |
+-------------------+--------+------------------------------------------------------------+
| enabled           |  否    | 是否启用 SSO 认证，默认 true。                              |
|                   |        | 设为 false 可以在本地开发时跳过 SSO（不推荐线上关闭）        |
+-------------------+--------+------------------------------------------------------------+
| client-id         |  是    | SSO 后台分配的 client_id，每个应用唯一。                     |
|                   |        | 这是你的应用在 SSO 体系中的"身份证号"                        |
+-------------------+--------+------------------------------------------------------------+
| secretKmsKey      |  是    | client_secret 在 KMS 中存储的 key 名称。                    |
|                   |        | 重要：不要把 secret 明文写在配置文件里！                     |
|                   |        | 必须通过 KMS 管理，SDK 会在运行时从 KMS 拉取 secret         |
+-------------------+--------+------------------------------------------------------------+
| excludedUriList   |  否    | 白名单路径列表，匹配的请求不走 SSO 认证。                    |
|                   |        | 例如：健康检查接口、公开 API、静态资源                       |
|                   |        | 支持 Ant 风格通配符：/api/public/**                         |
+-------------------+--------+------------------------------------------------------------+
| includedUriList   |  否    | 黑名单路径列表，匹配的请求强制走 SSO 认证。                  |
|                   |        | 优先级高于 excludedUriList！                                |
|                   |        | 当两者冲突时，includedUriList 赢                            |
+-------------------+--------+------------------------------------------------------------+
| accessEnv         |  否    | 访问环境，可选值：prod 或 test，默认 prod。                  |
|                   |        | prod 连接线上 SSO，test 连接测试 SSO                        |
+-------------------+--------+------------------------------------------------------------+
| ssoListener       |  否    | 自定义 SSO 登录事件监听器的全限定类名。                      |
|                   |        | 可以在用户登录成功后执行自定义逻辑                            |
|                   |        | （如同步用户信息、记录登录日志等）                            |
+-------------------+--------+------------------------------------------------------------+
| rewriteLocation   |  否    | 重写 SSO 回调地址。多域名部署时，                            |
|                   |        | SSO 回调可能打到错误的域名，用这个参数强制指定                |
+-------------------+--------+------------------------------------------------------------+
```

#### 5.2.5 KMS 地址

client_secret 必须存储在 KMS（Key Management Service）中，绝对不能明文写在代码或配置文件里。

```
+----------+-----------------------------------------------------+
|   环境   |                    KMS 地址                          |
+----------+-----------------------------------------------------+
| 线上     | https://kms.mws.sankuai.com/                        |
+----------+-----------------------------------------------------+
| 测试     | http://kms.mws-test.sankuai.com/key                 |
+----------+-----------------------------------------------------+
```

> 为什么 secret 不能写在代码里？想象一下：你的代码仓库有 100 个人有权限，如果 secret 明文提交到 Git，那 100 个人都知道你的 secret。一旦泄露，攻击者可以冒充你的应用。KMS 就是一个"保险箱"——secret 存在里面，只有你的应用运行时才能取出来用。

### 5.3 SSO 认证流程详解（三种场景）

SDK 接好了，现在来看用户访问时到底发生了什么。这里分三种场景：

#### 场景一：无登录态或登录态失效，登录 App1

> 这是最复杂的场景——用户第一次访问，或者登录已过期，需要走完整的认证流程。

```
  用户浏览器                  App1 (你的应用)              SSO Server (mtcas)
      │                           │                            │
      │  Step 1: 访问 App1        │                            │
      │  GET /dashboard           │                            │
      │ ─────────────────────────→│                            │
      │                           │                            │
      │                           │  Step 2: SSO Filter 检查   │
      │                           │  发现没有 ssoid cookie     │
      │                           │  或 ssoid 已过期           │
      │                           │                            │
      │  Step 3: 302 重定向到 SSO │                            │
      │  Location: sso.sankuai.   │                            │
      │  com/login?client_id=     │                            │
      │  xxx&redirect_uri=        │                            │
      │  app1.com/sso/callback    │                            │
      │ ←─────────────────────────│                            │
      │                           │                            │
      │  Step 4: 浏览器跳转到 SSO 登录页                       │
      │  用户输入 用户名 + 密码                                 │
      │  SSO 验证通过                                           │
      │ ─────────────────────────────────────────────────────→ │
      │                           │                            │
      │  Step 5: SSO 签发 TGC + code                           │
      │  Set-Cookie: TGC=xxx (SSO域名下)                       │
      │  302 重定向回 App1:                                     │
      │  Location: app1.com/sso/  │                            │
      │  callback?code=abc123     │                            │
      │ ←──────────────────────────────────────────────────────│
      │                           │                            │
      │  Step 6: 浏览器带着 code  │                            │
      │  访问 App1 回调接口        │                            │
      │  GET /sso/callback?       │                            │
      │  code=abc123              │                            │
      │ ─────────────────────────→│                            │
      │                           │                            │
      │                           │  Step 7: App1 后端拿 code  │
      │                           │  去 SSO 换 ssoid           │
      │                           │  POST sso.sankuai.com/     │
      │                           │  token                     │
      │                           │  {code=abc123,             │
      │                           │   client_id=xxx,           │
      │                           │   client_secret=yyy}       │
      │                           │ ───────────────────────────→│
      │                           │                            │
      │                           │  返回 ssoid                │
      │                           │ ←───────────────────────── │
      │                           │                            │
      │  Step 8: App1 设置 ssoid  │                            │
      │  Set-Cookie: ssoid=xxx    │                            │
      │  (App1域名下)             │                            │
      │  302 重定向到原始请求      │                            │
      │  /dashboard               │                            │
      │ ←─────────────────────────│                            │
      │                           │                            │
      │  用户看到 dashboard 页面   │                            │
      │  (整个过程 2-3 秒)         │                            │
      └───────────────────────────┴────────────────────────────┘
```

> 生活类比：你第一次去主题乐园，门口检票（Step 2 发现没票）→ 去售票处买票（Step 4 登录）→ 售票处给你手环+门票（Step 5 TGC+code）→ 拿门票回检票口（Step 6）→ 检票口验证门票（Step 7 换 ssoid）→ 给你盖个入场章（Step 8 设置 ssoid）→ 进入乐园。

#### 场景二：已登录 App1，再次访问 App1

> 最简单的场景——你已经有入场章了，直接进。

```
  用户浏览器                  App1 (你的应用)
      │                           │
      │  Step 1: 访问 App1        │
      │  GET /dashboard           │
      │  Cookie: ssoid=xxx        │
      │ ─────────────────────────→│
      │                           │
      │                           │  Step 2: SSO Filter 检查
      │                           │  发现 ssoid 有效
      │                           │  直接放行！
      │                           │
      │  返回 dashboard 页面       │
      │ ←─────────────────────────│
      │                           │
      └───────────────────────────┘

  整个过程：毫秒级，用户完全无感知
```

就两步——检查 ssoid cookie，有效就放行。这就是 SSO 的日常体验：大部分时候，用户根本感觉不到认证的存在。

#### 场景三：已登录 App1，登录 App2（SSO 单点登录的核心价值）

> 这才是 SSO "一次登录，到处使用"的精髓——你登录了 App1，再去 App2 时不需要再输密码。

```
  用户浏览器                  App2 (另一个应用)            SSO Server (mtcas)
      │                           │                            │
      │  前提：用户已登录 App1，浏览器中已有 TGC cookie          │
      │                           │                            │
      │  Step 1: 访问 App2        │                            │
      │  GET /reports             │                            │
      │  (没有 App2 的 ssoid)     │                            │
      │ ─────────────────────────→│                            │
      │                           │                            │
      │                           │  Step 2: SSO Filter 检查   │
      │                           │  发现没有 ssoid cookie     │
      │                           │                            │
      │  Step 3: 302 重定向到 SSO │                            │
      │  Location: sso.sankuai.   │                            │
      │  com/login?client_id=     │                            │
      │  yyy&redirect_uri=        │                            │
      │  app2.com/sso/callback    │                            │
      │ ←─────────────────────────│                            │
      │                           │                            │
      │  Step 4: 浏览器跳转到 SSO │                            │
      │  自动带上 TGC cookie      │                            │
      │  (因为 TGC 在 SSO 域名下) │                            │
      │ ─────────────────────────────────────────────────────→ │
      │                           │                            │
      │                           │    SSO 发现 TGC 有效       │
      │                           │    不需要用户再次登录！     │
      │                           │    直接签发新 code          │
      │                           │                            │
      │  Step 5: SSO 直接重定向    │                            │
      │  回 App2（不弹登录页！）   │                            │
      │  Location: app2.com/sso/  │                            │
      │  callback?code=def456     │                            │
      │ ←──────────────────────────────────────────────────────│
      │                           │                            │
      │  Step 6: 浏览器带 code    │                            │
      │  访问 App2 回调            │                            │
      │ ─────────────────────────→│                            │
      │                           │  App2 拿 code 换 ssoid     │
      │                           │ ───────────────────────────→│
      │                           │ ←───────────────────────── │
      │                           │                            │
      │  设置 App2 的 ssoid       │                            │
      │  重定向到 /reports         │                            │
      │ ←─────────────────────────│                            │
      │                           │                            │
      │  用户看到 reports 页面     │                            │
      │  (全程无需输入密码！)      │                            │
      └───────────────────────────┴────────────────────────────┘
```

**关键区别在 Step 4-5**：浏览器跳转到 SSO 时，自动带上了之前登录 App1 时获得的 TGC cookie。SSO 发现 TGC 有效，就不弹登录页了，直接签发 code 重定向回 App2。整个过程用户只看到页面闪了一下，完全不需要输密码。

> 生活类比：你已经买了主题乐园的手环（TGC），现在要去乐园里的另一个场馆（App2）。场馆门口扫一下你的手环，确认你有票，直接给你盖个本场馆的入场章（App2 的 ssoid）。不需要重新买票。

### 5.4 关键凭证生命周期管理

认证体系里涉及多种凭证，它们的有效期、用途、使用次数都不同。搞混了会出大问题，这里统一梳理：

```
+---------------------+------------------+------------------+----------------------------------+
|        凭证         |     有效期        |     使用次数     |              说明                 |
+---------------------+------------------+------------------+----------------------------------+
| client_id           | 永久             | 无限次           | 应用的"身份证号"，注册后不变      |
+---------------------+------------------+------------------+----------------------------------+
| code                | 10 秒            | 一次性           | 授权码，用完即废，极短生命周期    |
|                     |                  |                  | 防止被截获重放                    |
+---------------------+------------------+------------------+----------------------------------+
| TGC                 | 14 天            | 多次             | SSO 的全局登录态                  |
|                     |                  |                  | 存在 SSO 域名的 cookie 中        |
|                     |                  |                  | 14天内免登录                      |
+---------------------+------------------+------------------+----------------------------------+
| ssoid               | 3 天             | 多次             | 业务系统的局部登录态              |
|                     |                  |                  | 存在业务域名的 cookie 中          |
|                     |                  |                  | 过期后可用 TGC 无感续期           |
+---------------------+------------------+------------------+----------------------------------+
| Access Token        | 1-2 小时         | 多次             | 访问资源的短期令牌                |
|                     |                  |                  | 过期用 Refresh Token 刷新        |
+---------------------+------------------+------------------+----------------------------------+
| Refresh Token       | 数天到数周       | 多次             | 用于刷新 Access Token             |
|                     |                  |                  | 比 Access Token 活得久得多        |
+---------------------+------------------+------------------+----------------------------------+
| CIBA access_token   | 3 天             | 多次             | CIBA 流程获得的 Access Token      |
|                     |                  |                  | 比普通 Access Token 有效期更长    |
+---------------------+------------------+------------------+----------------------------------+
| CIBA refresh_token  | 14 天            | 一次性           | CIBA 流程的 Refresh Token         |
|                     |                  |                  | 一次性使用，刷新后获得新的        |
|                     |                  |                  | refresh_token（轮转机制）         |
+---------------------+------------------+------------------+----------------------------------+
```

> 一个类比帮你记住：  
> - **client_id** = 身份证号（永久不变）  
> - **code** = 排号小票（叫到号就作废）  
> - **TGC** = 游乐园年卡（14天有效）  
> - **ssoid** = 单个场馆的入场章（3天有效）  
> - **Access Token** = 临时工牌（1-2小时，干完活就收回）  
> - **Refresh Token** = 续签条（用它换新工牌）

### 5.5 登出机制

登出不只是"删掉 cookie"这么简单。SSO 体系下有三种登出方式：

#### 5.5.1 主动登出

用户点击"退出登录"按钮，触发完整的登出流程：

```
  用户浏览器                  App1                     SSO Server
      │                        │                          │
      │  点击"退出登录"         │                          │
      │ ──────────────────────→│                          │
      │                        │                          │
      │                        │  清除 App1 的 ssoid      │
      │                        │  重定向到 SSO 登出接口    │
      │                        │                          │
      │  302 到 SSO logout     │                          │
      │ ←──────────────────────│                          │
      │                        │                          │
      │  访问 SSO logout       │                          │
      │ ──────────────────────────────────────────────── →│
      │                        │                          │
      │                        │    SSO 清除 TGC cookie   │
      │                        │    通知所有已登录的应用   │
      │                        │    (可选的全局登出)       │
      │                        │                          │
      │  重定向回登录页         │                          │
      │ ←──────────────────────────────────────────────── │
      └────────────────────────┴──────────────────────────┘
```

**注意**：主动登出会清除 TGC，意味着所有系统的单点登录态都会失效。用户下次访问任何系统都需要重新登录。

#### 5.5.2 TGC 失效

TGC 有效期 14 天。超过 14 天没有活动，TGC 自动过期：

- 用户下次访问任何系统时，SSO 发现 TGC 无效
- 弹出登录页，用户需要重新登录
- 重新签发新的 TGC

#### 5.5.3 ssoid 失效与自动续期

ssoid 有效期 3 天，但不是"3 天后就要重新登录"：

```
  ssoid 过期后的处理：

  ┌─────────────────────────────────────────────────────┐
  │  ssoid 过期了？                                      │
  │       │                                              │
  │       ▼                                              │
  │  检查 TGC 是否还有效                                  │
  │       │                                              │
  │       ├── TGC 有效 → 无感续期！                      │
  │       │   走"场景三"的流程（无需输密码）               │
  │       │   自动获得新的 ssoid                          │
  │       │   用户完全无感知                              │
  │       │                                              │
  │       └── TGC 也过期 → 需要重新登录                  │
  │           弹出登录页                                  │
  │           走"场景一"的完整流程                        │
  └─────────────────────────────────────────────────────┘
```

所以实际体验是：只要 14 天内有活动（TGC 还在），ssoid 过期了也能无感续期。只有超过 14 天没用过，才需要重新输密码。

## 第六章：工程实践 — CLI/工具场景认证

前面讲的都是 Web 应用的认证——浏览器跳转、cookie、回调。但如果你在命令行（CLI）或脚本里需要认证呢？

> 类比：Web 认证就像去银行柜台办业务（面对面），CLI 认证就像电话银行（没有"面对面"的浏览器）。方式不同，但目的一样——证明你是你。

---

### 6.1 sso-auth-cli 工具

#### 6.1.1 项目背景和目标

在日常开发中，很多场景需要在命令行环境下获取认证凭证：

- CI/CD 流水线里调用内部 API
- 本地脚本批量处理数据
- CLI 工具访问需要认证的服务
- 开发调试时获取 Access Token

以前这些场景要么手动去浏览器抓 Token（麻烦），要么硬编码密码（不安全），要么用一些临时方案（不统一）。**sso-auth-cli** 就是为了解决这些问题而生的统一 CLI 认证工具。

#### 6.1.2 核心能力

```
+---------------------------+------------------------------------------------------+
|         能力              |                      说明                             |
+---------------------------+------------------------------------------------------+
| 多认证路径                | 支持 OIDC、Token Exchange、CIBA 三种认证方式         |
|                           | 自动选择最优路径                                      |
+---------------------------+------------------------------------------------------+
| Token 缓存               | 自动缓存 Token 到本地文件                            |
|                           | 下次使用时先检查缓存，避免重复认证                    |
+---------------------------+------------------------------------------------------+
| 自动刷新                  | Token 过期时自动使用 Refresh Token 刷新              |
|                           | 静默完成，无需用户干预                                |
+---------------------------+------------------------------------------------------+
| misId 解析                | 自动识别当前用户的 misId                             |
|                           | 支持多种来源                                          |
+---------------------------+------------------------------------------------------+
| 安全存储                  | Token 缓存文件权限 600（仅所有者可读写）             |
|                           | 不会意外泄露                                          |
+---------------------------+------------------------------------------------------+
| 环境适配                  | 自动区分 prod/test 环境                              |
|                           | 使用对应环境的 SSO 端点                               |
+---------------------------+------------------------------------------------------+
```

---

### 6.2 三条认证路径实现

sso-auth-cli 支持三条认证路径，每条路径都有完整的"先查缓存→尝试刷新→走完整流程"的策略：

#### 路径一：OIDC 认证

这是最常用的路径，适用于有浏览器环境的交互式场景。

```
  ┌────────────────────────────────────────────────────────────────────┐
  │                    OIDC 认证路径决策流程                           │
  │                                                                    │
  │  Step 1: 检查本地缓存                                             │
  │     │                                                              │
  │     ├── 缓存命中且 Access Token 未过期                             │
  │     │   → 直接返回缓存的 Token ✅                                  │
  │     │                                                              │
  │     ├── 缓存命中但 Access Token 已过期，Refresh Token 还在         │
  │     │   → Step 2: 用 Refresh Token 刷新                           │
  │     │     │                                                        │
  │     │     ├── 刷新成功 → 更新缓存，返回新 Token ✅                 │
  │     │     └── 刷新失败 → 继续走 Step 3                            │
  │     │                                                              │
  │     └── 缓存未命中 / Refresh Token 也过期                         │
  │         → Step 3: 走完整 OIDC PKCE 流程                           │
  │                                                                    │
  │  Step 3: 完整 PKCE 流程                                           │
  │     1. 生成 code_verifier 和 code_challenge                       │
  │     2. 启动本地临时 HTTP Server（监听回调）                        │
  │     3. 打开浏览器跳转到 SSO 登录页                                 │
  │        URL 包含：client_id, code_challenge, redirect_uri          │
  │     4. 用户在浏览器中登录                                          │
  │     5. SSO 重定向到本地 HTTP Server，带上 code                     │
  │     6. 用 code + code_verifier 换 Access Token                    │
  │     7. 缓存 Token 到本地文件                                       │
  │     8. 返回 Token ✅                                               │
  └────────────────────────────────────────────────────────────────────┘
```

> PKCE 是什么？简单说就是一个"密码信封"机制。CLI 先生成一个随机密码（code_verifier），把它的哈希（code_challenge）发给 SSO。SSO 返回 code 后，CLI 出示原始密码证明"领 code 的人和换 Token 的人是同一个"。这样即使 code 被中间人截获，没有 code_verifier 也换不了 Token。

#### 路径二：Token Exchange 认证

适用于已有一种 Token、需要换成另一种 Token 的场景（服务间调用、委托授权等）。

```
  ┌────────────────────────────────────────────────────────────────────┐
  │                Token Exchange 认证路径决策流程                      │
  │                                                                    │
  │  Step 1: 检查本地缓存                                             │
  │     │                                                              │
  │     ├── 缓存命中且目标 Token 未过期                                │
  │     │   → 直接返回 ✅                                              │
  │     │                                                              │
  │     └── 缓存未命中 / 已过期                                       │
  │         → Step 2: 寻找 source token                               │
  │                                                                    │
  │  Step 2: 寻找可用的 source token                                   │
  │     │                                                              │
  │     ├── 找到有效的 source token（如 MOA token 等）                 │
  │     │   → Step 3: 发起 Token Exchange                             │
  │     │                                                              │
  │     └── 没有可用的 source token                                   │
  │         → 兜底：使用 client_id + client_secret                    │
  │           走 Client Credentials 获取基础 Token                     │
  │           然后再用这个 Token 做 Exchange                            │
  │                                                                    │
  │  Step 3: POST /token-exchange                                      │
  │     请求参数：                                                     │
  │     {                                                              │
  │       grant_type: "urn:ietf:params:oauth:grant-type:token-exchange"│
  │       subject_token: <source_token>,                               │
  │       subject_token_type: "urn:ietf:params:oauth:token-type:       │
  │                            access_token",                          │
  │       audience: <target_client_id>                                 │
  │     }                                                              │
  │     │                                                              │
  │     ├── 成功 → 缓存目标 Token，返回 ✅                            │
  │     └── 失败 → 返回错误信息                                       │
  └────────────────────────────────────────────────────────────────────┘
```

> 生活类比：你有一张美元（source token），要去日本买东西，需要换成日元（target token）。Token Exchange 就是"货币兑换"——拿你已有的凭证，换成目标系统认可的凭证。

#### 路径三：CIBA 认证

适用于无浏览器环境，通过手机 MOA 确认身份。

```
  ┌────────────────────────────────────────────────────────────────────┐
  │                    CIBA 认证路径决策流程                            │
  │                                                                    │
  │  Step 1: 检查本地缓存                                             │
  │     │                                                              │
  │     ├── 缓存命中且 CIBA access_token 未过期                       │
  │     │   → 直接返回 ✅                                              │
  │     │                                                              │
  │     ├── 缓存命中但 access_token 过期，CIBA refresh_token 还在     │
  │     │   → Step 2: 刷新                                            │
  │     │     │                                                        │
  │     │     ├── 成功 → 更新缓存，返回新 Token ✅                     │
  │     │     │   (注意：refresh_token 也会轮转为新的)                 │
  │     │     └── 失败 → 继续走 Step 3                                │
  │     │                                                              │
  │     └── 缓存未命中 / refresh_token 也过期                         │
  │         → Step 3: 走完整 CIBA 流程                                │
  │                                                                    │
  │  Step 3: 完整 CIBA 流程                                           │
  │     1. POST /bc-authorize                                          │
  │        { login_hint: "misId:zhangsan", client_id: xxx }            │
  │     2. 获得 auth_req_id                                            │
  │     3. 用户手机 MOA 弹出确认通知                                   │
  │        "是否允许 xxx 应用登录？"                                    │
  │     4. CLI 开始轮询 POST /token                                    │
  │        { grant_type: "urn:openid:params:grant-type:ciba",          │
  │          auth_req_id: xxx }                                        │
  │     5. 用户在 MOA 上点击"确认"                                     │
  │     6. 轮询返回 access_token + refresh_token                       │
  │     7. 缓存 Token 到本地文件                                       │
  │     8. (可选) 再通过 Token Exchange 换成目标 Token                 │
  │     9. 返回最终 Token ✅                                           │
  └────────────────────────────────────────────────────────────────────┘
```

---

### 6.3 misId 解析

很多操作需要知道当前用户的 misId（如 CIBA 的 login_hint）。sso-auth-cli 按以下优先级自动解析 misId：

```
  优先级从高到低：

  ┌──────────┐     ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
  │ 优先级 1 │     │  优先级 2    │     │  优先级 3    │     │  优先级 4    │
  │          │     │              │     │              │     │              │
  │ 命令行   │ ──→ │  环境变量    │ ──→ │  本地 SSO   │ ──→ │  系统用户名  │
  │ 显式传入 │     │  SSO_MIS_ID  │     │  缓存文件   │     │  whoami      │
  └──────────┘     └──────────────┘     └──────────────┘     └──────────────┘
       │                   │                    │                    │
       │     最高优先级     │   CI/CD 常用       │  本地开发常用      │  兜底方案
       │     --mis=xxx     │                    │  之前登录时缓存    │  不一定准确
       │                   │                    │  的 misId          │
       └───────────────────┴────────────────────┴────────────────────┘
```

具体说明：

1. **命令行显式传入**：`--mis=zhangsan`，优先级最高，适合脚本明确指定用户
2. **环境变量**：`export SSO_MIS_ID=zhangsan`，适合 CI/CD 流水线统一配置
3. **本地 SSO 缓存文件**：之前 OIDC 登录成功后，Token 缓存文件中会包含 misId 信息
4. **系统用户名**：`whoami` 命令的输出，作为最后兜底方案，但注意共享机器上可能不准确

### 6.4 使用示例

#### 6.4.1 基础命令行使用

```bash
# 获取 Access Token（默认走 OIDC 路径）
$ sso-auth-cli token --client-id your-client-id
eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9...

# 指定走 CIBA 路径（无浏览器环境）
$ sso-auth-cli token --client-id your-client-id --method ciba --mis zhangsan
请在 MOA 上确认登录...
eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9...

# Token Exchange（已有 source token）
$ sso-auth-cli token --client-id target-client-id --method exchange \
    --source-token eyJhbGci...
eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9...

# 查看当前缓存的 Token 信息
$ sso-auth-cli info
Client ID:  your-client-id
MIS ID:     zhangsan
Token Type: access_token
Expires At: 2024-03-15 16:30:00
Cached:     ~/.sso-auth/token-cache.json

# 清除缓存（重新认证）
$ sso-auth-cli clear
Cache cleared.
```

#### 6.4.2 代码集成使用

在 Node.js 脚本中集成：

```javascript
const { getToken } = require('sso-auth-cli');

async function callInternalAPI() {
  // 获取 Token（自动处理缓存、刷新、认证）
  const token = await getToken({
    clientId: 'your-client-id',
    method: 'oidc',  // 或 'ciba', 'exchange'
  });

  // 用 Token 调用内部 API
  const response = await fetch('https://api.internal.sankuai.com/data', {
    headers: {
      'Authorization': `Bearer ${token}`,
    },
  });

  return response.json();
}
```

在 Python 脚本中集成：

```python
import subprocess
import requests

def get_sso_token(client_id):
    """调用 sso-auth-cli 获取 Token"""
    result = subprocess.run(
        ['sso-auth-cli', 'token', '--client-id', client_id],
        capture_output=True, text=True
    )
    return result.stdout.strip()

def call_internal_api():
    token = get_sso_token('your-client-id')
    response = requests.get(
        'https://api.internal.sankuai.com/data',
        headers={'Authorization': f'Bearer {token}'}
    )
    return response.json()
```

## 第七章：安全设计

认证系统是整个安全体系的基石——一旦认证系统被攻破，所有后续的授权、审计都形同虚设。这一章我们来看整个体系的安全设计。

> 类比：前面几章讲的是"怎么建房子"，这一章讲的是"怎么防火、防盗、防地震"。

---

### 7.1 安全机制总览

```
+---------------------------+----------------------------------------------------+
|       安全机制            |                    说明                             |
+---------------------------+----------------------------------------------------+
| Code 一次性使用           | 授权码 code 只能用一次，且 10 秒过期。              |
|                           | 即使被截获，攻击者必须在 10 秒内抢先使用，           |
|                           | 而且一旦被用过就废了。                               |
+---------------------------+----------------------------------------------------+
| PKCE                      | Proof Key for Code Exchange。                       |
|                           | CLI/移动端没有安全存储 client_secret 的能力，        |
|                           | 用 code_verifier/code_challenge 替代。               |
|                           | 防止授权码被中间人截获后冒用。                       |
+---------------------------+----------------------------------------------------+
| state 参数                | 随机字符串，防止 CSRF 攻击。                         |
|                           | 应用生成 state 放入请求，SSO 原样返回，              |
|                           | 应用校验一致性。                                     |
|                           | 攻击者无法伪造合法的 state。                         |
+---------------------------+----------------------------------------------------+
| client_secret             | 应用的"密码"，存储在 KMS 中，                       |
|                           | 只在后端使用，永远不会暴露到前端。                   |
|                           | 用于 code 换 Token 时证明应用身份。                  |
+---------------------------+----------------------------------------------------+
| Token audience 绑定       | Token 中包含 audience（受众）字段，                 |
|                           | 指定这个 Token 只能用于访问特定的 API。              |
|                           | 防止 Token 被滥用到非目标服务。                      |
+---------------------------+----------------------------------------------------+
| 短生命周期                | Access Token 只有 1-2 小时有效期。                   |
|                           | 即使泄露，攻击窗口也很短。                           |
|                           | 比起"一个 Token 用一年"，安全太多了。                |
+---------------------------+----------------------------------------------------+
| Refresh Token 轮转        | 每次使用 Refresh Token 刷新时，                     |
|                           | 旧 Refresh Token 作废，签发新的。                    |
|                           | 如果旧的被盗用，合法用户刷新时会发现冲突，           |
|                           | 系统可以立即吊销所有 Token。                         |
+---------------------------+----------------------------------------------------+
```

这些机制不是"选一个用"，而是**层层叠加**，形成纵深防御：

```
  攻击者要突破的层次：

  ┌─────────────────────────────────────────────────────┐
  │  第1层：HTTPS 加密传输                               │
  │  ┌─────────────────────────────────────────────────┐│
  │  │  第2层：state 参数防 CSRF                        ││
  │  │  ┌─────────────────────────────────────────────┐││
  │  │  │  第3层：PKCE 防授权码截获                     │││
  │  │  │  ┌─────────────────────────────────────────┐│││
  │  │  │  │  第4层：code 一次性 + 10秒过期           ││││
  │  │  │  │  ┌─────────────────────────────────────┐││││
  │  │  │  │  │  第5层：client_secret 后端校验       │││││
  │  │  │  │  │  ┌─────────────────────────────────┐│││││
  │  │  │  │  │  │  第6层：Token audience 绑定      ││││││
  │  │  │  │  │  │  ┌─────────────────────────────┐│││││││
  │  │  │  │  │  │  │  第7层：Token 短生命周期     ││││││││
  │  │  │  │  │  │  │        1-2 小时过期         ││││││││
  │  │  │  │  │  │  └─────────────────────────────┘│││││││
  │  │  │  │  │  └─────────────────────────────────┘││││││
  │  │  │  │  └─────────────────────────────────────┘│││││
  │  │  │  └─────────────────────────────────────────┘││││
  │  │  └─────────────────────────────────────────────┘│││
  │  └─────────────────────────────────────────────────┘││
  └─────────────────────────────────────────────────────┘│
                                                         │
  攻击者：要穿透 7 层才能得手... 太难了                    │
```

### 7.2 CIBA 的安全边界

CIBA（Client Initiated Backchannel Authentication）允许后台通道认证，这很方便，但也有独特的安全风险：

#### login_hint 安全性

```
  CIBA 请求中必须携带 login_hint（告诉 SSO "要认证哪个用户"）

  风险：如果攻击者知道某人的 misId，能不能发起 CIBA 让那个人的手机弹确认？
  
  答案：可以弹确认，但攻击者拿不到 Token。因为：
  1. CIBA 请求本身需要 client_id 认证（你得有合法应用身份）
  2. auth_req_id 只返回给发起请求的客户端
  3. 用户在手机上可以选择"拒绝"
  
  但还是要注意：
  - 不要把 login_hint 暴露在日志中
  - 不要把 login_hint 放在 URL query 参数里（会被记录到 access log）
```

#### auth_req_id 保密性

```
  auth_req_id 是 CIBA 流程中的关键凭证：
  
  - 知道 auth_req_id 就能轮询获取 Token
  - 必须像 code 一样严格保密
  - 只能在安全通道中传输
  - 不能记录到日志
```

#### 超时设置

```
  CIBA 必须设置合理的超时时间：
  
  推荐值：2-5 分钟
  
  太短：用户来不及掏手机确认
  太长：攻击窗口变大
  
  ┌──────────────┬──────────────────────────────────┐
  │   超时设置   │             影响                  │
  ├──────────────┼──────────────────────────────────┤
  │  30 秒       │ 太短，用户体验差                  │
  │  2 分钟      │ 推荐最小值                        │
  │  5 分钟      │ 推荐最大值                        │
  │  30 分钟     │ 太长，安全风险大                  │
  └──────────────┴──────────────────────────────────┘
```

#### 频率限制

```
  必须对 CIBA 请求做频率限制：
  
  - 同一个 client_id 对同一个 login_hint：每分钟最多 3 次
  - 同一个 client_id 全局：每分钟最多 30 次
  - 防止攻击者恶意轰炸用户手机
```

### 7.3 缓存安全

sso-auth-cli 会把 Token 缓存到本地文件，这个文件的安全至关重要：

```
  ┌─────────────────────────────────────────────────────────┐
  │                   缓存安全规范                           │
  ├─────────────────────────────────────────────────────────┤
  │                                                         │
  │  1. 文件权限：600（仅所有者可读写）                      │
  │                                                         │
  │     $ ls -la ~/.sso-auth/token-cache.json               │
  │     -rw-------  1 zhangsan  staff  1234  token-cache    │
  │      ↑↑↑                                                │
  │      rw- 所有者可读写                                    │
  │      --- 同组用户无权限                                  │
  │      --- 其他用户无权限                                  │
  │                                                         │
  │  2. 不提交 Git                                          │
  │                                                         │
  │     在 .gitignore 中添加：                               │
  │     .sso-auth/                                          │
  │     **/token-cache*                                     │
  │                                                         │
  │     如果你不小心提交了，仅仅删除文件是不够的！            │
  │     Git 历史中还有！需要用 git filter-branch 彻底清除    │
  │                                                         │
  │  3. 共享机器注意                                        │
  │                                                         │

  │  4. 定期清理                                            │
  │                                                         │

  │                                                         │
  └─────────────────────────────────────────────────────────┘
```

### 7.4 Token Exchange 的委托安全

Token Exchange 允许"用 A 的 Token 换 B 的 Token"，这个能力如果不加限制，就会变成安全漏洞。

```
  ┌─────────────────────────────────────────────────────────────────┐
  │                 Token Exchange 安全控制                         │
  ├─────────────────────────────────────────────────────────────────┤
  │                                                                 │
  │  1. 严格控制换票权限                                            │
  │                                                                 │
  │     不是所有应用都能做 Token Exchange！                          │
  │     需要在 SSO 后台显式配置：                                   │
  │     "应用 A 允许将其 Token 换成应用 B 的 Token"                 │
  │                                                                 │
  │     没有配置的换票请求会被直接拒绝                              │
  │                                                                 │
  │  2. 审计日志                                                    │
  │                                                                 │
  │     每次 Token Exchange 都会记录：                               │
  │     - 谁发起的（source client_id）                              │
  │     - 换成了谁（target client_id）                              │
  │     - 什么时间                                                  │
  │     - 原始用户是谁（subject）                                   │
  │                                                                 │
  │     安全团队可以定期审计这些日志，                               │
  │     发现异常的换票行为                                          │
  │                                                                 │
  │  3. 最小权限原则                                                │
  │                                                                 │
  │     换出来的 Token 权限不能超过源 Token：                       │
  │                                                                 │
  │     源 Token 权限：[read, write]                                │
  │     目标 Token 权限：只能 <= [read, write]                      │
  │                                                                 │
  │     不能通过 Exchange "提权"——                                  │
  │     你不能把一张普通员工的工牌换成管理员工牌                    │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘
```

### 7.5 多因子认证安全

多因子认证（MFA）通过组合多种验证方式，大幅提升安全性。

#### 因子组合策略

```
  ┌───────────────────────────────────────────────────────────────────┐
  │                    多因子认证因子组合                              │
  ├───────────────────────────────────────────────────────────────────┤
  │                                                                   │
  │  因子类型：                                                       │
  │  ┌──────────────┬─────────────────────────────────────────────┐  │
  │  │  知识因子    │  你知道什么 → 密码、PIN、安全问题            │  │
  │  ├──────────────┼─────────────────────────────────────────────┤  │
  │  │  持有因子    │  你拥有什么 → 手机（MOA/短信）、硬件令牌     │  │
  │  ├──────────────┼─────────────────────────────────────────────┤  │
  │  │  生物因子    │  你是什么   → 人脸识别、指纹                 │  │
  │  └──────────────┴─────────────────────────────────────────────┘  │
  │                                                                   │
  │  组合策略（安全级别从低到高）：                                    │
  │                                                                   │
  │  普通操作：     密码                          (单因子)             │
  │  敏感操作：     密码 + MOA 确认               (双因子)             │
  │  高危操作：     密码 + MOA 确认 + 人脸识别    (三因子)             │
  │                                                                   │
  │  例如：                                                           │
  │  查看报表 → 密码登录即可                                          │
  │  修改配置 → 密码 + MOA 二次确认                                   │
  │  删除数据 → 密码 + MOA + 人脸识别                                 │
  │                                                                   │
  └───────────────────────────────────────────────────────────────────┘
```

#### 人脸识别白名单

```
  不是所有人都需要人脸识别：
  
  ┌────────────────────────────────────────────────────────────┐
  │  白名单机制：                                              │
  │                                                            │
  │  已入库人脸的员工 → 直接走人脸识别验证                      │
  │  未入库人脸的员工 → 跳过人脸，走其他因子补偿               │
  │  特殊场景（口罩/光线差）→ 降级到备选因子                   │
  │                                                            │
  │  注意：白名单不是"豁免"，而是"用别的方式替代"
  │  安全级别不降低                                          │
  └────────────────────────────────────────────────────────────┘
```

#### 同事辅助认证兜底

```
  极端场景：员工手机丢了、人脸无法识别、所有常规因子都不可用
  
  兜底方案：同事辅助认证
  
  ┌────────────────────────────────────────────────────────────┐
  │  流程：                                                    │
  │                                                            │
  │  1. 员工发起"辅助认证"请求                                 │
  │  2. 系统通知该员工的 N 位同事（通常是直属上级 + 2位同组同事）│
  │  3. 至少 M 位同事在 MOA 上确认"是本人"                     │
  │  4. 确认数达标 → 允许登录                                  │
  │                                                            │
  │  安全约束：                                                │
  │  - 辅助认证有冷却时间（如每月最多 2 次）                    │
  │  - 会触发安全告警，安全团队会关注                           │
  │  - 登录后自动要求重置认证因子（如重新绑定手机）             │
  └────────────────────────────────────────────────────────────┘
```

#### 风控决策

```
  认证不只是"对不对"的问题，还有"可不可疑"的问题：
  
  ┌────────────────────────────────────────────────────────────┐
  │  风控引擎实时评估每次登录的风险等级：                       │
  │                                                            │
  │  低风险（常用设备 + 常用地点 + 工作时间）                   │
  │    → 密码即可登录                                          │
  │                                                            │
  │  中风险（新设备 或 异地登录）                               │
  │    → 追加 MOA 二次验证                                     │
  │                                                            │
  │  高风险（从未见过的设备 + 异国IP + 凌晨3点）                │
  │    → 追加人脸识别 + 安全团队告警                            │
  │                                                            │
  │  极高风险（多次密码错误 + 疑似暴力破解）                    │
  │    → 直接锁定账号 + 通知安全团队                            │
  │                                                            │
  │  风控不是静态规则，而是动态评估，                            │
  │  同一个人在不同场景下会触发不同等级的验证                    │
  └────────────────────────────────────────────────────────────┘
```

```
  │  ║   │                  "能不能进这个系统？" (粗粒度)                           │  ║    │
  │  ║   └──────────────────────────────┬───────────────────────────────────────────┘  ║    │
  │  ║                                  │                                              ║    │
  │  ║                                  ▼                                              ║    │
  │  ║   ┌──────────────────────────────────────────────────────────────────────────┐  ║    │
  │  ║   │                         UAC (功能级鉴权)                                │  ║    │
  │  ║   │              "能用系统里的哪些功能？" (中粒度)                           │  ║    │
  │  ║   │       角色 → 权限 → 资源 (RBAC)                                        │  ║    │
  │  ║   └──────────────────────────────┬───────────────────────────────────────────┘  ║    │
  │  ║                                  │                                              ║    │
  │  ║                                  ▼                                              ║    │
  │  ║   ┌──────────────────────────────────────────────────────────────────────────┐  ║    │
  │  ║   │                       将军令 (数据级鉴权)                               │  ║    │
  │  ║   │              "能看哪些具体数据？" (细粒度)                               │  ║    │
  │  ║   │       数据权限策略 → 行/列过滤                                          │  ║    │
  │  ║   └──────────────────────────────────────────────────────────────────────────┘  ║    │
  │  ║                                                                                 ║    │
  │  ╚═════════════════════════════════════════════════════════════════════════════════╝    │
  │                                                                                          │
  └──────────────────────────────────────────────────────────────────────────────────────────┘
```

> 一句话总结：**mtcas 确认"你是谁"，IAM 决定"能不能进"，UAC 控制"能做什么"，将军令 限制"能看什么数据"**。四层从外到内，从粗到细。

### 8.2 一次完整请求的生命周期

让我们追踪一个真实请求，看它如何穿越整个体系：

> 场景：张三打开报表系统（已集成 SSO + UAC + 将军令），查看"华北区 Q3 销售数据"。

```
  ┌────────────────────────────────────────────────────────────────────────────────────────┐
  │                           一次完整请求的生命周期                                       │
  ├────────────────────────────────────────────────────────────────────────────────────────┤
  │                                                                                        │
  │  用户请求                                                                              │
  │  GET /api/report?region=华北&quarter=Q3                                                │
  │       │                                                                                │
  │       ▼                                                                                │
  │  ┌─────────────────────────────────────────────────────────────────────────────────┐   │
  │  │  Step 1: SSO Filter (sso-xframe-boot-starter)                                  │   │
  │  │                                                                                 │   │
  │  │  检查请求中的 ssoid cookie：                                                    │   │
  │  │    → ssoid 不存在或已过期？ → 重定向到 SSO 登录页（参见第五章场景一/三）         │   │
  │  │    → ssoid 有效？ → 解析出 misId: zhangsan → 放入请求上下文，继续往下             │   │
  │  │                                                                                 │   │
  │  │  结果：确认 "请求者是 zhangsan"                                                  │   │
  │  └─────────────────────────────┬───────────────────────────────────────────────────┘   │
  │                                │                                                       │
  │                                ▼                                                       │
  │  ┌─────────────────────────────────────────────────────────────────────────────────┐   │
  │  │  Step 2: IAM 一级拦截 (入口级鉴权)                                             │   │
  │  │                                                                                 │   │
  │  │  检查：zhangsan 是否有权访问"报表系统"？                                        │   │
  │  │    → 查询 IAM 权限配置                                                          │   │
  │  │    → 无权限？ → 返回 403 "您无权访问此系统，请联系管理员"                        │   │
  │  │    → 有权限？ → 继续往下                                                        │   │
  │  │                                                                                 │   │
  │  │  结果：确认 "zhangsan 可以进入报表系统"                                          │   │
  │  └─────────────────────────────┬───────────────────────────────────────────────────┘   │
  │                                │                                                       │
  │                                ▼                                                       │
  │  ┌─────────────────────────────────────────────────────────────────────────────────┐   │
  │  │  Step 3: UAC 二级鉴权 (功能级鉴权)                                             │   │
  │  │                                                                                 │   │
  │  │  检查：zhangsan 是否有权使用"查看报表"功能？                                    │   │
  │  │    → 查询 UAC：zhangsan 的角色 → 角色关联的权限 → 是否包含 report:read          │   │
  │  │    → 无权限？ → 返回 403 "您没有查看报表的权限"                                  │   │
  │  │    → 有权限？ → 继续往下                                                        │   │
  │  │                                                                                 │   │
  │  │  结果：确认 "zhangsan 有权查看报表"                                              │   │
  │  └─────────────────────────────┬───────────────────────────────────────────────────┘   │
  │                                │                                                       │
  │                                ▼                                                       │
  │  ┌─────────────────────────────────────────────────────────────────────────────────┐   │
  │  │  Step 4: 业务逻辑层                                                            │   │
  │  │                                                                                 │   │
  │  │  执行查询逻辑：查华北区 Q3 销售数据                                              │   │
  │  │    → SQL: SELECT * FROM sales WHERE region='华北' AND quarter='Q3'               │   │
  │  │    → 但在返回数据之前，还需要检查数据权限！                                      │   │
  │  │                                                                                 │   │
  │  └─────────────────────────────┬───────────────────────────────────────────────────┘   │
  │                                │                                                       │
  │                                ▼                                                       │
  │  ┌─────────────────────────────────────────────────────────────────────────────────┐   │
  │  │  Step 5: 将军令数据权限检查 (数据级鉴权)                                       │   │
  │  │                                                                                 │   │
  │  │  检查：zhangsan 是否有权看"华北区"的数据？                                      │   │
  │  │    → 查询将军令数据权限策略                                                     │   │
  │  │    → zhangsan 的数据范围是"华东区"？                                             │   │
  │  │      → 返回 403 "您没有华北区数据的查看权限"                                     │   │
  │  │    → zhangsan 的数据范围包含"华北区"？                                           │   │
  │  │      → 返回数据（可能还会过滤掉某些敏感列）                                     │   │
  │  │                                                                                 │   │
  │  │  结果：只返回 zhangsan 有权看到的数据                                            │   │
  │  └─────────────────────────────┬───────────────────────────────────────────────────┘   │
  │                                │                                                       │
  │                                ▼                                                       │
  │  ┌─────────────────────────────────────────────────────────────────────────────────┐   │
  │  │  Step 6: 返回响应                                                              │   │
  │  │                                                                                 │   │
  │  │  HTTP 200 OK                                                                    │   │
  │  │  Content-Type: application/json                                                 │   │
  │  │  { "data": [...华北区Q3销售数据...], "total": 1234 }                             │   │
  │  └─────────────────────────────────────────────────────────────────────────────────┘   │
  │                                                                                        │
  └────────────────────────────────────────────────────────────────────────────────────────┘
```

**总结一下这 6 步**：

```
  请求 → SSO(你是谁?) → IAM(能进吗?) → UAC(能用吗?) → 业务逻辑 → 将军令(能看吗?) → 响应
         身份认证        入口鉴权       功能鉴权       处理请求     数据鉴权        返回结果
```

任何一步失败，请求都会被拦截，返回相应的错误信息。

### 9.1 案例一：用户密码登录全链路

#### 9.1.1 业务流程图

```
  用户浏览器              LoginController           LoginService            Redis/DB
      │                        │                        │                     │
      │ POST /api/login        │                        │                     │
      │ {username,password,    │                        │                     │
      │  requestId}            │                        │                     │
      │ ───────────────────────→                        │                     │
      │                        │  Step1: 参数校验        │                     │
      │                        │  （非空/格式/长度）      │                     │
      │                        │                        │                     │
      │                        │  Step2: 幂等检查         │                     │
      │                        │  requestId 是否已处理    │                     │
      │                        │ ───────────────────────────────────────────→ │
      │                        │                        │  Step3: 限流+防爆破  │
      │                        │                        │  失败次数计数         │
      │                        │                        │ ───────────────────→│
      │                        │                        │  Step4: 查用户+验密码 │
      │                        │                        │  BCrypt.matches      │
      │                        │                        │ ───────────────────→│
      │                        │                        │  Step5: 生成JWT Token │
      │                        │                        │  Step6: 写入Session   │
      │                        │                        │  （Redis，含过期时间）│
      │                        │                        │ ───────────────────→│
      │  200 {token,expireIn}  │                        │                     │
      │ ←───────────────────────                        │                     │
      └────────────────────────┴────────────────────────┴─────────────────────┘
```

> 生活类比：这就像去银行柜台办业务——先填单子（参数校验），柜员先看你有没有重复排号（幂等），再看你今天是不是频繁尝试密码错误（防爆破），核对身份证和密码（验密码），最后给你发一张当日有效的取号凭证（Token + Session）。

#### 9.1.2 关键设计点

```
+------------------+--------------------------------------------------------------+
|      设计点       |                          说明                                 |
+------------------+--------------------------------------------------------------+
| 参数校验          | 用户名/密码非空、长度限制、格式白名单，防止 SQL 注入/XSS 载荷 |
+------------------+--------------------------------------------------------------+
| 幂等控制          | 前端生成 requestId（UUID），后端用 Redis SETNX 做请求级幂等， |
|                  | 防止用户手抖多次点击登录按钮导致重复处理                       |
+------------------+--------------------------------------------------------------+
| 防爆破            | 同一账号/IP 在时间窗口内失败次数超阈值，触发锁定或图形验证码   |
+------------------+--------------------------------------------------------------+
| 密码存储          | 绝不明文存储，使用 BCrypt（自带 salt，慢哈希抗暴力破解）      |
+------------------+--------------------------------------------------------------+
| Token 生成        | JWT，包含 userId、issuedAt、expireAt，用 HMAC-SHA256 签名     |
+------------------+--------------------------------------------------------------+
| Session 写入      | Redis 存储 token → 用户信息映射，支持后续踢人下线              |
+------------------+--------------------------------------------------------------+
| 日志记录          | 登录成功/失败都要记审计日志，脱敏处理（密码绝不落日志）        |
+------------------+--------------------------------------------------------------+
```

#### 9.1.3 完整代码实现

**Controller 层：**

```java

import com.example.auth.dto.LoginResponse;

import lombok.extern.slf4j.Slf4j;
import org.springframework.web.bind.annotation.*;

import javax.validation.Valid;
import javax.servlet.http.HttpServletRequest;

@Slf4j
@RestController
@RequestMapping("/api")
public class LoginController {

    private final LoginService loginService;

    public LoginController(LoginService loginService) {
        this.loginService = loginService;
    }

    /**
     * 用户密码登录接口
     * 全链路：参数校验 -> 幂等检查 -> 密码验证 -> 生成Token -> 写入Session -> 返回
     */
    @PostMapping("/login")
    public LoginResponse login(@Valid @RequestBody LoginRequest request,
                                HttpServletRequest httpRequest) {
        String clientIp = resolveClientIp(httpRequest);
        long start = System.currentTimeMillis();

        log.info("[login] 收到登录请求, username={}, requestId={}, ip={}",
                request.getUsername(), request.getRequestId(), clientIp);

        try {
            LoginResponse response = loginService.login(request, clientIp);
            log.info("[login] 登录成功, username={}, userId={}, cost={}ms",
                    request.getUsername(), response.getUserId(),
                    System.currentTimeMillis() - start);
            return response;
        } catch (BusinessException e) {
            // 业务异常：账号密码错误、账号被锁定等，记录 WARN，不打印堆栈
            log.warn("[login] 登录失败(业务异常), username={}, ip={}, code={}, msg={}",
                    request.getUsername(), clientIp, e.getCode(), e.getMessage());
            throw e;
        } catch (Exception e) {
            // 系统异常：Redis/DB 挂了，记录 ERROR，打印堆栈方便排查
            log.error("[login] 登录失败(系统异常), username={}, ip={}",
                    request.getUsername(), clientIp, e);
            throw new BusinessException("SYSTEM_ERROR", "系统繁忙，请稍后重试");
        }
    }

    /**
     * 优先取 X-Forwarded-For，兼容经过网关/负载均衡的场景
     */
    private String resolveClientIp(HttpServletRequest request) {
        String xff = request.getHeader("X-Forwarded-For");
        if (xff != null && !xff.isEmpty() && !"unknown".equalsIgnoreCase(xff)) {
            return xff.split(",")[0].trim();
        }
        return request.getRemoteAddr();
    }
```

**Service 层（核心业务逻辑，幂等 + 防爆破 + 密码校验 + Token 生成）：**

```java
package com.example.auth.service;

import com.example.auth.dto.LoginRequest;
import com.example.auth.dto.LoginResponse;
import com.example.auth.entity.UserEntity;
import com.example.auth.exception.BusinessException;
import com.example.auth.mapper.UserMapper;
import com.example.auth.util.JwtUtil;
import lombok.extern.slf4j.Slf4j;
import org.springframework.data.redis.core.StringRedisTemplate;
import org.springframework.security.crypto.bcrypt.BCryptPasswordEncoder;
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;

import java.time.Duration;
import java.util.UUID;
import java.util.concurrent.TimeUnit;

@Slf4j
@Service
public class LoginService {

    /** 幂等键前缀：requestId 维度，防止重复提交 */
    private static final String IDEMPOTENT_KEY_PREFIX = "login:idempotent:";
    /** 幂等键有效期：5分钟内的重复请求视为重复提交 */
    private static final long IDEMPOTENT_TTL_SECONDS = 300;

    /** 失败计数键前缀：账号维度防爆破 */
    private static final String FAIL_COUNT_KEY_PREFIX = "login:fail:";
    /** 失败次数阈值，超过则锁定账号 */
    private static final int MAX_FAIL_COUNT = 5;
    /** 账号锁定时长 */
    private static final long LOCK_SECONDS = 900; // 15分钟

    /** Session 存储前缀 */
    private static final String SESSION_KEY_PREFIX = "session:token:";
    /** Token 有效期：2小时 */
    private static final long TOKEN_EXPIRE_SECONDS = 7200;

    private final UserMapper userMapper;
    private final StringRedisTemplate redisTemplate;
    private final JwtUtil jwtUtil;
    private final BCryptPasswordEncoder passwordEncoder = new BCryptPasswordEncoder();

    public LoginService(UserMapper userMapper, StringRedisTemplate redisTemplate, JwtUtil jwtUtil) {
        this.userMapper = userMapper;
        this.redisTemplate = redisTemplate;
        this.jwtUtil = jwtUtil;
    }

    @Transactional(rollbackFor = Exception.class)
    public LoginResponse login(LoginRequest request, String clientIp) {

        // ---------- Step 1: 幂等检查（SETNX 原子操作） ----------
        String idempotentKey = IDEMPOTENT_KEY_PREFIX + request.getRequestId();
        Boolean firstTime = redisTemplate.opsForValue()
                .setIfAbsent(idempotentKey, "1", Duration.ofSeconds(IDEMPOTENT_TTL_SECONDS));
        if (Boolean.FALSE.equals(firstTime)) {
            log.warn("[login] 检测到重复请求, requestId={}, username={}",
                    request.getRequestId(), request.getUsername());
            throw new BusinessException("DUPLICATE_REQUEST", "请求正在处理中，请勿重复提交");
        }

        try {
            // ---------- Step 2: 防爆破检查 ----------
            String failCountKey = FAIL_COUNT_KEY_PREFIX + request.getUsername();
            String failCountStr = redisTemplate.opsForValue().get(failCountKey);
            int failCount = failCountStr == null ? 0 : Integer.parseInt(failCountStr);
            if (failCount >= MAX_FAIL_COUNT) {
                log.warn("[login] 账号已锁定, username={}, failCount={}, ip={}",
                        request.getUsername(), failCount, clientIp);
                throw new BusinessException("ACCOUNT_LOCKED",
                        "账号已被锁定，请15分钟后重试或联系管理员");
            }

            // ---------- Step 3: 查询用户 ----------
            UserEntity user = userMapper.selectByUsername(request.getUsername());
            if (user == null) {
                recordFailAndMaybeThrow(failCountKey, request.getUsername(), clientIp);
                throw new BusinessException("INVALID_CREDENTIAL", "用户名或密码错误");
            }
            if (user.getStatus() == 0) {
                log.warn("[login] 账号已被禁用, username={}, userId={}", request.getUsername(), user.getId());
                throw new BusinessException("ACCOUNT_DISABLED", "账号已被禁用，请联系管理员");
            }

            // ---------- Step 4: 密码校验（BCrypt，慢哈希，自带随机 salt） ----------
            boolean matched = passwordEncoder.matches(request.getPassword(), user.getPasswordHash());
            if (!matched) {
                recordFailAndMaybeThrow(failCountKey, request.getUsername(), clientIp);
                throw new BusinessException("INVALID_CREDENTIAL", "用户名或密码错误");
            }

            // 登录成功，清空失败计数
            redisTemplate.delete(failCountKey);

            // ---------- Step 5: 生成 Token（JWT） ----------
            String jti = UUID.randomUUID().toString();
            String token = jwtUtil.generateToken(user.getId(), user.getUsername(), jti, TOKEN_EXPIRE_SECONDS);

            // ---------- Step 6: 写入 Session（Redis，支持后续踢人下线） ----------
            String sessionKey = SESSION_KEY_PREFIX + jti;
            String sessionValue = String.format("{\"userId\":%d,\"username\":\"%s\",\"loginIp\":\"%s\",\"loginTime\":%d}",
                    user.getId(), user.getUsername(), clientIp, System.currentTimeMillis());
            redisTemplate.opsForValue().set(sessionKey, sessionValue, TOKEN_EXPIRE_SECONDS, TimeUnit.SECONDS);

            // 同时维护"用户 -> 当前有效jti集合"，用于单点登出/多端踢出（案例三会用到）
            String userTokenSetKey = "session:user:" + user.getId();
            redisTemplate.opsForSet().add(userTokenSetKey, jti);
            redisTemplate.expire(userTokenSetKey, TOKEN_EXPIRE_SECONDS, TimeUnit.SECONDS);

            // ---------- 审计日志（脱敏，密码绝不落日志） ----------
            log.info("[login][audit] 登录成功 userId={}, username={}, ip={}, jti={}",
                    user.getId(), user.getUsername(), clientIp, jti);

            LoginResponse response = new LoginResponse();
            response.setUserId(user.getId());
            response.setUsername(user.getUsername());
            response.setToken(token);
            response.setExpireIn(TOKEN_EXPIRE_SECONDS);
            return response;

        } finally {
            // 幂等键无论成功失败都保留一段时间，避免同一 requestId 的重试穿透
            // 若希望失败后允许立即重试，可以在异常分支主动删除 idempotentKey
        }
    }

    /**
     * 记录一次登录失败，并在达到阈值时锁定账号
     */
    private void recordFailAndMaybeThrow(String failCountKey, String username, String clientIp) {
        Long newCount = redisTemplate.opsForValue().increment(failCountKey);
        if (newCount != null && newCount == 1) {
            // 第一次失败时设置过期时间，形成滑动窗口
            redisTemplate.expire(failCountKey, LOCK_SECONDS, TimeUnit.SECONDS);
        }
        log.warn("[login] 密码校验失败, username={}, ip={}, 当前失败次数={}",
                username, clientIp, newCount);
        if (newCount != null && newCount >= MAX_FAIL_COUNT) {
            log.warn("[login][audit] 账号触发锁定, username={}, ip={}", username, clientIp);
        }
    }
}
```

```java
package com.example.auth.util;

import io.jsonwebtoken.Jwts;
import io.jsonwebtoken.SignatureAlgorithm;
import io.jsonwebtoken.security.Keys;
import org.springframework.beans.factory.annotation.Value;
import org.springframework.stereotype.Component;

import javax.crypto.SecretKey;
import java.nio.charset.StandardCharsets;
import java.util.Date;

@Component
public class JwtUtil {

    @Value("${jwt.secret}")
    private String secret;

    public String generateToken(Long userId, String username, String jti, long expireSeconds) {
        SecretKey key = Keys.hmacShaKeyFor(secret.getBytes(StandardCharsets.UTF_8));
        Date now = new Date();
        Date expireAt = new Date(now.getTime() + expireSeconds * 1000);
        return Jwts.builder()
                .setId(jti)
                .setSubject(String.valueOf(userId))
                .claim("username", username)
                .setIssuedAt(now)
                .setExpiration(expireAt)
                .signWith(key, SignatureAlgorithm.HS256)
                .compact();
    }
}
```

#### 9.2.1 业务流程图

以"用微信/企业身份第三方账号登录业务系统"为例：

```

      │ ←────────────────────                          │                       │
      │ 3.跳转到授权页，用户同意授权                       │                       │

      │                     │ 4.回调携带code+state       │                       │
      │ ←─────────────────────────────────────────────│                       │

      │                     │ 9.拿access_token查用户信息  │                       │
      │                     │ ────────────────────────→ │                       │

      │                     │ ←────────────────────────│                       │
      │                     │ 11.按unionId查/建本地账号                          │
      │                     │ ─────────────────────────────────────────────────→│
      │                     │ 12.生成本地登录态Token                             │
      │ 13.302重定向到首页并带上本地Token/Cookie                                   │
      │ ←────────────────────                          │                       │
      └─────────────────────┴──────────────────────────┴───────────────────────┘
```

> 生活类比：这就像用"微信授权登录"某个 App。你在 App 里点第三方登录（Step1），App 把你带到微信的授权页（Step2-3），微信问你"同意让 XX App 获取你的头像和昵称吗"，你点同意后微信给 App 一张"临时票据"（code），App 拿着这张票去微信后台换成正式的"访问证"（access_token），再凭证去问微信"这个人是谁"（获取用户信息），最后 App 在自己数据库里给你建一个账号或关联已有账号。

#### 9.2.2 关键设计点

```
+------------------+--------------------------------------------------------------+
|      设计点       |                          说明                                 |
+------------------+--------------------------------------------------------------+
| state 参数        | 随机字符串，跳转前生成并存入 Redis，回调时校验，防 CSRF 攻击 |
+------------------+--------------------------------------------------------------+
| PKCE（可选加固）  | code_verifier/code_challenge，防止授权码被截获重放           |
+------------------+--------------------------------------------------------------+
| code 一次性使用   | 换取 Token 后立即失效，重复使用需要报错并记录安全日志         |
+------------------+--------------------------------------------------------------+
| 账号关联幂等      | 用第三方 unionId 做唯一键，DB 层加唯一索引 + 加锁，           |
|                  | 防止并发回调导致重复创建账号                                  |
+------------------+--------------------------------------------------------------+
| client_secret    | 只能在服务端使用，绝不能出现在前端/浏览器可见的任何地方        |
+------------------+--------------------------------------------------------------+
| 异常降级          | 第三方接口超时/失败要有重试和友好提示，不能让用户卡在白屏      |
+------------------+--------------------------------------------------------------+
```

```java
                oAuthProperties.getClientId(),
                redirectUri,
                oAuthProperties.getScope(),
                state);

        log.info("[oauth] 跳转到第三方授权页, state={}", state);
```

```java
    private static final String CODE_USED_KEY_PREFIX = "oauth:code:used:";
    private static final long TOKEN_EXPIRE_SECONDS = 7200;

    // 简化示例：生产环境用分布式锁（如 Redisson），这里演示单机锁兜底思路
    private final Lock accountCreateLock = new ReentrantLock();

    private final OAuthProperties oAuthProperties;
    private final StringRedisTemplate redisTemplate;
    private final RestTemplate restTemplate;
    private final UserMapper userMapper;
    private final JwtUtil jwtUtil;

    public OAuthLoginService(OAuthProperties oAuthProperties,
                              StringRedisTemplate redisTemplate,
                              RestTemplate restTemplate,
                              UserMapper userMapper,
                              JwtUtil jwtUtil) {
        this.oAuthProperties = oAuthProperties;
        this.redisTemplate = redisTemplate;
        this.restTemplate = restTemplate;
        this.userMapper = userMapper;
        this.jwtUtil = jwtUtil;
    }

    public LoginResponse handleCallback(String code, String state) {

        // ---------- Step 1: 校验 state，防 CSRF ----------
        String stateKey = STATE_KEY_PREFIX + state;
        Boolean stateExists = redisTemplate.hasKey(stateKey);
        if (!Boolean.TRUE.equals(stateExists)) {
            log.warn("[oauth] state 校验失败，可能是CSRF攻击或state已过期, state={}", state);
            throw new BusinessException("INVALID_STATE", "登录请求已过期或非法，请重新登录");
        }
        // 校验通过立即删除，state 只能使用一次
        redisTemplate.delete(stateKey);

        // ---------- Step 2: code 幂等保护，防止 code 被重放 ----------
        String codeUsedKey = CODE_USED_KEY_PREFIX + code;
        Boolean firstUse = redisTemplate.opsForValue()
                .setIfAbsent(codeUsedKey, "1", Duration.ofMinutes(10));
        if (Boolean.FALSE.equals(firstUse)) {
            log.warn("[oauth] 检测到 code 重复使用，疑似重放攻击, code前缀={}",
                    code.substring(0, Math.min(6, code.length())));
            throw new BusinessException("CODE_REPLAY", "授权码已被使用，请重新登录");
        }

        // ---------- Step 3: 用 code 换取 access_token ----------
        OAuthTokenResponse tokenResponse = exchangeCodeForToken(code);

        // ---------- Step 4: 用 access_token 获取第三方用户信息 ----------
        OAuthUserInfo userInfo = fetchUserInfo(tokenResponse.getAccessToken());

        // ---------- Step 5: 按 unionId 创建/关联本地账号（防并发重复创建） ----------
        UserEntity user = createOrLinkLocalAccount(userInfo);

        // ---------- Step 6: 生成本地登录态 Token 并写入 Session ----------
        String jti = UUID.randomUUID().toString();
        String localToken = jwtUtil.generateToken(user.getId(), user.getUsername(), jti, TOKEN_EXPIRE_SECONDS);
        String sessionKey = "session:token:" + jti;
        redisTemplate.opsForValue().set(sessionKey,
                String.format("{\"userId\":%d,\"source\":\"oauth\",\"unionId\":\"%s\"}",
                        user.getId(), userInfo.getUnionId()),
                TOKEN_EXPIRE_SECONDS, TimeUnit.SECONDS);

        log.info("[oauth][audit] 第三方登录成功, userId={}, unionId={}, jti={}",
                user.getId(), userInfo.getUnionId(), jti);

        LoginResponse response = new LoginResponse();
        response.setUserId(user.getId());
        response.setUsername(user.getUsername());
        response.setToken(localToken);
        response.setExpireIn(TOKEN_EXPIRE_SECONDS);
        return response;
    }

    /**
     * 用授权码换取 access_token
     * 注意：client_secret 只能在服务端出现，绝不能暴露给前端
     */
    private OAuthTokenResponse exchangeCodeForToken(String code) {
        HttpHeaders headers = new HttpHeaders();
        headers.setContentType(MediaType.APPLICATION_FORM_URLENCODED);

        MultiValueMap<String, String> body = new LinkedMultiValueMap<>();
        body.add("grant_type", "authorization_code");
        body.add("code", code);
        body.add("client_id", oAuthProperties.getClientId());
        body.add("client_secret", oAuthProperties.getClientSecret());
        body.add("redirect_uri", oAuthProperties.getRedirectUri());

        HttpEntity<MultiValueMap<String, String>> entity = new HttpEntity<>(body, headers);

        try {
            OAuthTokenResponse response = restTemplate.postForObject(
                    oAuthProperties.getTokenEndpoint(), entity, OAuthTokenResponse.class);
            if (response == null || response.getAccessToken() == null) {
                throw new BusinessException("OAUTH_TOKEN_FAIL", "第三方Token交换失败");
            }
            return response;
        } catch (BusinessException e) {
            throw e;
        } catch (Exception e) {
            log.error("[oauth] code换token调用第三方接口异常", e);
            throw new BusinessException("OAUTH_TOKEN_FAIL", "第三方服务暂不可用，请稍后重试");
        }
    }

    /**
     * 用 access_token 获取第三方用户信息（unionId/openId/昵称/头像等）
     */
    private OAuthUserInfo fetchUserInfo(String accessToken) {
        try {
            String url = oAuthProperties.getUserInfoEndpoint() + "?access_token=" + accessToken;
            OAuthUserInfo userInfo = restTemplate.getForObject(url, OAuthUserInfo.class);
            if (userInfo == null || userInfo.getUnionId() == null) {
                throw new BusinessException("OAUTH_USERINFO_FAIL", "获取第三方用户信息失败");
            }
            return userInfo;
        } catch (BusinessException e) {
            throw e;
        } catch (Exception e) {
            log.error("[oauth] 获取第三方用户信息异常", e);
            throw new BusinessException("OAUTH_USERINFO_FAIL", "第三方服务暂不可用，请稍后重试");
        }
    }

    /**
     * 按 unionId 查找本地账号，不存在则创建；使用本地锁 + DB 唯一索引双重防并发，
     * 避免同一用户"同时用两个标签页扫码"导致并发创建两条重复账号记录
     */
    private UserEntity createOrLinkLocalAccount(OAuthUserInfo userInfo) {
        UserEntity existing = userMapper.selectByUnionId(userInfo.getUnionId());
        if (existing != null) {
            log.info("[oauth] 命中已关联账号, unionId={}, userId={}", userInfo.getUnionId(), existing.getId());
            return existing;
        }

        accountCreateLock.lock();
        try {
            // 加锁后二次查询（double-check），防止锁等待期间被其他线程创建
            existing = userMapper.selectByUnionId(userInfo.getUnionId());
            if (existing != null) {
                return existing;
            }

            UserEntity newUser = new UserEntity();
            newUser.setUsername("oauth_" + userInfo.getUnionId());
            newUser.setNickname(userInfo.getNickname());
            newUser.setUnionId(userInfo.getUnionId());
            newUser.setAvatar(userInfo.getAvatar());
            newUser.setStatus(1);
            // 第三方登录账号没有本地密码，随机生成不可登录的占位密码哈希
            newUser.setPasswordHash("$2a$10$" + UUID.randomUUID());

            try {
                userMapper.insert(newUser);
                log.info("[oauth][audit] 创建新账号, unionId={}, userId={}",
                        userInfo.getUnionId(), newUser.getId());
                return newUser;
            } catch (DuplicateKeyException e) {
                // 兜底：DB 唯一索引拦截了极端并发场景（跨机器/锁失效）
                log.warn("[oauth] 并发创建账号冲突，回退为查询已有账号, unionId={}", userInfo.getUnionId());
                return userMapper.selectByUnionId(userInfo.getUnionId());
            }
        } finally {
            accountCreateLock.unlock();
        }
    }
}
```

#### 9.2.4 踩坑总结

```
+----------------------+--------------------------------------------------------------+
|        坑             |                       正确做法                                |
+----------------------+--------------------------------------------------------------+
| 不校验 state          | 攻击者可以伪造回调请求，把自己的第三方账号绑定到受害者会话上， |
|                       | 必须每次跳转生成随机 state 并在回调时严格校验且只用一次       |
+----------------------+--------------------------------------------------------------+
| client_secret 暴露    | 只能在服务端持有，绝不能放在前端 JS / App 客户端代码里        |
| 在前端                |                                                                |
+----------------------+--------------------------------------------------------------+
| code 可重复使用       | 用 Redis SETNX 给 code 打上"已消费"标记，重复使用直接拒绝     |
+----------------------+--------------------------------------------------------------+
| 并发回调重复建账号     | unionId 唯一索引 + 应用层锁双保险，DuplicateKeyException 兜底 |
+----------------------+--------------------------------------------------------------+
| 第三方接口抖动导致     | 设置合理超时时间 + 有限重试，失败时返回友好提示而不是抛异常栈 |
| 用户卡死              |                                                                |
+----------------------+--------------------------------------------------------------+
```

### 9.3 案例三：Token 刷新与单点登出全链路

#### 9.3.1 业务流程图

```
  客户端(App/浏览器)          业务网关/Filter           TokenService              Redis
      │                          │                          │                     │
      │ 携带过期AccessToken请求API │                          │                     │
      │ ─────────────────────────→                          │                     │
      │                          │ Step1: 校验Token签名+过期时间                    │
      │                          │ 已过期 → 401                                    │
      │ ←─────────────────────────                          │                     │
      │                          │                          │                     │
      │ 携带RefreshToken请求刷新   │                          │                     │
      │ POST /api/token/refresh   │                          │                     │
      │ ─────────────────────────→                          │                     │
      │                          │ Step2: 转发到TokenService  │                     │
      │                          │ ─────────────────────────→│                     │
      │                          │                          │ Step3: 校验RefreshToken有效性(是否在黑名单/是否轮转过) │
      │                          │                          │ ───────────────────→│
      │                          │                          │ Step4: 生成新AccessToken+新RefreshToken(轮转) │
      │                          │                          │ Step5: 旧RefreshToken加入黑名单（防重放） │
      │                          │                          │ ───────────────────→│
      │  返回新token对             │                          │                     │
      │ ←─────────────────────────                          │                     │
      │                          │                          │                     │
      │  === 场景：用户在设备A上点击"退出所有设备"/修改密码 ===                        │
      │                          │                          │ Step6: 遍历该用户所有jti│
      │                          │                          │ ───────────────────→│
      │                          │                          │ Step7: 批量删除Session，│
      │                          │                          │ 全部加入Token黑名单     │
      │                          │                          │ ───────────────────→│
      │  设备B再次请求API（旧token）│                          │                     │
      │ ─────────────────────────→ Step8: 命中黑名单 → 401强制重新登录               │
      │ ←─────────────────────────                          │                     │
      └──────────────────────────┴──────────────────────────┴─────────────────────┘
```

> 生活类比：Access Token 像一张"当日通行证"，过期了不能直接续，得拿"月卡存根"（Refresh Token）去前台换一张新的通行证，同时旧存根作废换新存根（轮转，防止存根被偷后一直有效）。如果你挂失月卡（修改密码/主动退出所有设备），前台会把你名下所有通行证和存根都拉入黑名单，不管是哪台设备用哪张卡来，一律拒绝进门。

#### 9.3.2 关键设计点

```
+----------------------+------------------------------------------------------------+
|        设计点         |                          说明                               |
+----------------------+------------------------------------------------------------+
| Refresh Token 轮转    | 每次刷新都生成新的 Refresh Token，旧的立即失效并拉黑，       |
|                       | 防止 Refresh Token 泄露后被长期滥用                          |
+----------------------+------------------------------------------------------------+
| 黑名单机制             | Redis 存储已失效的 jti，Filter 层每次校验先查黑名单           |
+----------------------+------------------------------------------------------------+
| 幂等刷新               | 同一个 Refresh Token 短时间内并发多次刷新请求，只有第一个生效，|
|                       | 其余返回同一批新 Token（避免多个标签页同时触发刷新导致混乱）  |
+----------------------+------------------------------------------------------------+
| 多端踢出/单点登出       | 维护 "用户ID -> 所有存活jti" 集合，登出/改密时批量拉黑        |
+----------------------+------------------------------------------------------------+
| 安全校验               | 刷新时校验设备指纹/IP合理性，异常时要求二次验证（呼应第七章） |
+----------------------+------------------------------------------------------------+
```

**Token 校验 Filter：**

```java
package com.example.auth.filter;

import com.example.auth.util.JwtUtil;
import io.jsonwebtoken.Claims;
import io.jsonwebtoken.ExpiredJwtException;
import io.jsonwebtoken.JwtException;
import lombok.extern.slf4j.Slf4j;
import org.springframework.data.redis.core.StringRedisTemplate;
import org.springframework.stereotype.Component;

import javax.servlet.*;
import javax.servlet.http.HttpServletRequest;
import javax.servlet.http.HttpServletResponse;
import java.io.IOException;

@Slf4j
@Component
public class TokenAuthFilter implements Filter {

    private static final String BLACKLIST_KEY_PREFIX = "token:blacklist:";

    private final JwtUtil jwtUtil;
    private final StringRedisTemplate redisTemplate;

    public TokenAuthFilter(JwtUtil jwtUtil, StringRedisTemplate redisTemplate) {
        this.jwtUtil = jwtUtil;
        this.redisTemplate = redisTemplate;
    }

    @Override
    public void doFilter(ServletRequest req, ServletResponse res, FilterChain chain)
            throws IOException, ServletException {
        HttpServletRequest request = (HttpServletRequest) req;
        HttpServletResponse response = (HttpServletResponse) res;

        String token = extractToken(request);
        if (token == null) {
            unauthorized(response, "MISSING_TOKEN", "缺少认证凭证");
            return;
        }

        try {
            Claims claims = jwtUtil.parseAndValidate(token);
            String jti = claims.getId();

            // 校验黑名单：登出/改密/刷新轮转后的旧 token 都会进黑名单
            Boolean blacklisted = redisTemplate.hasKey(BLACKLIST_KEY_PREFIX + jti);
            if (Boolean.TRUE.equals(blacklisted)) {
                log.warn("[auth-filter] token命中黑名单，拒绝访问, jti={}", jti);
                unauthorized(response, "TOKEN_REVOKED", "登录状态已失效，请重新登录");
                return;
            }

            // 校验 Session 是否还在（可能被单点登出提前清理）
            Boolean sessionExists = redisTemplate.hasKey("session:token:" + jti);
            if (!Boolean.TRUE.equals(sessionExists)) {
                log.warn("[auth-filter] session不存在，拒绝访问, jti={}", jti);
                unauthorized(response, "SESSION_EXPIRED", "登录已过期，请重新登录");
                return;
            }

            request.setAttribute("userId", Long.parseLong(claims.getSubject()));
            chain.doFilter(request, response);

        } catch (ExpiredJwtException e) {
            log.info("[auth-filter] AccessToken已过期, jti={}", e.getClaims().getId());
            unauthorized(response, "TOKEN_EXPIRED", "登录已过期，请使用RefreshToken刷新");
        } catch (JwtException e) {
            log.warn("[auth-filter] token非法或被篡改", e);
            unauthorized(response, "TOKEN_INVALID", "非法的认证凭证");
        }
    }

    private String extractToken(HttpServletRequest request) {
        String header = request.getHeader("Authorization");
        if (header != null && header.startsWith("Bearer ")) {
            return header.substring(7);
        }
        return null;
    }

    private void unauthorized(HttpServletResponse response, String code, String message) throws IOException {
        response.setStatus(HttpServletResponse.SC_UNAUTHORIZED);
        response.setContentType("application/json;charset=UTF-8");
        response.getWriter().write(String.format(
                "{\"success\":false,\"code\":\"%s\",\"message\":\"%s\"}", code, message));
    }

    @Override
    public void init(FilterConfig filterConfig) { }

    @Override
    public void destroy() { }
}
```

```java
package com.example.auth.controller;

import com.example.auth.dto.RefreshTokenRequest;
import com.example.auth.dto.RefreshTokenResponse;
import com.example.auth.exception.BusinessException;
import com.example.auth.service.TokenService;
import lombok.extern.slf4j.Slf4j;
import org.springframework.web.bind.annotation.PostMapping;
import org.springframework.web.bind.annotation.RequestBody;
import org.springframework.web.bind.annotation.RestController;

@Slf4j
@RestController
public class TokenController {

    private final TokenService tokenService;

    public TokenController(TokenService tokenService) {
        this.tokenService = tokenService;
    }

    /**
     * 刷新 Token 接口
     * 全链路：校验RefreshToken -> 幂等去重 -> 轮转生成新Token对 -> 旧Token拉黑
     */
    @PostMapping("/api/token/refresh")
    public RefreshTokenResponse refresh(@RequestBody RefreshTokenRequest request) {
        log.info("[token-refresh] 收到刷新请求, refreshToken前缀={}",
                safePrefix(request.getRefreshToken()));
        try {
            return tokenService.refresh(request.getRefreshToken(), request.getDeviceId());
        } catch (BusinessException e) {
            log.warn("[token-refresh] 刷新失败(业务异常), code={}, msg={}", e.getCode(), e.getMessage());
            throw e;
        } catch (Exception e) {
            log.error("[token-refresh] 刷新失败(系统异常)", e);
            throw new BusinessException("SYSTEM_ERROR", "系统繁忙，请稍后重试");
        }
    }

    private String safePrefix(String token) {
        if (token == null) return "null";
        return token.length() > 8 ? token.substring(0, 8) + "***" : "***";
    }
```

**TokenService：轮转刷新 + 幂等 + 单点登出/多端踢出**

```java
package com.example.auth.service;

import com.example.auth.dto.RefreshTokenResponse;
import com.example.auth.exception.BusinessException;
import com.example.auth.util.JwtUtil;
import io.jsonwebtoken.Claims;
import io.jsonwebtoken.JwtException;
import lombok.extern.slf4j.Slf4j;
import org.springframework.data.redis.core.StringRedisTemplate;
import org.springframework.stereotype.Service;

import java.time.Duration;
import java.util.Set;
import java.util.UUID;
import java.util.concurrent.TimeUnit;

@Slf4j
@Service
public class TokenService {

    private static final String BLACKLIST_KEY_PREFIX = "token:blacklist:";
    private static final String SESSION_KEY_PREFIX = "session:token:";
    private static final String USER_TOKEN_SET_PREFIX = "session:user:";
    /** 刷新接口的幂等窗口：同一个 refreshToken 短时间内重复请求只处理一次 */
    private static final String REFRESH_LOCK_KEY_PREFIX = "token:refresh:lock:";
    private static final long REFRESH_LOCK_TTL_SECONDS = 10;

    private static final long ACCESS_TOKEN_EXPIRE_SECONDS = 7200;      // 2小时
    private static final long REFRESH_TOKEN_EXPIRE_SECONDS = 604800;   // 7天

    private final JwtUtil jwtUtil;
    private final StringRedisTemplate redisTemplate;

    public TokenService(JwtUtil jwtUtil, StringRedisTemplate redisTemplate) {
        this.jwtUtil = jwtUtil;
        this.redisTemplate = redisTemplate;
    }

    /**
     * 刷新 Token：Access Token 过期后，用 Refresh Token 换取新的一对 Token
     */
    public RefreshTokenResponse refresh(String refreshToken, String deviceId) {

        // ---------- Step 1: 幂等保护，防止并发刷新导致 refreshToken 被多次轮转、相互踢下线 ----------
        String lockKey = REFRESH_LOCK_KEY_PREFIX + shortHash(refreshToken);
        Boolean acquired = redisTemplate.opsForValue()
                .setIfAbsent(lockKey, "1", Duration.ofSeconds(REFRESH_LOCK_TTL_SECONDS));
        if (Boolean.FALSE.equals(acquired)) {
            log.warn("[token-refresh] 检测到并发刷新请求，拒绝本次，deviceId={}", deviceId);
            throw new BusinessException("REFRESH_IN_PROGRESS", "刷新请求处理中，请稍后重试");
        }

        try {
            // ---------- Step 2: 解析并校验 RefreshToken ----------
            Claims claims;
            try {
                claims = jwtUtil.parseAndValidate(refreshToken);
            } catch (JwtException e) {
                log.warn("[token-refresh] RefreshToken非法或已过期", e);
                throw new BusinessException("REFRESH_TOKEN_INVALID", "登录状态已失效，请重新登录");
            }

            String oldJti = claims.getId();
            Long userId = Long.parseLong(claims.getSubject());

            // 黑名单校验：防止已被轮转/已登出的 RefreshToken 被重放使用
            Boolean blacklisted = redisTemplate.hasKey(BLACKLIST_KEY_PREFIX + oldJti);
            if (Boolean.TRUE.equals(blacklisted)) {
                // 一个已经失效的 RefreshToken 被重新使用，是典型的"Token被盗用"信号
                log.error("[token-refresh][security] 检测到已失效RefreshToken被重放，疑似Token泄露！" +
                        " userId={}, oldJti={}, deviceId={}", userId, oldJti, deviceId);
                // 安全兜底：直接把该用户所有 Token 拉黑，强制全端下线，要求重新登录
                revokeAllTokensForUser(userId, "SUSPECTED_TOKEN_LEAK");
                throw new BusinessException("SECURITY_ALERT", "检测到异常登录行为，已强制下线，请重新登录");
            }

            // ---------- Step 3: 生成新的 Access Token + Refresh Token（轮转） ----------
            String newJti = UUID.randomUUID().toString();
            String username = claims.get("username", String.class);
            String newAccessToken = jwtUtil.generateToken(userId, username, newJti, ACCESS_TOKEN_EXPIRE_SECONDS);
            String newRefreshToken = jwtUtil.generateToken(userId, username, newJti, REFRESH_TOKEN_EXPIRE_SECONDS);

            // 新 Session 落地
            redisTemplate.opsForValue().set(SESSION_KEY_PREFIX + newJti,
                    String.format("{\"userId\":%d,\"deviceId\":\"%s\"}", userId, deviceId),
                    ACCESS_TOKEN_EXPIRE_SECONDS, TimeUnit.SECONDS);
            redisTemplate.opsForSet().add(USER_TOKEN_SET_PREFIX + userId, newJti);
            redisTemplate.expire(USER_TOKEN_SET_PREFIX + userId, REFRESH_TOKEN_EXPIRE_SECONDS, TimeUnit.SECONDS);

            // ---------- Step 4: 旧 Token 拉黑（轮转核心：旧的一律不能再用） ----------
            redisTemplate.opsForValue().set(BLACKLIST_KEY_PREFIX + oldJti, "1",
                    REFRESH_TOKEN_EXPIRE_SECONDS, TimeUnit.SECONDS);
            redisTemplate.delete(SESSION_KEY_PREFIX + oldJti);
            redisTemplate.opsForSet().remove(USER_TOKEN_SET_PREFIX + userId, oldJti);

            log.info("[token-refresh][audit] Token刷新成功, userId={}, oldJti={}, newJti={}, deviceId={}",
                    userId, oldJti, newJti, deviceId);

            RefreshTokenResponse response = new RefreshTokenResponse();
            response.setAccessToken(newAccessToken);
            response.setRefreshToken(newRefreshToken);
            response.setExpireIn(ACCESS_TOKEN_EXPIRE_SECONDS);
            return response;

        } finally {
            redisTemplate.delete(lockKey);
        }
    }

    /**
     * 单点登出/多端踢出：把用户名下所有存活的 jti 全部拉黑并清理 Session
     * 触发场景：用户主动"退出所有设备"、修改密码、检测到账号异常
     */
    public void revokeAllTokensForUser(Long userId, String reason) {
        String userTokenSetKey = USER_TOKEN_SET_PREFIX + userId;
        Set<String> jtiSet = redisTemplate.opsForSet().members(userTokenSetKey);

        if (jtiSet == null || jtiSet.isEmpty()) {
            log.info("[token-revoke] 用户当前无存活会话, userId={}, reason={}", userId, reason);
            return;
        }

        for (String jti : jtiSet) {
            redisTemplate.opsForValue().set(BLACKLIST_KEY_PREFIX + jti, "1",
                    REFRESH_TOKEN_EXPIRE_SECONDS, TimeUnit.SECONDS);
            redisTemplate.delete(SESSION_KEY_PREFIX + jti);
        }
        redisTemplate.delete(userTokenSetKey);

        log.warn("[token-revoke][audit] 用户所有Token已被强制吊销, userId={}, count={}, reason={}",
                userId, jtiSet.size(), reason);
    }

    private String shortHash(String token) {
        return String.valueOf(token.hashCode());
    }
}
```

#### 9.3.4 踩坑总结

```
+------------------------+------------------------------------------------------------+
|          坑             |                       正确做法                              |
+------------------------+------------------------------------------------------------+
| Refresh Token 永不失效  | 必须设置合理有效期（如7天），并配合轮转机制降低泄露风险       |
| 且可重复使用             |                                                              |
+------------------------+------------------------------------------------------------+
| 刷新时不做轮转，         | 每次刷新都生成新 Refresh Token 并拉黑旧的，                  |
| 一个RefreshToken用到天荒地老 | 一旦发现"已拉黑的Token被重放"，立即判定为泄露并全端下线      |
+------------------------+------------------------------------------------------------+
| 并发刷新导致相互踢下线    | 用 Redis 分布式锁给同一个 refreshToken 的刷新请求加互斥窗口   |
+------------------------+------------------------------------------------------------+
| 修改密码后旧Token仍可用  | 改密码/退出所有设备时，必须遍历该用户的 jti 集合批量拉黑      |
+------------------------+------------------------------------------------------------+
| Access Token 校验不查    | Filter 层必须同时校验：签名有效性 + 是否在黑名单 + Session   |
| 黑名单/Session           | 是否存在，三者缺一不可                                        |
+------------------------+------------------------------------------------------------+
```

### 9.4 三个案例的共性设计模式总结

```
+------------------+--------------------------------------------------------------+
|      共性模式      |                          具体体现                             |
+------------------+--------------------------------------------------------------+
| 分层异常处理       | BusinessException（预期业务错误，WARN日志）区分于             |
|                  | 系统异常（未预期错误，ERROR日志+堆栈），全局异常处理器统一收口 |
+------------------+--------------------------------------------------------------+
| 幂等控制三板斧     | Redis SETNX（请求级） + 唯一索引（数据级） +                  |
|                  | 分布式锁/本地锁 double-check（并发级），三者按场景组合使用    |
+------------------+--------------------------------------------------------------+
| 审计日志规范       | 关键动作（登录成功/失败/Token吊销）打 [audit] 标签，          |
|                  | 敏感信息（密码/完整Token）一律脱敏或不落盘                    |
+------------------+--------------------------------------------------------------+
| 防重放/防篡改       | state 一次性、code 一次性、Refresh Token 轮转，               |
|                  | 核心思想都是"用过就作废，重放即拒绝并告警"                     |
+------------------+--------------------------------------------------------------+
| 快速失败 + 友好提示  | 校验类失败（参数/state/黑名单）直接短路返回，                  |
|                  | 不做无意义的后续调用，同时给用户可理解的提示而非堆栈信息        |
+------------------+--------------------------------------------------------------+
```

> 这三个案例分别对应了认证链路里最高频的三类问题：**"怎么安全地验证一个人"**（案例一）、**"怎么信任另一个系统验证过的身份"**（案例二）、**"怎么在不打断用户体验的前提下持续保证安全"**（案例三）。把这三条链路的代码模式吃透，就能应对绝大多数登录认证系统的工程落地需求。
