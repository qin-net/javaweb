## 2. 总则

本文档是期刊投稿管理系统开发过程中 **必须严格遵守** 的编码规范与要求。所有开发成员在编写代码前须通读本文档，任何违反本文档要求的代码在代码审查中将被退回修改。

### 2.1 核心原则

| 编号 | 原则 | 说明 |
|------|------|------|
| P-01 | 分层架构 | 严格遵循 Model → Interface → DAO → Impl → Service 五层架构，禁止跨层调用 |
| P-02 | 接口编程 | 面向接口编程，Service 层通过接口引用调用 Impl，禁止直接依赖实现类 |
| P-03 | 泛型集合 | 所有集合操作必须使用泛型，禁止使用原始类型（Raw Type） |
| P-04 | 公用类复用 | 公共逻辑抽取到工具类（DBUtil、DateUtil 等），禁止在业务代码中重复编写 |
| P-05 | 异常处理 | 所有异常必须被捕获或声明抛出，禁止空 catch 块 |
| P-06 | 非法输入识别 | 所有用户输入必须经过校验，禁止将未校验的输入直接传入业务逻辑 |
| P-07 | 文档注释 | 每个类、接口、方法必须有完整的 Javadoc 注释 |

## 3. 文档注释规范

### 3.1 总体要求

- **每个 Java 源文件** 的类声明前必须有 Javadoc 注释
- **每个方法** 声明前必须有 Javadoc 注释
- 注释必须包含 **编写者** 和 **完成时间**，不可遗漏
- 一个包一般由一名成员负责，编写者字段填写该包负责人标识

### 3.2 类级注释模板

```java
/**
 * [类的功能简述]
 *
 * 编写者：[包负责人姓名/标识]
 * 完成时间：[yyyy-MM-dd]
 * 所属包：[完整包名]
 *
 * 功能详述：
 *   [详细描述该类的职责和核心功能]
 *
 * 依赖：
 *   [列出该类依赖的其他类/接口，如无特殊依赖可省略]
 */
public class ClassName {
    // ...
}
```

### 3.3 接口级注释模板

```java
/**
 * [接口的功能简述]
 *
 * 编写者：[包负责人姓名/标识]
 * 完成时间：[yyyy-MM-dd]
 * 所属包：[完整包名]
 *
 * 功能详述：
 *   [详细描述该接口定义的业务契约]
 *
 * 实现类：
 *   [列出实现该接口的类，如 impl/XXXImpl.java]
 */
public interface InterfaceName {
    // ...
}
```

### 3.4 方法级注释模板

```java
/**
 * [方法功能简述]
 *
 * 编写者：[包负责人姓名/标识]
 * 完成时间：[yyyy-MM-dd]
 *
 * @param paramName 参数说明
 * @return 返回值说明
 * @throws BusinessException 业务异常说明（如有）
 */
public ReturnType methodName(ParamType paramName) {
    // ...
}
```

### 3.5 各包负责人与注释填写规范

| 包名 | 包路径 | 负责人标识 | 注释中编写者填写示例 |
|------|--------|-----------|---------------------|
| Model | `com.journal.model` | 成员A | `编写者：成员A` |
| Interface | `com.journal.interfacemodel` | 成员B | `编写者：成员B` |
| DAO | `com.journal.dao` | 成员C | `编写者：成员C` |
| Impl | `com.journal.impl` | 成员D | `编写者：成员D` |
| Service | `com.journal.service` | 成员E | `编写者：成员E` |
| Main + Exception + Util | `com.journal.main` / `com.journal.exception` / `com.journal.util` | 成员F | `编写者：成员F` |

> **扩展规则**：若后续新增包（如 `com.journal.filter`、`com.journal.cache` 等），须在本文档中补充包负责人分配，并遵循相同的注释规范。

### 3.6 注释时间填写规则

| 场景 | 填写规则 |
|------|----------|
| 首次编写 | 填写实际完成日期，如 `2026-07-15` |
| 后续修改 | 在方法注释末尾追加修改记录：`修改者：xxx，修改时间：yyyy-MM-dd，修改内容：xxx` |
| 包负责人变更 | 更新类注释中的编写者为新负责人，原编写者信息保留在修改记录中 |

### 4.2 各层职责与约束

| 层 | 包名 | 职责 | 允许依赖 | 禁止事项 |
|----|------|------|----------|----------|
| Model | `model` | 定义实体类（POJO），对应数据库表 | 无外部依赖 | 禁止包含业务逻辑；禁止依赖其他业务包 |
| Interface | `interfacemodel` | 定义业务接口契约 | 可引用 `model` | 禁止包含实现逻辑；禁止依赖 `dao`/`impl`/`service` |
| DAO | `dao` | 数据库 CRUD 操作 | 可引用 `model`、`util` | 禁止包含业务逻辑判断；禁止被 Service 以外的层直接调用 |
| Impl | `impl` | 实现接口定义的方法 | 可引用 `model`、`interfacemodel`、`dao` | 禁止绕过 DAO 直接操作数据库；禁止包含复杂业务编排 |
| Service | `service` | 业务逻辑编排与校验 | 可引用 `model`、`interfacemodel`、`impl`、`exception` | 禁止直接操作数据库；禁止包含 SQL 语句 |
| Util | `util` | 公共工具方法 | 仅依赖 JDBC 驱动 | 禁止包含业务逻辑 |
| Exception | `exception` | 自定义异常类 | 无 | 禁止包含业务逻辑 |
| Main | `main` | 控制台菜单与用户交互 | 可引用 `service`、`model`、`exception` | 禁止直接调用 DAO；禁止包含业务逻辑 |

### 4.3 层间调用规则

```
允许的调用方向（自上而下）：

Main ──→ Service ──→ Impl ──→ DAO ──→ DBUtil
                │                ↑
                └──→ Interface ──┘ （接口约束）
Model ←── 被所有层引用

禁止的调用方向：
  ✗ DAO → Service（下层禁止调用上层）
  ✗ Impl → Service（实现层禁止调用服务层）
  ✗ Model → 任何业务包（实体层禁止反向依赖）
  ✗ Main → DAO（入口层禁止跨层调用）
```

### 4.4 包扩展规则

当系统需要新增功能模块时，须遵循以下扩展流程：

1. 在 `model` 包中新增对应实体类
2. 在 `interfacemodel` 包中新增对应接口
3. 在 `dao` 包中新增对应 DAO 类
4. 在 `impl` 包中新增对应实现类
5. 在 `service` 包中新增对应服务类
6. 更新本文档中的包负责人分配表

> **禁止** 在不完整的分层下提交代码（如新增了 DAO 但没有对应 Interface）。

## 5. JDBC 与 DAO 模式要求

### 5.1 数据库连接

| 要求项 | 规范 |
|--------|------|
| 数据库 | MySQL |
| 驱动方式 | JDBC 纯 Java 驱动（`com.mysql.jdbc.Driver` 或 `com.mysql.cj.jdbc.Driver`） |
| 连接管理 | 统一通过 `DBUtil` 工具类获取和关闭连接 |
| 连接参数 | 在 `DBUtil` 中以常量形式定义，禁止硬编码在业务代码中 |

### 5.2 DAO 模式强制要求

| 编号 | 要求 | 说明 |
|------|------|------|
| DAO-01 | 每个 DAO 类对应一张数据库表 | 如 `ManuscriptDAO` 对应 `manuscript` 表 |
| DAO-02 | 所有 SQL 操作必须使用 `PreparedStatement` | 禁止使用 `Statement`，防止 SQL 注入 |
| DAO-03 | 参数绑定使用占位符 `?` | 禁止字符串拼接 SQL |
| DAO-04 | 查询结果使用 `ResultSet` 映射到实体对象 | 禁止直接返回 `ResultSet` |
| DAO-05 | 资源关闭必须在 `finally` 块中执行 | 确保连接、语句、结果集始终被关闭 |
| DAO-06 | 事务操作须显式管理 | 使用 `setAutoCommit(false)` + `commit()` + `rollback()` |
| DAO-07 | 批量操作使用 `addBatch()` / `executeBatch()` | 如参考文献批量插入 |

### 5.3 PreparedStatement 使用规范

```java
// ✓ 正确：使用占位符
String sql = "SELECT * FROM author WHERE name = ? AND password = ?";
pstmt = conn.prepareStatement(sql);
pstmt.setString(1, name);
pstmt.setString(2, password);

// ✗ 错误：字符串拼接
String sql = "SELECT * FROM author WHERE name = '" + name + "'";
```

### 5.4 资源关闭规范

```java
Connection conn = null;
PreparedStatement pstmt = null;
ResultSet rs = null;
try {
    conn = DBUtil.getConnection();
    // ... 数据库操作 ...
} catch (SQLException e) {
    // 异常处理
} finally {
    DBUtil.close(conn, pstmt, rs);  // 必须在 finally 中关闭
}
```

> **强制要求**：所有 DAO 方法必须遵循上述 try-catch-finally 结构，无例外。

### 5.5 事务处理规范

涉及多表操作（如投稿：稿件 + 参考文献）时，必须使用事务：

```java
try {
    conn = DBUtil.getConnection();
    conn.setAutoCommit(false);  // 开启事务
    // ... 多步数据库操作 ...
    conn.commit();              // 提交事务
} catch (SQLException e) {
    try {
        if (conn != null) conn.rollback();  // 回滚事务
    } catch (SQLException ex) {
        ex.printStackTrace();
    }
} finally {
    DBUtil.close(conn, pstmt, rs);
}
```

## 6. 泛型集合使用要求

### 6.1 强制使用泛型

| 编号 | 要求 | 正确示例 | 错误示例 |
|------|------|----------|----------|
| G-01 | List 必须指定泛型类型 | `List<Manuscript>` | `List` |
| G-02 | Map 必须指定键值泛型 | `Map<String, Object>` | `Map` |
| G-03 | 方法返回值使用泛型集合 | `public List<Manuscript> findAll()` | `public List findAll()` |
| G-04 | 方法参数使用泛型集合 | `public boolean submit(Manuscript m, List<Reference> refs)` | `public boolean submit(Manuscript m, List refs)` |

### 6.2 集合类型选择

| 场景 | 推荐集合类型 | 说明 |
|------|-------------|------|
| 查询结果列表 | `ArrayList<T>` | 随机访问效率高，适合查询结果 |
| 需要去重 | `HashSet<T>` | 自动去重 |
| 键值映射 | `HashMap<K, V>` | 通用映射场景 |
| 实体中组合关系 | `List<T>` | 如 `Manuscript` 中的 `List<Reference>` |

### 6.3 禁止事项

- 禁止使用原始类型集合（如 `ArrayList` 不带泛型）
- 禁止使用 `@SuppressWarnings("unchecked")` 绕过泛型检查
- 禁止将 `Object` 作为集合元素类型

## 7. 接口使用要求

### 7.1 接口定义规范

| 编号 | 要求 | 说明 |
|------|------|------|
| IF-01 | 每个业务模块必须定义接口 | 在 `interfacemodel` 包中定义 |
| IF-02 | 接口方法签名须明确参数和返回类型 | 使用泛型集合作为返回类型 |
| IF-03 | 接口命名统一后缀为 `Interface` | 如 `ManuscriptInterface` |
| IF-04 | 实现类命名统一后缀为 `Impl` | 如 `ManuscriptImpl` |

### 7.2 面向接口编程

```java
// ✓ 正确：Service 层通过接口引用调用实现
public class ManuscriptService {
    private ManuscriptInterface manuscriptImpl = new ManuscriptImpl();

    public void publishManuscript(int id, int journalId, Date date) {
        Manuscript m = manuscriptImpl.findById(id);
        // ...
    }
}

// ✗ 错误：直接依赖实现类
public class ManuscriptService {
    private ManuscriptImpl manuscriptImpl = new ManuscriptImpl();
}
```

### 7.3 接口方法定义要求

- 方法命名须语义清晰：`insert`、`update`、`delete`、`findById`、`findAll`
- 查询方法返回泛型集合：`List<Manuscript>` 而非 `List`
- 增删改方法返回受影响行数或布尔值，明确操作结果
- 须在接口注释中说明每个方法的业务含义

## 9. 非法输入识别与处理

### 9.1 输入校验总则

**所有来自用户的输入均不可信**，必须在进入业务逻辑前完成校验。

### 9.2 校验场景与规则

| 编号 | 校验场景 | 非法情况 | 处理方式 | 涉及 Java 知识点 |
|------|----------|----------|----------|------------------|
| V-01 | 菜单选择输入 | 输入非数字字符 | 捕获 `InputMismatchException`，清空缓冲区，提示重新输入 | `try-catch`、`Scanner.next()` |
| V-02 | 菜单选择范围 | 输入数字不在有效范围内 | 提示"请输入有效选项"，重新输入 | `if-else` 条件判断 |
| V-03 | 空字符串输入 | 输入仅含空格或为空 | `trim()` 后判断 `isEmpty()`，提示不能为空 | `String.trim()`、`String.isEmpty()` |
| V-04 | 必填字段为空 | 题目、姓名、密码等必填项未填 | 提示该字段为必填项，要求重新输入 | `String` 方法 |
| V-05 | 日期格式错误 | 输入日期格式不符合 `yyyy-MM-dd` | 捕获 `ParseException`，提示正确格式 | `SimpleDateFormat.parse()` |
| V-06 | 数字格式错误 | 需要数字但输入了非数字 | 捕获 `NumberFormatException`，提示重新输入 | `Integer.parseInt()` |
| V-07 | 邮箱格式错误 | 邮箱不符合基本格式 | 正则校验，提示正确格式 | `String.matches()` |
| V-08 | 字符长度超限 | 输入超出数据库字段长度 | 截断或提示超长，要求重新输入 | `String.length()` |
| V-09 | 稿件状态非法 | 状态值不在预定义范围内 | 使用 `ManuscriptStatus` 常量校验 | 常量类 |
| V-10 | SQL 注入风险 | 输入含 SQL 特殊字符 | 使用 `PreparedStatement` 参数绑定 | `PreparedStatement` |

### 9.3 输入校验代码模板

```java
// 菜单输入校验模板
Scanner scanner = new Scanner(System.in);
int choice = -1;
while (true) {
    try {
        System.out.print("请选择：");
        String input = scanner.nextLine().trim();
        if (input.isEmpty()) {
            System.out.println("输入不能为空，请重新输入！");
            continue;
        }
        choice = Integer.parseInt(input);
        if (choice < 0 || choice > maxOption) {
            System.out.println("请输入有效选项（0-" + maxOption + "）！");
            continue;
        }
        break;
    } catch (NumberFormatException e) {
        System.out.println("请输入数字！");
    }
}
```

### 9.4 Service 层业务校验

Service 层须对业务参数进行二次校验：

```java
public void publishManuscript(int manuscriptId, int journalId, Date publishDate) {
    if (manuscriptId <= 0) {
        throw new BusinessException("稿件ID无效");
    }
    if (journalId <= 0) {
        throw new BusinessException("期刊ID无效");
    }
    if (publishDate == null) {
        throw new BusinessException("发表日期不能为空");
    }
    Manuscript m = manuscriptImpl.findById(manuscriptId);
    if (m == null) {
        throw new BusinessException("稿件不存在");
    }
    if (!"已收录".equals(m.getStatus())) {
        throw new BusinessException("只有已收录的稿件才能发表");
    }
    // ... 执行业务逻辑
}
```

## 10. 异常处理要求

### 10.1 异常处理总则

| 编号 | 要求 | 说明 |
|------|------|------|
| EH-01 | 禁止空 catch 块 | catch 块中必须有处理逻辑（至少打印日志） |
| EH-02 | 禁止吞没异常 | 禁止 catch 后不做任何处理 |
| EH-03 | finally 中关闭资源 | 数据库连接等资源必须在 finally 中释放 |
| EH-04 | 业务异常使用自定义类 | 使用 `BusinessException` 封装业务错误 |
| EH-05 | DAO 层异常向上传播 | DAO 层捕获 `SQLException` 后包装为 `BusinessException` 抛出 |
| EH-06 | Main 层友好提示 | 控制台捕获异常后显示用户友好的提示信息 |

### 10.2 异常分类与处理策略

| 异常类型 | 触发场景 | 处理层级 | 处理方式 |
|----------|----------|----------|----------|
| `ClassNotFoundException` | JDBC 驱动加载失败 | DBUtil | 打印错误信息，程序终止 |
| `SQLException` | 数据库连接/操作失败 | DAO | 包装为 `BusinessException` 抛出，回滚事务 |
| `InputMismatchException` | 控制台输入类型不匹配 | Main | 捕获后清空缓冲区，提示重新输入 |
| `NumberFormatException` | 数字解析失败 | Main / Service | 提示输入正确格式，要求重新输入 |
| `ParseException` | 日期格式解析失败 | Main / DateUtil | 提示正确日期格式，要求重新输入 |
| `NullPointerException` | 查询结果为空 | Service | 返回空集合或抛出 `BusinessException` |
| `BusinessException` | 业务逻辑违规 | Service 抛出，Main 捕获 | 显示异常信息给用户 |

### 10.3 自定义异常类

```java
/**
 * 业务异常类
 *
 * 编写者：成员F
 * 完成时间：2026-07-xx
 * 所属包：com.journal.exception
 *
 * 功能详述：
 *   封装业务逻辑校验中的异常信息，用于 Service 层向 Main 层传递业务错误
 */
public class BusinessException extends RuntimeException {
    public BusinessException(String message) {
        super(message);
    }

    public BusinessException(String message, Throwable cause) {
        super(message, cause);
    }
}
```

### 10.4 DAO 层异常处理模板

```java
try {
    conn = DBUtil.getConnection();
    // ... 数据库操作 ...
} catch (SQLException e) {
    e.printStackTrace();
    throw new BusinessException("数据库操作失败：" + e.getMessage(), e);
} finally {
    DBUtil.close(conn, pstmt, rs);
}
```

## 13. 违规等级与处理

| 违规等级 | 说明 | 示例 | 处理方式 |
|----------|------|------|----------|
| **严重** | 违反核心架构原则 | 跨层调用、缺少接口定义、使用 Statement 拼接 SQL | 必须立即修改，代码审查不通过 |

| **一般** | 不影响功能但不符合最佳实践 | 注释不够详细、变量命名不规范 | 建议修改，可后续优化 |
