# 权限系统架构设计

## 一、系统概述

### 1.1 业务场景

权限系统是企业级应用的基础设施，核心职责是回答两个问题："谁能做什么"（功能权限）和"谁能看什么"（数据权限）。典型场景如下：

- 普通员工只能查看自己创建的订单，部门经理可以查看本部门所有订单，总监可以查看所有部门订单
- 运营人员可以编辑商品信息，但不能删除；管理员可以执行所有操作
- 不同部门的用户登录后看到的菜单和功能按钮不同
- 敏感数据（如薪资、合同）只有特定角色才能访问

权限系统通常包含三个维度：
1. **菜单权限**：用户能看到哪些菜单和页面
2. **操作权限**：用户能执行哪些功能按钮（如新增、删除、导出）
3. **数据权限**：用户能看到哪些范围的数据（全部、本部门、仅本人）

### 1.2 核心挑战

| 挑战维度 | 具体问题 | 设计目标 |
|---------|---------|---------|
| 模型选择 | ACL/RBAC/ABAC各有优劣，如何选择最合适的模型 | 兼顾灵活性和可维护性 |
| 性能 | 每次请求都需要鉴权，不能成为性能瓶颈 | 鉴权延迟 < 5ms |
| 灵活性 | 权限配置要灵活，支持角色继承、权限组合 | 运行时可动态调整 |
| 可维护性 | 权限规则复杂后，维护难度急剧上升 | 权限规则可视化管理 |
| 实时生效 | 权限变更后需要立即生效，不能有延迟 | 秒级生效 |
| 数据权限 | 不同角色看到的数据范围不同，需要动态拼接SQL | 对业务代码无侵入 |
| 安全性 | 权限绕过、越权访问是重大安全漏洞 | 多层校验，纵深防御 |

## 二、权限模型设计

### 2.1 三种权限模型对比

业界主流的权限模型有三种：ACL、RBAC、ABAC。它们各有优劣，适用于不同场景。

#### 2.1.1 ACL模型（Access Control List，访问控制列表）

**核心思想**：规定资源可以被哪些主体进行哪些操作。围绕"资源"来设定权限。

```
┌──────────┐           ┌──────────┐
│  用户A    │──读取──→  │ 文件X    │
│  用户B    │──写入──→  │ 文件X    │
│  用户C    │──删除──→  │ 文件Y    │
└──────────┘           └──────────┘
    主体                    资源
```

**数据模型**：

```sql
-- ACL模型的核心表：每一行表示"某主体对某资源有某操作权限"
CREATE TABLE acl_entry (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    subject_id BIGINT NOT NULL COMMENT '主体ID（用户或用户组）',
    subject_type VARCHAR(16) NOT NULL COMMENT 'USER/GROUP',
    resource_id BIGINT NOT NULL COMMENT '资源ID',
    resource_type VARCHAR(32) NOT NULL COMMENT '资源类型（FILE/ORDER/MENU）',
    operation VARCHAR(16) NOT NULL COMMENT '操作类型（READ/WRITE/DELETE）',
    effect TINYINT NOT NULL DEFAULT 1 COMMENT '1允许 0拒绝',
    INDEX idx_subject (subject_id, subject_type),
    INDEX idx_resource (resource_id, resource_type)
) ENGINE=InnoDB;
```

**Java实现**：

```java
/**
 * ACL模型实现
 * 
 * 算法：哈希表查找，时间复杂度O(1)
 * 设计模式：策略模式（Strategy Pattern）—— 将ACL校验封装为独立策略
 */
public class AclPermissionChecker {

    /**
     * 使用HashMap存储ACL表，key为"主体ID:资源类型:资源ID"，value为操作集合
     * 算法选择：哈希表查找O(1)，适合快速判断单一资源的权限
     */
    private final Map<String, Set<String>> aclTable = new ConcurrentHashMap<>();

    /**
     * 加载ACL表到内存
     */
    public void loadAcl(List<AclEntry> entries) {
        for (AclEntry entry : entries) {
            String key = buildKey(entry.getSubjectId(),
                                  entry.getResourceType(),
                                  entry.getResourceId());
            aclTable.computeIfAbsent(key, k -> ConcurrentHashMap.newKeySet())
                    .add(entry.getOperation());
        }
    }

    /**
     * 检查权限：用户是否有权对某资源执行某操作
     */
    public boolean hasPermission(Long subjectId, String resourceType,
                                  Long resourceId, String operation) {
        String key = buildKey(subjectId, resourceType, resourceId);
        Set<String> operations = aclTable.get(key);
        return operations != null && operations.contains(operation);
    }

    private String buildKey(Long subjectId, String resourceType, Long resourceId) {
        return subjectId + ":" + resourceType + ":" + resourceId;
    }
}
```

**优点**：
- 快速检查用户是否有权限访问特定资源，O(1)时间复杂度
- 适合文件系统等资源粒度明确的场景（Linux文件权限就是ACL）

**缺点**：
- 需要维护大量的"主体-资源-操作"关联记录，N个用户 * M个资源 = N*M条记录
- 难以批量授权：给100个用户授权访问同一资源，需要插入100条记录
- 权限管理分散在各个资源上，全局视角缺失

#### 2.1.2 RBAC模型（Role-Based Access Control，基于角色的访问控制）

**核心思想**：引入"角色"作为中间层，规定角色可以对哪些资源进行哪些操作，再将角色分配给用户。用户通过激活角色来获得访问权限。

```
┌──────────┐         ┌──────────┐         ┌──────────┐
│  用户     │──N:N──→│  角色     │──N:N──→│  权限     │
│  · 张三   │         │  · 管理员 │         │  · 订单查看│
│  · 李四   │         │  · 运营   │         │  · 订单删除│
│  · 王五   │         │  · 客服   │         │  · 用户管理│
└──────────┘         └──────────┘         └──────────┘
```

**模型的核心是角色**，围绕角色做权限的分配。用户和权限通过角色解耦，赋权更灵活。给100个用户授权同一组权限，只需要创建一个角色，然后把100个用户分配到这个角色下即可。

**RBAC的四个层次**：

| 层次 | 名称 | 特性 | 说明 |
|------|------|------|------|
| RBAC0 | 基础模型 | 用户-角色-权限 | 最基本的RBAC，三要素多对多关系 |
| RBAC1 | 角色继承 | 角色层次关系 | 高级角色继承低级角色的权限 |
| RBAC2 | 约束模型 | 互斥角色/基数约束 | 同一用户不能同时拥有互斥角色 |
| RBAC3 | 统一模型 | RBAC1 + RBAC2 | 同时支持角色继承和约束 |

#### 2.1.3 ABAC模型（Attribute-Based Access Control，基于属性的访问控制）

**核心思想**：规定哪些属性的主体可以对哪些属性的资源在哪些属性的情况下进行哪些操作。一个请求逐条匹配策略，没有匹配到则返回默认效果（拒绝）。

ABAC的四类属性：
- **主体属性**（Subject Attributes）：年龄、性别、职位、部门、安全等级
- **资源属性**（Resource Attributes）：创建时间、密级、所属部门、文件类型
- **环境属性**（Environment Attributes）：当前时间、IP地址、地理位置、设备类型
- **操作属性**（Action Attributes）：读、写、删除、审批

```java
/**
 * ABAC模型实现
 *
 * 算法：策略链逐条匹配，类似防火墙规则匹配
 * 设计模式：解释器模式（Interpreter Pattern）—— 将策略规则解析为可执行的表达式
 *          + 责任链模式（Chain of Responsibility）—— 策略链依次匹配
 */
public class AbacPolicyEngine {

    private final List<AbacPolicy> policies = new ArrayList<>();

    /**
     * 策略定义
     */
    @Data
    public static class AbacPolicy {
        private String policyId;
        private String description;
        private AbacEffect effect;       // ALLOW 或 DENY
        private AbacCondition condition;  // 匹配条件
    }

    public enum AbacEffect {
        ALLOW, DENY
    }

    /**
     * 访问请求上下文
     */
    @Data
    public static class AccessRequest {
        private Map<String, Object> subjectAttributes;   // 主体属性
        private Map<String, Object> resourceAttributes;  // 资源属性
        private Map<String, Object> environmentAttributes; // 环境属性
        private String action;                            // 操作
    }

    /**
     * 策略评估：逐条匹配策略链
     * 
     * 算法描述：
     * 1. 遍历所有策略，逐条检查条件是否匹配
     * 2. 第一条匹配的策略决定最终结果（First-Applicable策略合并算法）
     * 3. 如果没有任何策略匹配，返回默认效果（DENY）
     */
    public AbacEffect evaluate(AccessRequest request) {
        for (AbacPolicy policy : policies) {
            if (policy.getCondition().matches(request)) {
                return policy.getEffect();
            }
        }
        // 默认拒绝（Deny-by-Default原则）
        return AbacEffect.DENY;
    }

    /**
     * 条件接口：可组合的布尔表达式
     * 设计模式：组合模式（Composite Pattern）—— 简单条件和复合条件统一接口
     */
    public interface AbacCondition {
        boolean matches(AccessRequest request);
    }

    /**
     * 属性等值匹配条件
     */
    public static class AttributeEquals implements AbacCondition {
        private final String attributeCategory; // subject/resource/environment
        private final String attributeName;
        private final Object expectedValue;

        public AttributeEquals(String category, String name, Object value) {
            this.attributeCategory = category;
            this.attributeName = name;
            this.expectedValue = value;
        }

        @Override
        public boolean matches(AccessRequest request) {
            Map<String, Object> attrs = getAttributes(request, attributeCategory);
            Object actualValue = attrs.get(attributeName);
            return expectedValue.equals(actualValue);
        }

        private Map<String, Object> getAttributes(AccessRequest req, String category) {
            switch (category) {
                case "subject": return req.getSubjectAttributes();
                case "resource": return req.getResourceAttributes();
                case "environment": return req.getEnvironmentAttributes();
                default: return Collections.emptyMap();
            }
        }
    }

    /**
     * AND组合条件：所有子条件都满足
     */
    public static class AndCondition implements AbacCondition {
        private final List<AbacCondition> conditions;

        public AndCondition(List<AbacCondition> conditions) {
            this.conditions = conditions;
        }

        @Override
        public boolean matches(AccessRequest request) {
            return conditions.stream().allMatch(c -> c.matches(request));
        }
    }

    /**
     * OR组合条件：任一子条件满足
     */
    public static class OrCondition implements AbacCondition {
        private final List<AbacCondition> conditions;

        public OrCondition(List<AbacCondition> conditions) {
            this.conditions = conditions;
        }

        @Override
        public boolean matches(AccessRequest request) {
            return conditions.stream().anyMatch(c -> c.matches(request));
        }
    }
}
```

**使用示例**：

```java
// 策略示例：只有"部门=技术部"且"职级>=P6"的用户，在"工作时间(9:00-18:00)"，
// 才能访问"密级=机密"的文档
AbacPolicy policy = new AbacPolicy();
policy.setPolicyId("policy-001");
policy.setDescription("技术部高级员工可在工作时间访问机密文档");
policy.setEffect(AbacEffect.ALLOW);
policy.setCondition(new AndCondition(Arrays.asList(
    new AttributeEquals("subject", "department", "技术部"),
    new AttributeEquals("resource", "securityLevel", "机密"),
    new AttributeEquals("environment", "isWorkingHour", true)
)));
```

### 2.2 三种模型对比与选择

| 维度 | ACL | RBAC | ABAC |
|------|-----|------|------|
| 核心概念 | 资源访问列表 | 角色 | 属性 |
| 权限粒度 | 资源级 | 角色级 | 属性级 |
| 授权方式 | 直接授权到用户 | 通过角色间接授权 | 通过策略规则授权 |
| 批量授权 | 困难 | 简单 | 中等 |
| 灵活性 | 低 | 中 | 高 |
| 实现复杂度 | 低 | 中 | 高 |
| 维护成本 | 高（大量关联记录） | 中 | 中（策略规则管理） |
| 适用场景 | 文件系统、简单应用 | 企业管理系统 | 复杂动态策略 |
| 典型产品 | Linux文件权限、AWS S3 | 企业ERP、后台管理 | AWS IAM、K8s RBAC+ABAC |

### 2.3 本系统的模型选择：RBAC为核心 + ABAC扩展

对于大多数企业级B端系统，**RBAC是最佳选择**，原因如下：

1. **角色抽象贴近业务**：企业天然有"管理员"、"运营"、"客服"等角色概念
2. **批量授权高效**：新员工入职，分配角色即可获得一组权限
3. **可维护性好**：权限变更只需修改角色的权限集合，所有拥有该角色的用户自动生效
4. **实现成熟**：RBAC有标准规范（NIST RBAC标准），业界实践丰富

同时引入ABAC思想作为补充，处理RBAC难以覆盖的动态场景（如基于时间、IP的访问控制）。

## 三、RBAC模型实现

### 3.1 数据库设计

RBAC模型的核心是五张表：用户表、角色表、权限表、用户-角色关联表、角色-权限关联表。再加上部门表和角色数据范围表来支持数据权限。

```sql
-- ========================================
-- RBAC核心表结构（共7张表）
-- ========================================

-- 1. 用户表
CREATE TABLE sys_user (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL UNIQUE COMMENT '用户ID（业务主键）',
    username VARCHAR(64) NOT NULL COMMENT '用户名',
    password VARCHAR(128) NOT NULL COMMENT '密码（BCrypt加密）',
    real_name VARCHAR(32) COMMENT '真实姓名',
    email VARCHAR(64) COMMENT '邮箱',
    phone VARCHAR(16) COMMENT '手机号',
    dept_id BIGINT COMMENT '所属部门ID',
    status TINYINT DEFAULT 1 COMMENT '状态：1启用 0禁用',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_dept_id (dept_id),
    INDEX idx_status (status)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户表';

-- 2. 角色表
CREATE TABLE sys_role (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    role_id BIGINT NOT NULL UNIQUE COMMENT '角色ID',
    role_name VARCHAR(64) NOT NULL COMMENT '角色名称',
    role_code VARCHAR(64) NOT NULL COMMENT '角色编码（唯一标识）',
    parent_role_id BIGINT DEFAULT 0 COMMENT '父角色ID（用于角色继承，0表示顶级）',
    data_scope TINYINT DEFAULT 4 COMMENT '数据范围：1全部 2本部门 3本部门及以下 4仅本人 5自定义',
    sort_order INT DEFAULT 0 COMMENT '排序号',
    status TINYINT DEFAULT 1 COMMENT '状态：1启用 0禁用',
    remark VARCHAR(256) COMMENT '备注',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY uk_role_code (role_code),
    INDEX idx_parent_role (parent_role_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='角色表';

-- 3. 权限表（树形结构）
CREATE TABLE sys_permission (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    perm_id BIGINT NOT NULL UNIQUE COMMENT '权限ID',
    parent_id BIGINT DEFAULT 0 COMMENT '父权限ID（0表示顶级节点）',
    perm_name VARCHAR(64) NOT NULL COMMENT '权限名称',
    perm_code VARCHAR(128) NOT NULL COMMENT '权限编码，格式：模块:操作，如order:view',
    perm_type TINYINT NOT NULL COMMENT '权限类型：1菜单 2按钮 3API 4数据',
    path VARCHAR(256) COMMENT 'API路径或前端路由',
    method VARCHAR(16) COMMENT 'HTTP方法：GET/POST/PUT/DELETE',
    icon VARCHAR(64) COMMENT '菜单图标',
    sort_order INT DEFAULT 0 COMMENT '排序号',
    visible TINYINT DEFAULT 1 COMMENT '是否可见：1可见 0隐藏',
    status TINYINT DEFAULT 1 COMMENT '状态：1启用 0禁用',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uk_perm_code (perm_code),
    INDEX idx_parent_id (parent_id),
    INDEX idx_perm_type (perm_type)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='权限表（树形结构）';

-- 4. 用户-角色关联表
CREATE TABLE sys_user_role (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL COMMENT '用户ID',
    role_id BIGINT NOT NULL COMMENT '角色ID',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uk_user_role (user_id, role_id),
    INDEX idx_role_id (role_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户-角色关联表';

-- 5. 角色-权限关联表
CREATE TABLE sys_role_permission (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    role_id BIGINT NOT NULL COMMENT '角色ID',
    perm_id BIGINT NOT NULL COMMENT '权限ID',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uk_role_perm (role_id, perm_id),
    INDEX idx_perm_id (perm_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='角色-权限关联表';

-- 6. 部门表（树形结构，用于数据权限）
CREATE TABLE sys_dept (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    dept_id BIGINT NOT NULL UNIQUE COMMENT '部门ID',
    parent_id BIGINT DEFAULT 0 COMMENT '父部门ID',
    dept_name VARCHAR(64) NOT NULL COMMENT '部门名称',
    ancestors VARCHAR(512) COMMENT '祖先节点ID链，如 0,1,2（用于快速查询子部门）',
    sort_order INT DEFAULT 0,
    status TINYINT DEFAULT 1,
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_parent_id (parent_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='部门表';

-- 7. 角色自定义数据范围表（当data_scope=5时使用）
```

### 3.3 角色继承实现

角色继承是RBAC1的核心特性，高级角色自动继承低级角色的所有权限。例如"部门经理"继承"普通员工"的权限，再额外拥有"审批"权限。

```java
/**
 * 角色继承服务
 *
 * 算法：递归向上遍历角色继承链，收集所有继承权限
 *       使用深度优先搜索（DFS）遍历角色继承树
 * 设计模式：组合模式（Composite Pattern）—— 角色可以包含子角色，形成树状继承结构
 */
@Service
public class RoleInheritanceService {

    @Autowired
    private SysRoleMapper roleMapper;

    @Autowired
    private SysRolePermissionMapper rolePermMapper;

    /**
     * 获取角色的全部权限（含继承权限）
     * 
     * 算法：DFS深度优先搜索
     * 从当前角色出发，沿parentRoleId向上递归，收集继承链上所有角色的权限
     * 使用Set去重，避免重复权限
     * 使用visited集合防止循环继承导致的无限递归
     *
     * 时间复杂度：O(R * P)，R为继承链上的角色数，P为每个角色的平均权限数
     */
    public Set<Long> getAllPermissionIds(Long roleId) {
        Set<Long> allPermIds = new HashSet<>();
        Set<Long> visited = new HashSet<>(); // 防止循环继承
        collectPermissions(roleId, allPermIds, visited);
        return allPermIds;
    }

    /**
     * DFS递归收集权限
     */
    private void collectPermissions(Long roleId, Set<Long> allPermIds, Set<Long> visited) {
        // 防止循环继承：如果角色已经访问过，跳过
        if (roleId == null || roleId == 0 || visited.contains(roleId)) {
            return;
        }
        visited.add(roleId);

        // 收集当前角色的直接权限
        List<Long> directPermIds = rolePermMapper.selectPermIdsByRoleId(roleId);
        allPermIds.addAll(directPermIds);

        // 查询父角色，递归收集继承权限
        SysRole role = roleMapper.selectByRoleId(roleId);
        if (role != null && role.getParentRoleId() != null && role.getParentRoleId() > 0) {
            collectPermissions(role.getParentRoleId(), allPermIds, visited);
        }
    }

    /**
     * 获取角色继承树（用于前端展示）
     * 
     * 算法：自顶向下BFS构建树结构
     */
    public List<RoleTreeNode> buildRoleInheritanceTree() {
        List<SysRole> allRoles = roleMapper.selectAll();

        // 构建HashMap加速查找，O(N)
        Map<Long, RoleTreeNode> nodeMap = new HashMap<>();
        List<RoleTreeNode> roots = new ArrayList<>();

        for (SysRole role : allRoles) {
            RoleTreeNode node = new RoleTreeNode();
            node.setRoleId(role.getRoleId());
            node.setRoleName(role.getRoleName());
            node.setRoleCode(role.getRoleCode());
            node.setChildren(new ArrayList<>());
            nodeMap.put(role.getRoleId(), node);
        }

        for (SysRole role : allRoles) {
            RoleTreeNode node = nodeMap.get(role.getRoleId());
            if (role.getParentRoleId() == null || role.getParentRoleId() == 0) {
                roots.add(node);
            } else {
                RoleTreeNode parent = nodeMap.get(role.getParentRoleId());
                if (parent != null) {
                    parent.getChildren().add(node);
                }
            }
        }

        return roots;
    }

    /**
     * 校验角色互斥约束（RBAC2）
     * 
     * 设计模式：规格模式（Specification Pattern）—— 将业务规则封装为可复用的约束
     */
    public boolean checkMutualExclusion(Long userId, Long newRoleId) {
        // 查询用户已有的角色
        List<Long> existingRoleIds = roleMapper.selectRoleIdsByUserId(userId);

        // 查询新角色的互斥角色列表
        List<Long> mutexRoleIds = roleMapper.selectMutexRoleIds(newRoleId);

        // 检查是否冲突
        for (Long existingRoleId : existingRoleIds) {
            if (mutexRoleIds.contains(existingRoleId)) {
                return false; // 存在互斥角色，不能分配
            }
        }
        return true;
    }
}
```

## 四、权限树与菜单管理

### 4.1 权限树数据结构

权限在数据库中以`parent_id`形式存储为扁平化列表，需要在内存中构建为树形结构。这是典型的"列表转树"问题。

```java
/**
 * 权限树节点
 */
@Data
public class PermissionTreeNode implements Serializable {
    private Long permId;
    private Long parentId;
    private String permName;
    private String permCode;
    private Integer permType;   // 1菜单 2按钮 3API 4数据
    private String path;
    private String method;
    private String icon;
    private Integer sortOrder;
    private Integer visible;
    private List<PermissionTreeNode> children;
}

/**
 * 权限树构建服务
 *
 * 算法：两次遍历法构建树
 *   第一次遍历：将所有节点放入HashMap，O(N)
 *   第二次遍历：根据parentId建立父子关系，O(N)
 *   总时间复杂度：O(N)，空间复杂度：O(N)
 *
 * 对比递归法：递归法时间复杂度O(N^2)（每个节点都要遍历列表找子节点），
 *            两次遍历法更优
 *
 * 设计模式：建造者模式（Builder Pattern）—— 分步构建复杂的树形结构
 */
@Service
public class PermissionTreeService {

    @Autowired
    private SysPermissionMapper permMapper;

    /**
     * 构建完整权限树
     * 
     * 算法步骤：
     * 1. 查询所有权限记录（扁平列表）
     * 2. 第一次遍历：创建所有树节点，放入HashMap（permId -> node）
     * 3. 第二次遍历：根据parentId建立父子关系
     * 4. 提取根节点（parentId=0的节点）
     * 5. 递归排序所有层级
     */
    public List<PermissionTreeNode> buildFullTree() {
        List<SysPermission> allPerms = permMapper.selectAll();
        return buildTree(allPerms);
    }

    /**
     * 构建用户的权限树（只包含用户有权限的节点）
     */
    public List<PermissionTreeNode> buildUserTree(Long userId, Set<Long> userPermIds) {
        List<SysPermission> allPerms = permMapper.selectAll();

        // 过滤出用户有权限的节点，以及这些节点的所有祖先节点（确保树的完整性）
        Set<Long> visiblePermIds = new HashSet<>(userPermIds);
        enrichAncestors(allPerms, visiblePermIds);

        List<SysPermission> filteredPerms = allPerms.stream()
            .filter(p -> visiblePermIds.contains(p.getPermId()))
            .collect(Collectors.toList());

        return buildTree(filteredPerms);
    }

    /**
     * 补充祖先节点：确保树结构完整
     * 
     * 算法：自底向上递归查找所有祖先节点
     * 如果用户有"订单:删除"这个叶子节点的权限，
     * 需要把"订单管理"这个父菜单也加入，否则树会断裂
     */
    private void enrichAncestors(List<SysPermission> allPerms, Set<Long> visiblePermIds) {
        Map<Long, SysPermission> permMap = allPerms.stream()
            .collect(Collectors.toMap(SysPermission::getPermId, p -> p));

        Set<Long> ancestorIds = new HashSet<>();
        for (Long permId : visiblePermIds) {
            traceAncestors(permId, permMap, ancestorIds);
        }
        visiblePermIds.addAll(ancestorIds);
    }

    /**
     * 递归追溯祖先节点
     */
    private void traceAncestors(Long permId, Map<Long, SysPermission> permMap,
                                 Set<Long> ancestorIds) {
        SysPermission perm = permMap.get(permId);
        if (perm == null || perm.getParentId() == 0) {
            return;
        }
        if (ancestorIds.add(perm.getParentId())) { // 避免重复追溯
            traceAncestors(perm.getParentId(), permMap, ancestorIds);
        }
    }

    /**
     * 列表转树的核心算法：两次遍历法
     */
    public List<PermissionTreeNode> buildTree(List<SysPermission> permissions) {
        // 第一次遍历：创建所有节点，O(N)
        Map<Long, PermissionTreeNode> nodeMap = new LinkedHashMap<>();
        for (SysPermission perm : permissions) {
            PermissionTreeNode node = convertToNode(perm);
            nodeMap.put(perm.getPermId(), node);
        }

        // 第二次遍历：建立父子关系，O(N)
        List<PermissionTreeNode> roots = new ArrayList<>();
        for (SysPermission perm : permissions) {
            PermissionTreeNode node = nodeMap.get(perm.getPermId());
            if (perm.getParentId() == null || perm.getParentId() == 0) {
                roots.add(node);
            } else {
                PermissionTreeNode parent = nodeMap.get(perm.getParentId());
                if (parent != null) {
                    parent.getChildren().add(node);
                } else {
                    // 父节点不存在（可能被过滤掉了），作为根节点处理
                    roots.add(node);
                }
            }
        }

        // 递归排序
        sortTreeRecursive(roots);
        return roots;
    }

    private PermissionTreeNode convertToNode(SysPermission perm) {
        PermissionTreeNode node = new PermissionTreeNode();
        node.setPermId(perm.getPermId());
        node.setParentId(perm.getParentId());
        node.setPermName(perm.getPermName());
        node.setPermCode(perm.getPermCode());
        node.setPermType(perm.getPermType());
        node.setPath(perm.getPath());
        node.setMethod(perm.getMethod());
        node.setIcon(perm.getIcon());
        node.setSortOrder(perm.getSortOrder());
        node.setVisible(perm.getVisible());
        node.setChildren(new ArrayList<>());
        return node;
    }

    /**
     * 递归排序树的每一层
     * 算法：对每一层的子节点按sortOrder升序排列，然后递归排序子节点的子节点
     */
    private void sortTreeRecursive(List<PermissionTreeNode> nodes) {
        if (nodes == null || nodes.isEmpty()) {
            return;
        }
        nodes.sort(Comparator.comparingInt(n ->
            n.getSortOrder() != null ? n.getSortOrder() : Integer.MAX_VALUE));
        for (PermissionTreeNode node : nodes) {
            sortTreeRecursive(node.getChildren());
        }
    }
}
```

### 4.2 递归权限校验算法

在某些场景下，需要判断用户是否对某个权限节点有访问权——不仅要判断当前节点，还要判断其所有祖先节点是否都可访问（即路径上的每一级菜单都要有权限）。

```java
/**
 * 递归权限校验器
 *
 * 算法：自底向上递归校验（Bottom-Up Recursive Validation）
 * 从目标节点出发，沿着parentId向上递归，确保路径上每一个节点都在用户权限集合中
 *
 * 设计模式：模板方法模式（Template Method Pattern）
 * 定义校验流程骨架，子类可以扩展具体校验逻辑
 */
@Service
public class RecursivePermissionValidator {

    @Autowired
    private PermissionTreeService treeService;

    /**
     * 校验用户是否有某个权限节点的完整路径访问权
     *
     * @param targetPermId   目标权限ID
     * @param userPermIds    用户拥有的权限ID集合
     * @param allPermissions 系统全部权限（扁平列表）
     * @return true表示有权限
     *
     * 算法复杂度：O(D)，D为树的深度（通常不超过5层）
     */
    public boolean validateWithPath(Long targetPermId, Set<Long> userPermIds,
                                     List<SysPermission> allPermissions) {
        // 构建快速查找的Map
        Map<Long, SysPermission> permMap = allPermissions.stream()
            .collect(Collectors.toMap(SysPermission::getPermId, p -> p));

        // 从目标节点向上递归校验
        return validateUpward(targetPermId, userPermIds, permMap);
    }

    /**
     * 自底向上递归校验
     */
    private boolean validateUpward(Long permId, Set<Long> userPermIds,
                                    Map<Long, SysPermission> permMap) {
        // 基线条件：到达根节点
        if (permId == null || permId == 0) {
            return true;
        }

        // 当前节点不在用户权限集合中，校验失败
        if (!userPermIds.contains(permId)) {
            return false;
        }

        // 递归校验父节点
        SysPermission perm = permMap.get(permId);
        if (perm == null) {
            return false;
        }

        return validateUpward(perm.getParentId(), userPermIds, permMap);
    }

    /**
     * 批量校验：找出用户有权限访问的所有叶子节点
     * 
     * 算法：广度优先遍历（BFS）+ 过滤
     * 遍历权限树，对每个叶子节点执行自底向上校验
     */
    public List<PermissionTreeNode> filterAccessibleLeaves(
            List<PermissionTreeNode> tree, Set<Long> userPermIds) {
        List<PermissionTreeNode> result = new ArrayList<>();
        Queue<PermissionTreeNode> queue = new LinkedList<>(tree);

        while (!queue.isEmpty()) {
            PermissionTreeNode node = queue.poll();
            if (node.getChildren() == null || node.getChildren().isEmpty()) {
                // 叶子节点：检查是否在用户权限集合中
                if (userPermIds.contains(node.getPermId())) {
                    result.add(node);
                }
            } else {
                // 非叶子节点：将子节点加入队列继续遍历
                queue.addAll(node.getChildren());
            }
        }

        return result;
    }
}
```

```java
        }
        return SELF; // 默认仅本人
    }
}
```

### 5.2 数据权限服务

```java
/**
 * 数据权限服务
 *
 * 核心职责：根据用户的角色和数据范围配置，动态生成SQL过滤条件
 *
 * 算法：
 *   1. 查询用户的所有角色
 *   2. 取数据范围最大的角色（值最小 = 范围最大）
 *   3. 根据数据范围类型生成不同的SQL条件
 *   4. 对于"本部门及以下"，需要递归查询部门树获取所有子部门ID
 *
 * 设计模式：策略模式（Strategy Pattern）
 *   不同的DataScope对应不同的SQL生成策略
 */
@Service
public class DataScopeService {

    @Autowired
    private SysUserMapper userMapper;

    @Autowired
    private SysRoleMapper roleMapper;

    @Autowired
    private SysDeptMapper deptMapper;

    @Autowired
    private SysRoleDataScopeMapper dataScopeMapper;

    /**
     * 构建数据权限SQL片段
     *
     * @param userId     当前用户ID
     * @param deptAlias  SQL中部门表的别名（如 "o" 代表 order表的别名）
     * @param userAlias  SQL中用户字段的别名
     * @return SQL条件片段，如 "o.dept_id IN (1,2,3)" 或 "o.creator = 123"
     *         返回null表示不加条件（全部数据可见）
     */
    public String buildDataScopeSql(Long userId, String deptAlias, String userAlias) {
        // 1. 获取用户信息
        SysUser user = userMapper.selectByUserId(userId);
        if (user == null) {
            return "1=0"; // 无权限，拼接永假条件
        }

        // 2. 获取用户的所有角色
        List<SysRole> roles = roleMapper.selectRolesByUserId(userId);
        if (roles.isEmpty()) {
            return "1=0"; // 没有角色，无权限
        }

        // 3. 取数据范围最大的角色（数值越小，范围越大）
        // 算法：遍历找最小值，O(N)
        int maxScope = roles.stream()
            .mapToInt(SysRole::getDataScope)
            .min()
            .orElse(DataScope.SELF.getCode());

        // 4. 根据数据范围生成SQL片段（策略模式）
        DataScope scope = DataScope.fromCode(maxScope);
        switch (scope) {
            case ALL:
                // 全部数据：不加任何过滤条件
                return null;

            case DEPT:
                // 本部门数据
                return String.format("%s.dept_id = %d", deptAlias, user.getDeptId());

            case DEPT_AND_CHILDREN:
                // 本部门及以下：需要递归查询部门树
                List<Long> childDeptIds = getChildDeptIds(user.getDeptId());
                if (childDeptIds.isEmpty()) {
                    return String.format("%s.dept_id = %d", deptAlias, user.getDeptId());
                }
                String deptIdStr = childDeptIds.stream()
                    .map(String::valueOf)
                    .collect(Collectors.joining(","));
                return String.format("%s.dept_id IN (%s)", deptAlias, deptIdStr);

            case SELF:
                // 仅本人数据
                return String.format("%s.creator = %d", userAlias, userId);

            case CUSTOM:
                // 自定义数据范围：取角色配置的部门列表
                Set<Long> customDeptIds = getCustomDeptIds(roles);
                if (customDeptIds.isEmpty()) {
                    return "1=0";
                }
                String customDeptStr = customDeptIds.stream()
                    .map(String::valueOf)
                    .collect(Collectors.joining(","));
                return String.format("%s.dept_id IN (%s)", deptAlias, customDeptStr);

            default:
                return "1=0";
        }
    }

    /**
     * 获取部门及所有子部门的ID列表
     *
     * 算法：利用ancestors字段实现快速子树查询
     * ancestors存储了从根到当前节点的路径，如 "0,1,3"
     * 查询所有ancestors包含当前部门ID的记录即可得到所有子部门
     * 
     * 这比递归查询效率高得多：
     * - 递归查询：需要多次SQL查询，O(D)次（D为树深度）
     * - ancestors方案：一次LIKE查询即可，O(1)次SQL
     */
    private List<Long> getChildDeptIds(Long deptId) {
        List<Long> result = new ArrayList<>();
        result.add(deptId); // 包含自身

        // 使用ancestors字段快速查询：SELECT dept_id FROM sys_dept
        // WHERE FIND_IN_SET(#{deptId}, ancestors) > 0
        List<Long> childIds = deptMapper.selectChildDeptIds(deptId);
        result.addAll(childIds);
        return result;
    }

    /**
     * 获取自定义数据范围的部门ID集合
     */
    private Set<Long> getCustomDeptIds(List<SysRole> roles) {
        return roles.stream()
            .filter(r -> r.getDataScope() == DataScope.CUSTOM.getCode())
            .flatMap(r -> dataScopeMapper.selectDeptIdsByRoleId(r.getRoleId()).stream())
            .collect(Collectors.toSet());
    }
}
```

### 5.3 MyBatis拦截器实现自动数据权限注入

手动在每个SQL中添加数据权限条件既繁琐又容易遗漏。更好的方案是使用MyBatis拦截器自动注入SQL条件，对业务代码完全无侵入。

```java
/**
 * 数据权限注解：标注在Mapper方法上，表示该查询需要数据权限过滤
 */
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface DataPermission {
    /** SQL中部门字段的表别名，默认为主表 */
    String deptAlias() default "";
    /** SQL中用户字段的表别名 */
    String userAlias() default "";
    /** 部门字段名，默认dept_id */
    String deptColumn() default "dept_id";
    /** 用户字段名，默认creator */
    String userColumn() default "creator";
}

/**
 * MyBatis数据权限拦截器
 *
 * 设计模式：拦截器模式（Interceptor Pattern）/ AOP思想
 * 在SQL执行前，自动追加数据权限的WHERE条件
 *
 * 实现原理：
 * 1. 拦截MyBatis的StatementHandler.prepare方法
 * 2. 检查当前Mapper方法是否标注了@DataPermission注解
 * 3. 如果标注了，调用DataScopeService生成SQL条件
 * 4. 将条件追加到原始SQL的WHERE子句中
 *
 * 技术方案：JSqlParser解析和改写SQL语法树
 */
@Intercepts({
    @Signature(type = StatementHandler.class, method = "prepare",
               args = {Connection.class, Integer.class})
})
@Component
public class DataPermissionInterceptor implements Interceptor {

    @Autowired
    private DataScopeService dataScopeService;

    @Override
    public Object intercept(Invocation invocation) throws Throwable {
        StatementHandler handler = (StatementHandler) invocation.getTarget();

        // 通过反射获取MetaObject，用于读写StatementHandler内部属性
        MetaObject metaObject = SystemMetaObject.forObject(handler);

        // 获取MappedStatement（包含Mapper方法信息）
        MappedStatement ms = (MappedStatement) metaObject
            .getValue("delegate.mappedStatement");

        // 检查是否标注了@DataPermission注解
        DataPermission annotation = getDataPermissionAnnotation(ms);
        if (annotation == null) {
            return invocation.proceed(); // 未标注注解，直接放行
        }

        // 获取当前用户ID（从ThreadLocal中获取）
        Long userId = SecurityContextHolder.getUserId();
        if (userId == null) {
            return invocation.proceed();
        }

        // 生成数据权限SQL条件
        String condition = dataScopeService.buildDataScopeSql(
            userId,
            annotation.deptAlias().isEmpty() ? "" : annotation.deptAlias() + ".",
            annotation.userAlias().isEmpty() ? "" : annotation.userAlias() + "."
        );

        if (condition == null) {
            return invocation.proceed(); // 全部数据可见，不需要条件
        }

        // 获取原始SQL
        BoundSql boundSql = handler.getBoundSql();
        String originalSql = boundSql.getSql();

        // 使用JSqlParser解析并改写SQL
        String newSql = appendCondition(originalSql, condition);

        // 替换原始SQL
        metaObject.setValue("delegate.boundSql.sql", newSql);

        return invocation.proceed();
    }

    /**
     * 使用JSqlParser将条件追加到SQL的WHERE子句
     *
     * 算法：
     * 1. 解析SQL为AST（抽象语法树）
     * 2. 获取SELECT语句的WHERE子句
     * 3. 如果已有WHERE，用AND拼接新条件
     * 4. 如果没有WHERE，创建新的WHERE子句
     * 5. 将AST转回SQL字符串
     */
    private String appendCondition(String originalSql, String condition) {
        try {
            Statement statement = CCJSqlParserUtil.parse(originalSql);
            if (statement instanceof Select) {
                Select select = (Select) statement;
                PlainSelect plainSelect = (PlainSelect) select.getSelectBody();

                Expression where = plainSelect.getWhere();
                Expression newCondition = CCJSqlParserUtil.parseCondExpression(condition);

                if (where == null) {
                    plainSelect.setWhere(newCondition);
                } else {
                    // 用AND连接原有条件和新条件
                    AndExpression andExpression = new AndExpression(where, newCondition);
                    plainSelect.setWhere(andExpression);
                }

                return select.toString();
            }
        } catch (Exception e) {
            // SQL解析失败，降级处理：直接字符串拼接
            if (originalSql.toUpperCase().contains("WHERE")) {
                return originalSql + " AND " + condition;
            } else {
                return originalSql + " WHERE " + condition;
            }
        }
        return originalSql;
    }

    /**
     * 通过反射获取Mapper方法上的@DataPermission注解
     */
    private DataPermission getDataPermissionAnnotation(MappedStatement ms) {
        try {
            String id = ms.getId();
            String className = id.substring(0, id.lastIndexOf('.'));
            String methodName = id.substring(id.lastIndexOf('.') + 1);

            Class<?> clazz = Class.forName(className);
            for (Method method : clazz.getDeclaredMethods()) {
                if (method.getName().equals(methodName)) {
                    return method.getAnnotation(DataPermission.class);
                }
            }
        } catch (Exception e) {
            // 忽略反射异常
        }
        return null;
    }

    @Override
    public Object plugin(Object target) {
        return Plugin.wrap(target, this);
    }

    @Override
    public void setProperties(Properties properties) {
    }
}

// ========== 使用示例 ==========

/**
 * Mapper层使用示例：只需要加上@DataPermission注解
 */
@Mapper
public interface OrderMapper {

    /**
     * 查询订单列表（自动加上数据权限过滤）
     * 原始SQL：SELECT * FROM t_order o LEFT JOIN sys_dept d ON o.dept_id = d.dept_id
     * 拦截器会自动追加：WHERE o.dept_id IN (1,2,3) 或 WHERE o.creator = 123
     */
    @DataPermission(deptAlias = "o", userAlias = "o")
    @Select("SELECT o.* FROM t_order o LEFT JOIN sys_dept d ON o.dept_id = d.dept_id")
    List<Order> selectOrderList();

    /**
     * 无需数据权限的查询：不加注解即可
     */
    @Select("SELECT COUNT(*) FROM t_order")
    int countAll();
}
```

## 六、权限缓存与性能优化

### 6.1 性能挑战

权限校验发生在每一次API请求中，如果每次都查数据库，性能不可接受。一个用户的权限校验链路涉及：

1. 查用户信息 → 1次SQL
2. 查用户的角色 → 1次SQL
3. 查角色的权限 → 1次SQL
4. 构建权限树 → 内存计算
5. 数据权限SQL生成 → 1次SQL

**不做缓存的情况下，每次请求至少4次SQL查询**，在高并发场景下会成为瓶颈。

### 6.2 二级缓存架构

采用**本地缓存（L1）+ 远程缓存（L2）**的二级缓存架构：

```
请求 → L1本地缓存（Caffeine）→ 未命中 → L2远程缓存（Redis）→ 未命中 → 数据库
                                                                        │
                                                                   回填L2和L1
```

```java
/**
 * 权限缓存服务
 *
 * 设计模式：装饰器模式（Decorator Pattern）
 * 在原有的权限查询逻辑外面包装缓存层，对外接口不变
 *
 * 缓存架构：二级缓存（Two-Level Cache）
 * L1：Caffeine本地缓存（进程内，速度最快，纳秒级）
 * L2：Redis远程缓存（跨进程共享，毫秒级）
 *
 * 一致性策略：
 * L1 TTL较短（5分钟），容忍短暂不一致
 * L2 TTL较长（30分钟），作为主缓存
 * 权限变更时主动清除L1和L2
 */
@Service
public class PermissionCacheService {

    /** L1本地缓存：Caffeine */
    private final Cache<String, Set<String>> localCache = Caffeine.newBuilder()
        .maximumSize(10_000)          // 最多缓存1万个用户的权限
        .expireAfterWrite(5, TimeUnit.MINUTES)  // 5分钟过期
        .recordStats()                 // 记录命中率统计
        .build();

    /** L2远程缓存：Redis */
    @Autowired
    private StringRedisTemplate redisTemplate;

    @Autowired
    private SysUserRoleMapper userRoleMapper;

    @Autowired
    private SysRolePermissionMapper rolePermMapper;

    @Autowired
    private RoleInheritanceService roleInheritanceService;

    private static final String CACHE_KEY_PREFIX = "perm:user:";
    private static final long REDIS_TTL_MINUTES = 30;

    /**
     * 获取用户的所有权限编码
     *
     * 查找顺序：L1 → L2 → DB
     * 算法：Cache-Aside模式（旁路缓存）
     */
    public Set<String> getUserPermissions(Long userId) {
        String cacheKey = CACHE_KEY_PREFIX + userId;

        // Step 1：查L1本地缓存
        Set<String> permissions = localCache.getIfPresent(cacheKey);
        if (permissions != null) {
            return permissions;
        }

        // Step 2：查L2 Redis缓存
        String redisValue = redisTemplate.opsForValue().get(cacheKey);
        if (redisValue != null) {
            permissions = JSON.parseObject(redisValue, new TypeReference<Set<String>>() {});
            // 回填L1
            localCache.put(cacheKey, permissions);
            return permissions;
        }

        // Step 3：查数据库
        permissions = loadPermissionsFromDB(userId);

        // 回填L2
        redisTemplate.opsForValue().set(
            cacheKey,
            JSON.toJSONString(permissions),
            REDIS_TTL_MINUTES,
            TimeUnit.MINUTES
        );

        // 回填L1
        localCache.put(cacheKey, permissions);

        return permissions;
    }

    /**
     * 从数据库加载权限
     * 链路：用户 → 角色 → 权限（含继承）
     */
    private Set<String> loadPermissionsFromDB(Long userId) {
        // 1. 查用户的角色
        List<Long> roleIds = userRoleMapper.selectRoleIdsByUserId(userId);
        if (roleIds.isEmpty()) {
            return Collections.emptySet();
        }

        // 2. 查角色的权限（含继承权限）
        Set<Long> allPermIds = new HashSet<>();
        for (Long roleId : roleIds) {
            Set<Long> permIds = roleInheritanceService.getAllPermissionIds(roleId);
            allPermIds.addAll(permIds);
        }

        // 3. 将权限ID转为权限编码
        if (allPermIds.isEmpty()) {
            return Collections.emptySet();
        }
        List<String> permCodes = rolePermMapper.selectPermCodesByPermIds(new ArrayList<>(allPermIds));
        return new HashSet<>(permCodes);
    }

    /**
     * 清除用户权限缓存（权限变更时调用）
     * 同时清除L1和L2
     */
    public void clearUserPermissionCache(Long userId) {
        String cacheKey = CACHE_KEY_PREFIX + userId;
        localCache.invalidate(cacheKey);   // 清除L1
        redisTemplate.delete(cacheKey);     // 清除L2
    }

    /**
     * 批量清除：角色权限变更时，清除该角色下所有用户的缓存
     */
    public void clearRoleUsersCache(Long roleId) {
        List<Long> userIds = userRoleMapper.selectUserIdsByRoleId(roleId);
        for (Long userId : userIds) {
            clearUserPermissionCache(userId);
        }
    }

    /**
     * 获取缓存命中率统计信息
     */
    public CacheStats getCacheStats() {
        return localCache.stats();
    }
}
```

### 6.3 权限变更广播

在分布式部署环境下（多台应用服务器），一台机器上的权限变更需要通知其他机器清除本地缓存。使用Redis Pub/Sub实现跨节点缓存失效通知。

```java
/**
 * 权限变更广播服务
 *
 * 设计模式：观察者模式（Observer Pattern）/ 发布-订阅模式（Pub-Sub Pattern）
 * 当权限发生变更时，发布消息通知所有订阅者清除本地缓存
 *
 * 技术方案：Redis Pub/Sub
 * 优点：实时性好，所有节点几乎同时收到通知
 * 缺点：如果节点不在线，会丢失消息（对于缓存失效场景可接受，因为有TTL兜底）
 */
@Component
public class PermissionChangeBroadcaster {

    private static final String CHANNEL = "permission:change";

    @Autowired
    private StringRedisTemplate redisTemplate;

    @Autowired
    private PermissionCacheService cacheService;

    /**
     * 发布权限变更消息
     *
     * @param changeType 变更类型：ROLE_PERM_CHANGE / USER_ROLE_CHANGE / PERM_UPDATE
     * @param targetId   目标ID（角色ID或用户ID）
     */
    public void publishChange(String changeType, Long targetId) {
        PermissionChangeMessage message = new PermissionChangeMessage();
        message.setChangeType(changeType);
        message.setTargetId(targetId);
        message.setTimestamp(System.currentTimeMillis());
        message.setSourceNodeId(getNodeId()); // 发送节点标识

        String json = JSON.toJSONString(message);
        redisTemplate.convertAndSend(CHANNEL, json);
    }

    /**
     * 订阅权限变更消息
     */
    @Bean
    public RedisMessageListenerContainer permissionChangeListener(
            RedisConnectionFactory connectionFactory) {
        RedisMessageListenerContainer container = new RedisMessageListenerContainer();
        container.setConnectionFactory(connectionFactory);
        container.addMessageListener((message, pattern) -> {
            String json = new String(message.getBody(), StandardCharsets.UTF_8);
            PermissionChangeMessage changeMsg = JSON.parseObject(json,
                PermissionChangeMessage.class);

            // 处理权限变更：清除本地缓存
            handlePermissionChange(changeMsg);
        }, new ChannelTopic(CHANNEL));
        return container;
    }

    /**
     * 处理权限变更通知
     */
    private void handlePermissionChange(PermissionChangeMessage message) {
        switch (message.getChangeType()) {
            case "USER_ROLE_CHANGE":
                // 用户的角色变了，清除该用户的缓存
                cacheService.clearUserPermissionCache(message.getTargetId());
                break;
            case "ROLE_PERM_CHANGE":
                // 角色的权限变了，清除该角色下所有用户的缓存
                cacheService.clearRoleUsersCache(message.getTargetId());
                break;
            case "PERM_UPDATE":
                // 权限本身变更（如权限编码修改），需要全量刷新
                // 这是一个重量级操作，应该很少发生
                cacheService.clearAllCache();
                break;
            default:
                break;
        }
    }

    private String getNodeId() {
        try {
            return InetAddress.getLocalHost().getHostName();
        } catch (Exception e) {
            return UUID.randomUUID().toString().substring(0, 8);
        }
    }
}

/**
 * 权限变更消息体
 */
@Data
public class PermissionChangeMessage implements Serializable {
    /** 变更类型 */
    private String changeType;
    /** 目标ID（用户ID或角色ID） */
    private Long targetId;
    /** 时间戳 */
    private Long timestamp;
    /** 发送节点标识 */
    private String sourceNodeId;
}
```

### 6.4 缓存预热与防穿透

```java
/**
 * 缓存预热与防穿透策略
 *
 * 设计模式：代理模式（Proxy Pattern）—— 在缓存层代理数据库访问
 */
@Component
public class PermissionCacheWarmUp {

    @Autowired
    private PermissionCacheService cacheService;

    @Autowired
    private SysUserMapper userMapper;

    /**
     * 系统启动时预热活跃用户的权限缓存
     *
     * 策略：只预热最近7天登录过的活跃用户
     * 避免一次性加载所有用户导致内存压力
     */
    @PostConstruct
    public void warmUp() {
        // 异步预热，不阻塞系统启动
        CompletableFuture.runAsync(() -> {
            List<Long> activeUserIds = userMapper.selectActiveUserIds(7); // 最近7天登录的用户
            for (Long userId : activeUserIds) {
                try {
                    cacheService.getUserPermissions(userId); // 触发缓存加载
                } catch (Exception e) {
                    // 预热失败不影响系统启动
                }
            }
        });
    }

    /**
     * 防缓存穿透：对于不存在的用户，缓存空结果
     *
     * 算法：布隆过滤器（Bloom Filter）—— 快速判断用户是否存在
     * 特点：可能有误判（false positive），但不会漏判（no false negative）
     */
    private final BloomFilter<Long> userBloomFilter =
        BloomFilter.create(Funnels.longFunnel(), 1_000_000, 0.01); // 100万容量，1%误判率

    /**
     * 初始化布隆过滤器
     */
    @PostConstruct
    public void initBloomFilter() {
        List<Long> allUserIds = userMapper.selectAllUserIds();
        allUserIds.forEach(userBloomFilter::put);
    }

    /**
     * 通过布隆过滤器快速判断用户是否存在
     * 不存在的用户直接返回空权限，避免穿透到数据库
     */
    public boolean mightExist(Long userId) {
        return userBloomFilter.mightContain(userId);
    }
}
```

## 七、权限校验链

### 7.1 责任链模式设计

权限校验是一个多步骤的过程，每个步骤关注不同的维度。使用责任链模式将各个校验步骤解耦，形成一个可扩展的校验链。

校验链路：**认证（Authentication）→ 功能鉴权（Authorization）→ 数据权限（Data Scope）→ 操作权限（Operation）**

```java
/**
 * 权限校验链
 *
 * 设计模式：责任链模式（Chain of Responsibility Pattern）
 * 将多个校验步骤组织为一条链，每个节点负责一种校验。
 * 如果某个节点校验失败，整条链终止并返回失败结果。
 * 如果通过，传递给下一个节点继续校验。
 *
 * 优点：
 * 1. 每个校验步骤独立封装，职责单一
 * 2. 可以灵活增减校验步骤，无需修改已有代码（开闭原则）
 * 3. 校验顺序可以动态调整
 */
public abstract class PermissionCheckHandler {

    /** 下一个校验节点 */
    protected PermissionCheckHandler next;

    /**
     * 设置下一个处理器
     * 返回下一个处理器，支持链式调用
     */
    public PermissionCheckHandler setNext(PermissionCheckHandler next) {
        this.next = next;
        return next;
    }

    /**
     * 执行校验
     *
     * @param context 校验上下文，包含请求信息
     * @return 校验结果
     */
    public PermissionCheckResult check(PermissionCheckContext context) {
        // 执行当前节点的校验逻辑
        PermissionCheckResult result = doCheck(context);

        if (!result.isPassed()) {
            return result; // 校验失败，终止链路
        }

        // 传递给下一个节点
        if (next != null) {
            return next.check(context);
        }

        return PermissionCheckResult.pass(); // 所有校验通过
    }

    /**
     * 子类实现具体的校验逻辑
     */
    protected abstract PermissionCheckResult doCheck(PermissionCheckContext context);
}

/**
 * 校验上下文：贯穿整个校验链的请求信息
 */
@Data
public class PermissionCheckContext {
    private Long userId;               // 当前用户ID
    private String token;              // 认证令牌
    private String requestUri;         // 请求路径
    private String httpMethod;         // HTTP方法
    private String permCode;           // 需要的权限编码
    private SysUser user;              // 用户信息（认证后填充）
    private Set<String> permissions;   // 用户权限集合（鉴权阶段填充）
    private String dataScopeSql;       // 数据权限SQL（数据权限阶段填充）
}

/**
 * 校验结果
 */
@Data
public class PermissionCheckResult {
    private boolean passed;
    private String message;
    private int httpStatus;

    public static PermissionCheckResult pass() {
        PermissionCheckResult r = new PermissionCheckResult();
        r.setPassed(true);
        return r;
    }

    public static PermissionCheckResult fail(int httpStatus, String message) {
        PermissionCheckResult r = new PermissionCheckResult();
        r.setPassed(false);
        r.setHttpStatus(httpStatus);
        r.setMessage(message);
        return r;
    }
}
```

### 7.2 各校验节点实现

```java
/**
 * 第一环：认证校验（Authentication）
 * 职责：验证用户身份是否合法（Token是否有效）
 */
@Component
public class AuthenticationHandler extends PermissionCheckHandler {

    @Autowired
    private TokenService tokenService;

    @Autowired
    private SysUserMapper userMapper;

    @Override
    protected PermissionCheckResult doCheck(PermissionCheckContext context) {
        // 1. 校验Token
        String token = context.getToken();
        if (token == null || token.isEmpty()) {
            return PermissionCheckResult.fail(401, "未提供认证令牌");
        }

        // 2. 解析Token，获取用户ID
        Long userId = tokenService.parseToken(token);
        if (userId == null) {
            return PermissionCheckResult.fail(401, "认证令牌无效或已过期");
        }

        // 3. 查询用户信息
        SysUser user = userMapper.selectByUserId(userId);
        if (user == null || user.getStatus() == 0) {
            return PermissionCheckResult.fail(401, "用户不存在或已禁用");
        }

        // 4. 填充上下文
        context.setUserId(userId);
        context.setUser(user);

        return PermissionCheckResult.pass();
    }
}

/**
 * 第二环：功能鉴权（Authorization）
 * 职责：验证用户是否拥有访问该接口的权限
 */
@Component
public class AuthorizationHandler extends PermissionCheckHandler {

    @Autowired
    private PermissionCacheService cacheService;

    /** 白名单路径：不需要鉴权的公共接口 */
    private static final Set<String> WHITE_LIST = Set.of(
        "/api/auth/login",
        "/api/auth/logout",
        "/api/health",
        "/api/public/**"
    );

    @Override
    protected PermissionCheckResult doCheck(PermissionCheckContext context) {
        // 1. 白名单放行
        if (isWhiteListed(context.getRequestUri())) {
            return PermissionCheckResult.pass();
        }

        // 2. 获取用户权限集合
        Set<String> permissions = cacheService.getUserPermissions(context.getUserId());
        context.setPermissions(permissions);

        // 3. 超级管理员放行
        if (permissions.contains("*")) {
            return PermissionCheckResult.pass();
        }

        // 4. 校验是否有目标权限
        String permCode = context.getPermCode();
        if (permCode != null && !permissions.contains(permCode)) {
            return PermissionCheckResult.fail(403,
                "无权限访问: " + permCode);
        }

        return PermissionCheckResult.pass();
    }

    /**
     * 白名单匹配：支持Ant风格的路径模式
     * 算法：逐个模式匹配，支持*和**通配符
     */
    private boolean isWhiteListed(String uri) {
        AntPathMatcher matcher = new AntPathMatcher();
        return WHITE_LIST.stream().anyMatch(pattern -> matcher.match(pattern, uri));
    }
}

/**
 * 第三环：数据权限（Data Scope）
 * 职责：计算用户的数据可见范围，生成SQL条件
 */
@Component
public class DataScopeHandler extends PermissionCheckHandler {

    @Autowired
    private DataScopeService dataScopeService;

    @Override
    protected PermissionCheckResult doCheck(PermissionCheckContext context) {
        // 生成数据权限SQL（存入上下文，由后续的MyBatis拦截器使用）
        String dataScopeSql = dataScopeService.buildDataScopeSql(
            context.getUserId(), "t", "t");
        context.setDataScopeSql(dataScopeSql);

        // 将数据权限SQL放入ThreadLocal，供MyBatis拦截器使用
        DataScopeContextHolder.setDataScopeSql(dataScopeSql);

        return PermissionCheckResult.pass(); // 数据权限不阻断请求，只限制数据范围
    }
}

/**
 * 第四环：操作权限（Operation Permission）
 * 职责：细粒度校验，如检查用户是否有权操作特定资源
 * 例如：用户只能编辑自己创建的订单，即使有"order:edit"权限
 */
@Component
public class OperationPermissionHandler extends PermissionCheckHandler {

    @Override
    protected PermissionCheckResult doCheck(PermissionCheckContext context) {
        // 操作权限通常由业务层自行校验
        // 这里提供通用的资源归属校验框架
        return PermissionCheckResult.pass();
    }
}

/**
 * ThreadLocal持有器：在请求线程中传递数据权限SQL
 */
public class DataScopeContextHolder {

    private static final ThreadLocal<String> DATA_SCOPE_SQL = new ThreadLocal<>();

    public static void setDataScopeSql(String sql) {
        DATA_SCOPE_SQL.set(sql);
    }

    public static String getDataScopeSql() {
        return DATA_SCOPE_SQL.get();
    }

    public static void clear() {
        DATA_SCOPE_SQL.remove();
    }
}
```

### 7.3 校验链的组装与调用

```java
/**
 * 权限校验链的组装与执行
 *
 * 设计模式：建造者模式（Builder Pattern）—— 灵活组装校验链
 */
@Configuration
public class PermissionCheckChainConfig {

    @Autowired
    private AuthenticationHandler authHandler;

    @Autowired
    private AuthorizationHandler authzHandler;

    @Autowired
    private DataScopeHandler dataScopeHandler;

    @Autowired
    private OperationPermissionHandler operationHandler;

    /**
     * 构建完整的校验链：认证 → 鉴权 → 数据权限 → 操作权限
     */
    @Bean("permissionCheckChain")
    public PermissionCheckHandler buildChain() {
        authHandler.setNext(authzHandler);
        authzHandler.setNext(dataScopeHandler);
        dataScopeHandler.setNext(operationHandler);
        return authHandler; // 返回链头
    }
}

/**
 * Spring MVC拦截器：在请求到达Controller之前执行权限校验链
 *
 * 设计模式：拦截器模式（Interceptor Pattern）
 */
@Component
public class PermissionInterceptor implements HandlerInterceptor {

    @Autowired
    @Qualifier("permissionCheckChain")
    private PermissionCheckHandler checkChain;

    @Override
    public boolean preHandle(HttpServletRequest request,
                             HttpServletResponse response,
                             Object handler) throws Exception {
        // 构建校验上下文
        PermissionCheckContext context = new PermissionCheckContext();
        context.setToken(request.getHeader("Authorization"));
        context.setRequestUri(request.getRequestURI());
        context.setHttpMethod(request.getMethod());

        // 从Handler方法上获取权限编码
        if (handler instanceof HandlerMethod) {
            HandlerMethod hm = (HandlerMethod) handler;
            RequirePermission annotation = hm.getMethodAnnotation(RequirePermission.class);
            if (annotation != null) {
                context.setPermCode(annotation.value());
            }
        }

        // 执行校验链
        PermissionCheckResult result = checkChain.check(context);

        if (!result.isPassed()) {
            response.setStatus(result.getHttpStatus());
            response.setContentType("application/json;charset=UTF-8");
            response.getWriter().write(
                JSON.toJSONString(Result.fail(result.getHttpStatus(), result.getMessage())));
            return false;
        }

        // 将用户信息存入ThreadLocal，供Controller使用
        SecurityContextHolder.setUser(context.getUser());

        return true;
    }

    @Override
    public void afterCompletion(HttpServletRequest request,
                                HttpServletResponse response,
                                Object handler, Exception ex) {
        // 请求结束后清理ThreadLocal，防止内存泄漏
        SecurityContextHolder.clear();
        DataScopeContextHolder.clear();
    }
}
```

## 八、动态权限配置

### 8.1 运行时权限调整

在生产环境中，经常需要在不重启服务的情况下动态调整权限配置。例如临时给某个用户添加权限、临时关闭某个功能模块的访问等。

```java
/**
 * 动态权限配置服务
 *
 * 设计模式：观察者模式（Observer Pattern）
 * 权限配置变更时通知所有观察者（缓存服务、前端推送服务等）
 *
 * 核心能力：
 * 1. 运行时增删改角色、权限、用户角色关系
 * 2. 变更后立即生效（清除缓存 + 广播通知）
 * 3. 支持灰度发布（按百分比、按白名单逐步放开新权限）
 */
@Service
public class DynamicPermissionService {

    @Autowired
    private SysRolePermissionMapper rolePermMapper;

    @Autowired
    private SysUserRoleMapper userRoleMapper;

    @Autowired
    private PermissionCacheService cacheService;

    @Autowired
    private PermissionChangeBroadcaster broadcaster;

    @Autowired
    private PermissionChangeLogMapper changeLogMapper;

    /**
     * 动态添加角色权限
     * 事务保证：数据库变更 + 缓存清除 + 广播通知在同一事务中
     */
    @Transactional(rollbackFor = Exception.class)
    public void addRolePermission(Long roleId, Long permId, Long operatorId) {
        // 1. 插入角色-权限关联
        SysRolePermission record = new SysRolePermission();
        record.setRoleId(roleId);
        record.setPermId(permId);
        rolePermMapper.insert(record);

        // 2. 记录变更日志（审计追踪）
        logChange("ADD_ROLE_PERM", roleId, permId, operatorId);

        // 3. 清除缓存 + 广播
        cacheService.clearRoleUsersCache(roleId);
        broadcaster.publishChange("ROLE_PERM_CHANGE", roleId);
    }

    /**
     * 动态移除角色权限
     */
    @Transactional(rollbackFor = Exception.class)
    public void removeRolePermission(Long roleId, Long permId, Long operatorId) {
        rolePermMapper.deleteByRoleIdAndPermId(roleId, permId);
        logChange("REMOVE_ROLE_PERM", roleId, permId, operatorId);
        cacheService.clearRoleUsersCache(roleId);
        broadcaster.publishChange("ROLE_PERM_CHANGE", roleId);
    }

    /**
     * 动态分配用户角色
     */
    @Transactional(rollbackFor = Exception.class)
    public void assignUserRole(Long userId, Long roleId, Long operatorId) {
        SysUserRole record = new SysUserRole();
        record.setUserId(userId);
        record.setRoleId(roleId);
        userRoleMapper.insert(record);

        logChange("ASSIGN_USER_ROLE", userId, roleId, operatorId);
        cacheService.clearUserPermissionCache(userId);
        broadcaster.publishChange("USER_ROLE_CHANGE", userId);
    }

    /**
     * 动态撤销用户角色
     */
    @Transactional(rollbackFor = Exception.class)
    public void revokeUserRole(Long userId, Long roleId, Long operatorId) {
        userRoleMapper.deleteByUserIdAndRoleId(userId, roleId);
        logChange("REVOKE_USER_ROLE", userId, roleId, operatorId);
        cacheService.clearUserPermissionCache(userId);
        broadcaster.publishChange("USER_ROLE_CHANGE", userId);
    }

    /**
     * 记录权限变更日志（审计追踪）
     *
     * 所有权限变更操作都必须记录日志，用于安全审计和问题排查
     */
    private void logChange(String action, Long targetId1, Long targetId2, Long operatorId) {
        PermissionChangeLog log = new PermissionChangeLog();
        log.setAction(action);
        log.setTargetId1(targetId1);
        log.setTargetId2(targetId2);
        log.setOperatorId(operatorId);
        log.setOperateTime(LocalDateTime.now());
        changeLogMapper.insert(log);
    }
}

 * 权限变更日志表
 */
@Data
@TableName("sys_permission_change_log")
public class PermissionChangeLog {
    private Long id;
    private String action;         // 操作类型
    private Long targetId1;        // 目标ID1（角色ID/用户ID）
    private Long targetId2;        // 目标ID2（权限ID/角色ID）
    private Long operatorId;       // 操作人ID
    private LocalDateTime operateTime; // 操作时间
}
```

### 8.2 权限生效机制

```java
/**
 * 权限生效策略
 *
 * 设计模式：策略模式（Strategy Pattern）
 * 不同的生效策略适用于不同的业务场景
 */
public interface PermissionEffectStrategy {
    /**
     * 执行权限生效
     * @param changeEvent 变更事件
     */
    void apply(PermissionChangeEvent changeEvent);
}

/**
 * 立即生效策略
 * 适用于：紧急权限变更，如安全事件时立即撤销用户权限
 */
@Component("immediateEffect")
public class ImmediateEffectStrategy implements PermissionEffectStrategy {

    @Autowired
    private PermissionCacheService cacheService;

    @Autowired
    private PermissionChangeBroadcaster broadcaster;

    @Override
    public void apply(PermissionChangeEvent event) {
        // 立即清除缓存
        if (event.getAffectedUserIds() != null) {
            event.getAffectedUserIds().forEach(cacheService::clearUserPermissionCache);
        }
        // 立即广播到所有节点
        broadcaster.publishChange(event.getChangeType(), event.getTargetId());
    }
}

/**
 * 延迟生效策略
 * 适用于：批量权限调整，避免瞬间大量缓存失效导致数据库压力
 * 
 * 算法：平滑过期（Smooth Expiration）
 * 不是立即清除缓存，而是设置较短的TTL让缓存自然过期
 * 避免"缓存雪崩"——大量缓存同时失效导致数据库瞬时压力过大
 */
@Component("delayedEffect")
public class DelayedEffectStrategy implements PermissionEffectStrategy {

    @Autowired
    private StringRedisTemplate redisTemplate;

    private static final String CACHE_KEY_PREFIX = "perm:user:";

    @Override
    public void apply(PermissionChangeEvent event) {
        if (event.getAffectedUserIds() != null) {
            // 随机化过期时间：1~5分钟内过期
            // 避免所有缓存同时失效（缓存雪崩防护）
            Random random = new Random();
            for (Long userId : event.getAffectedUserIds()) {
                String cacheKey = CACHE_KEY_PREFIX + userId;
                long ttlSeconds = 60 + random.nextInt(240); // 60~300秒
                redisTemplate.expire(cacheKey, ttlSeconds, TimeUnit.SECONDS);
            }
        }
    }
}

/**
 * 权限变更事件
 */
@Data
public class PermissionChangeEvent {
    private String changeType;
    private Long targetId;
    private List<Long> affectedUserIds;
    private String effectStrategy; // "immediate" 或 "delayed"
}
```

### 8.3 灰度发布

新权限的上线需要谨慎，一旦权限配置错误可能导致用户无法使用系统或者越权访问。灰度发布机制允许逐步放开新权限。

```java
/**
 * 权限灰度发布服务
 *
 * 设计模式：策略模式 + 模板方法模式
 * 支持多种灰度策略：白名单灰度、百分比灰度、部门灰度
 *
 * 算法：
 * - 百分比灰度：对用户ID取哈希后取模，判断是否在灰度范围内
 * - 白名单灰度：直接查找用户是否在白名单集合中
 */
@Service
public class PermissionGrayReleaseService {

    @Autowired
    private StringRedisTemplate redisTemplate;

    private static final String GRAY_CONFIG_KEY = "perm:gray:config:";

    /**
     * 灰度配置
     */
    @Data
    public static class GrayReleaseConfig {
        private String permCode;            // 权限编码
        private GrayStrategy strategy;      // 灰度策略
        private int percentage;             // 灰度百分比（0-100）
        private Set<Long> whiteListUserIds; // 白名单用户ID
        private Set<Long> whiteListDeptIds; // 白名单部门ID
        private boolean enabled;            // 是否启用灰度
    }

    public enum GrayStrategy {
        PERCENTAGE,   // 按百分比灰度
        WHITE_LIST,   // 按白名单灰度
        DEPT          // 按部门灰度
    }

    /**
     * 判断用户是否命中灰度
     *
     * @param userId   用户ID
     * @param permCode 权限编码
     * @return true表示用户在灰度范围内，可以使用新权限
     */
    public boolean isInGrayRelease(Long userId, String permCode) {
        GrayReleaseConfig config = getGrayConfig(permCode);
        if (config == null || !config.isEnabled()) {
            return true; // 没有灰度配置，说明已全量发布
        }

        switch (config.getStrategy()) {
            case PERCENTAGE:
                return isInPercentage(userId, config.getPercentage());
            case WHITE_LIST:
                return config.getWhiteListUserIds().contains(userId);
            case DEPT:
                Long deptId = getUserDeptId(userId);
                return config.getWhiteListDeptIds().contains(deptId);
            default:
                return false;
        }
    }

    /**
     * 百分比灰度算法
     *
     * 算法：对用户ID计算哈希值，取模100，判断是否小于灰度百分比
     * 特点：
     * 1. 同一个用户多次判断结果一致（确定性）
     * 2. 用户分布均匀（哈希的均匀性）
     * 3. 增大百分比时，原来命中的用户仍然命中（单调递增性）
     */
    private boolean isInPercentage(Long userId, int percentage) {
        if (percentage <= 0) return false;
        if (percentage >= 100) return true;

        // 使用MurmurHash保证分布均匀
        int hash = Math.abs(Hashing.murmur3_32_fixed()
            .hashLong(userId).asInt());
        return (hash % 100) < percentage;
    }

    /**
     * 获取灰度配置（从Redis读取）
     */
    private GrayReleaseConfig getGrayConfig(String permCode) {
        String json = redisTemplate.opsForValue().get(GRAY_CONFIG_KEY + permCode);
        if (json == null) return null;
        return JSON.parseObject(json, GrayReleaseConfig.class);
    }

    /**
     * 设置灰度配置
     */
    public void setGrayConfig(GrayReleaseConfig config) {
        redisTemplate.opsForValue().set(
            GRAY_CONFIG_KEY + config.getPermCode(),
            JSON.toJSONString(config)
        );
    }

    /**
     * 全量发布：移除灰度配置
     */
    public void fullRelease(String permCode) {
        redisTemplate.delete(GRAY_CONFIG_KEY + permCode);
    }

    private Long getUserDeptId(Long userId) {
        // 查询用户部门ID（省略具体实现）
        return null;
    }
}
```

## 九、系统总结

### 9.1 核心设计决策

| 决策点 | 选择 | 设计模式/算法 | 理由 |
|--------|------|--------------|------|
| 权限模型 | RBAC为核心 + ABAC扩展 | — | RBAC适合企业管理场景，ABAC补充动态策略 |
| 权限树构建 | 两次遍历法（HashMap） | 建造者模式 | O(N)时间复杂度，优于递归法O(N^2) |
| 角色继承 | DFS递归遍历 | 组合模式 | 支持多层角色继承，visited集合防循环 |
| 角色互斥 | 约束校验 | 规格模式 | RBAC2约束模型实现 |
| 权限校验 | 四级校验链 | 责任链模式 | 认证→鉴权→数据权限→操作权限，各自独立可扩展 |
| 数据权限 | MyBatis拦截器自动注入 | 拦截器模式 | 对业务代码无侵入，JSqlParser改写SQL |
| 权限缓存 | L1 Caffeine + L2 Redis | 装饰器模式 | 二级缓存兼顾速度和一致性 |
| 缓存失效 | Redis Pub/Sub广播 | 观察者模式 | 多节点实时同步缓存失效 |
| 动态配置 | 运行时增删改 + 审计日志 | 策略模式 | 不重启服务即可调整权限 |
| 灰度发布 | 百分比/白名单/部门 | 策略模式 | MurmurHash取模保证分布均匀和确定性 |
| 防穿透 | 布隆过滤器 | — | 快速判断用户是否存在，避免缓存穿透 |
| ABAC策略引擎 | 策略链逐条匹配 | 解释器模式 + 组合模式 | 支持复杂条件组合和动态策略 |

### 9.2 算法复杂度汇总

| 操作 | 时间复杂度 | 说明 |
|------|-----------|------|
| 权限树构建 | O(N) | N为权限总数，两次遍历HashMap法 |
| 权限校验（有缓存） | O(1) | HashSet.contains操作 |
| 权限校验（无缓存） | O(R*P) | R为角色数，P为平均权限数 |
| 角色继承链遍历 | O(D*P) | D为继承深度，P为每层权限数 |
| 数据权限SQL生成 | O(K) | K为部门树节点数 |
| 灰度百分比判断 | O(1) | 哈希取模运算 |
| 布隆过滤器判断 | O(K) | K为哈希函数个数（通常为3~5） |
| 递归权限路径校验 | O(D) | D为树深度（通常不超过5层） |

### 9.3 常见问题与解决方案

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 权限变更不生效 | 缓存未及时清除 | 清除缓存 + Pub/Sub广播通知所有节点 |
| 数据权限绕过 | 某些接口忘记加@DataPermission | 统一MyBatis拦截器兜底 + 代码审查 |
| 缓存雪崩 | 大量权限缓存同时过期 | 随机化TTL（延迟生效策略） |
| 缓存穿透 | 查询不存在的用户权限 | 布隆过滤器 + 缓存空结果 |
| 角色继承循环 | 误操作导致A继承B、B继承A | DFS visited集合检测 |
| 超级管理员硬编码 | if(userId==1)判断超管 | 使用权限编码"*"标识超管 |
| 权限粒度过细 | 每个按钮一个权限编码 | 按功能模块划分，合理抽象 |
| 数据权限SQL性能差 | IN子查询部门ID过多 | 预计算部门列表 + ancestors字段优化 |
| 分布式环境本地缓存不一致 | 单机更新无法通知其他机器 | Redis Pub/Sub + L1短TTL兜底 |
| 权限变更无法追溯 | 没有操作日志 | 所有变更操作记录审计日志 |

#### 10.1.2 网关层：Token 解析与身份透传

网关是权限校验的第一道关卡，职责是"认证"（你是谁），把无状态的 JWT 解析为可信的用户身份，透传给下游业务服务；下游只做"鉴权"（你能干什么），无需重复验签。

```java
/**
 * 网关全局过滤器：JWT 解析 + 身份透传
 *
 * 设计模式：责任链模式（Spring Cloud Gateway 的 GlobalFilter 本身即过滤器链）
 *
 * 职责边界：
 *   网关只负责"认证"——验证 Token 合法性，解析出 userId/roles，透传给下游；
 *   不负责"鉴权"——具体接口需要什么权限由业务服务判断，避免网关与业务耦合。
 *
 * 关键设计：
 *   1. 白名单放行（登录、健康检查等无需 Token）
 *   2. 验签 + 过期校验 + 登出黑名单校验（多层防御）
 *   3. traceId 贯穿全链路，日志可串联
 *   4. 解析结果通过请求头 X-User-Id / X-User-Roles 透传，下游不再解析 Token
 */
@Component
@Slf4j
public class AuthGatewayFilter implements GlobalFilter, Ordered {

    @Autowired
    private TokenService tokenService;

    @Autowired
    private StringRedisTemplate redisTemplate;

    /** 登出黑名单 key 前缀：token 登出后写入，TTL 与 token 剩余有效期一致 */
    private static final String TOKEN_BLACKLIST_PREFIX = "auth:blacklist:";

    /** 网关白名单：无需认证的路径 */
    private static final Set<String> WHITE_LIST = Set.of(
        "/api/auth/login", "/api/auth/refresh", "/api/health"
    );

    private static final AntPathMatcher PATH_MATCHER = new AntPathMatcher();

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        ServerHttpRequest request = exchange.getRequest();
        String path = request.getURI().getPath();

        // Step 0：生成全链路 traceId，贯穿网关与下游服务日志
        String traceId = request.getHeaders().getFirst("X-Trace-Id");
        if (traceId == null || traceId.isEmpty()) {
            traceId = UUID.randomUUID().toString().replace("-", "");
        }
        MDC.put("traceId", traceId);

        try {
            // Step 1：白名单放行
            if (isWhiteListed(path)) {
                log.info("[网关认证] traceId={} path={} 命中白名单，放行", traceId, path);
                return chain.filter(mutateTraceId(exchange, traceId));
            }

            // Step 2：提取 Authorization 头
            String authHeader = request.getHeaders().getFirst(HttpHeaders.AUTHORIZATION);
            if (authHeader == null || !authHeader.startsWith("Bearer ")) {
                log.warn("[网关认证] traceId={} path={} 缺少 Bearer Token", traceId, path);
                return unauthorized(exchange, "未提供认证令牌");
            }
            String token = authHeader.substring(7);

            // Step 3：验签 + 过期校验（TokenService 内部完成 JWT 解析）
            TokenPayload payload;
            try {
                payload = tokenService.parseAndVerify(token);
            } catch (ExpiredJwtException e) {
                log.warn("[网关认证] traceId={} token 已过期 userId={}",
                    traceId, e.getClaims().getSubject());
                return unauthorized(exchange, "登录已过期，请重新登录");
            } catch (JwtException | IllegalArgumentException e) {
                // 验签失败可能是伪造 token，记录 warn 级别便于安全审计
                log.warn("[网关认证] traceId={} token 非法：{}", traceId, e.getMessage());
                return unauthorized(exchange, "认证令牌无效");
            }

            // Step 4：登出黑名单校验（防止已登出的 token 在有效期内被复用）
            String jti = payload.getJti();
            Boolean blacklisted = redisTemplate.hasKey(TOKEN_BLACKLIST_PREFIX + jti);
            if (Boolean.TRUE.equals(blacklisted)) {
                log.warn("[网关认证] traceId={} token 已登出 jti={} userId={}",
                    traceId, jti, payload.getUserId());
                return unauthorized(exchange, "登录状态已失效");
            }

            // Step 5：解析成功，将身份透传给下游（下游无需再解析 token）
            log.info("[网关认证] traceId={} userId={} roles={} 认证通过，转发至 {}",
                traceId, payload.getUserId(), payload.getRoles(), path);

            ServerHttpRequest mutated = request.mutate()
                .header("X-Trace-Id", traceId)
                .header("X-User-Id", String.valueOf(payload.getUserId()))
                .header("X-User-Roles", String.join(",", payload.getRoles()))
                .build();
            return chain.filter(exchange.mutate().request(mutated).build());

        } finally {
            // 响应式链路中 MDC 清理需谨慎，此处示意；生产建议用 Reactor Context 传递 traceId
            MDC.remove("traceId");
        }
    }

    private boolean isWhiteListed(String path) {
        return WHITE_LIST.stream().anyMatch(p -> PATH_MATCHER.match(p, path));
    }

    private ServerWebExchange mutateTraceId(ServerWebExchange exchange, String traceId) {
        return exchange.mutate()
            .request(exchange.getRequest().mutate().header("X-Trace-Id", traceId).build())
            .build();
    }

    /**
     * 统一返回 401，响应体为标准 Result 结构
     */
    private Mono<Void> unauthorized(ServerWebExchange exchange, String msg) {
        ServerHttpResponse response = exchange.getResponse();
        response.setStatusCode(HttpStatus.UNAUTHORIZED);
        response.getHeaders().setContentType(MediaType.APPLICATION_JSON);
        String body = JSON.toJSONString(Result.fail(401, msg));
        DataBuffer buffer = response.bufferFactory()
            .wrap(body.getBytes(StandardCharsets.UTF_8));
        return response.writeWith(Mono.just(buffer));
    }

    @Override
    public int getOrder() {
        return -100; // 优先级高于限流、日志等过滤器
    }
}

/**
 * Token 载荷（JWT 解析后的结构化结果）
 */
@Data
public class TokenPayload {
    private Long userId;
    private List<String> roles;   // 角色编码列表，如 ["operator","viewer"]
    private String jti;           // JWT ID，用于登出黑名单
    private long expireAt;        // 过期时间戳（毫秒）
}
```

#### 10.1.3 业务服务层：RBAC 权限匹配

请求穿过网关后带着 `X-User-Id` 到达业务服务。第七章的 `PermissionInterceptor` + 校验链负责鉴权。本案例给出一个更贴近实战、带完整日志与异常兜底的鉴权处理器实现，聚焦"提取角色 → RBAC 权限匹配 → 放行/拒绝"这一段。

```java
/**
 * RBAC 功能鉴权处理器（案例一实战版）
 *
 * 相比第 7.2 节的 AuthorizationHandler，本实现补充了：
 *   1. 从网关透传的请求头读取 userId，避免二次解析 token
 *   2. 完整的 traceId 日志埋点（准入/命中/拒绝三态）
 *   3. 缓存降级：缓存异常时回源 DB，保证可用性
 *   4. 鉴权失败写入访问审计日志（安全合规要求）
 *
 * 设计模式：责任链模式（继承 PermissionCheckHandler）+ 策略模式（超管/白名单/普通用户分支）
 */
@Component
@Slf4j
public class RbacAuthorizationHandler extends PermissionCheckHandler {

    @Autowired
    private PermissionCacheService cacheService;

    @Autowired
    private AccessAuditLogService auditLogService;

    /** 超级管理员权限通配符 */
    private static final String SUPER_ADMIN = "*";

    @Override
    protected PermissionCheckResult doCheck(PermissionCheckContext context) {
        long start = System.currentTimeMillis();
        Long userId = context.getUserId();
        String permCode = context.getPermCode();
        String uri = context.getRequestUri();

        // 接口未标注权限编码 → 视为登录即可访问，直接放行
        if (permCode == null || permCode.isEmpty()) {
            log.debug("[RBAC鉴权] userId={} uri={} 无需权限编码，放行", userId, uri);
            return PermissionCheckResult.pass();
        }

        Set<String> permissions;
        try {
            // 走二级缓存（L1 Caffeine → L2 Redis → DB），正常路径 O(1)
            permissions = cacheService.getUserPermissions(userId);
        } catch (Exception e) {
            // 缓存/Redis 异常时不能直接放行（越权风险），也不能直接拒绝（可用性）
            // 折中策略：记录 error，回源 DB 兜底；DB 也失败则拒绝
            log.error("[RBAC鉴权] userId={} 读取权限缓存异常，尝试回源DB", userId, e);
            try {
                permissions = cacheService.reloadFromDbWithoutCache(userId);
            } catch (Exception dbEx) {
                log.error("[RBAC鉴权] userId={} 回源DB仍失败，拒绝请求", userId, dbEx);
                return PermissionCheckResult.fail(503, "权限服务暂时不可用，请稍后重试");
            }
        }

        context.setPermissions(permissions);

        // 分支一：超级管理员放行
        if (permissions.contains(SUPER_ADMIN)) {
            log.info("[RBAC鉴权] userId={} 超级管理员放行 uri={} cost={}ms",
                userId, uri, System.currentTimeMillis() - start);
            return PermissionCheckResult.pass();
        }

        // 分支二：RBAC 权限匹配（HashSet.contains，O(1)）
        if (permissions.contains(permCode)) {
            log.info("[RBAC鉴权] userId={} 命中权限 [{}] 放行 uri={} cost={}ms",
                userId, permCode, uri, System.currentTimeMillis() - start);
            return PermissionCheckResult.pass();
        }

        // 分支三：无权限 → 拒绝 + 写审计日志（安全事件需要可追溯）
        log.warn("[RBAC鉴权] userId={} 缺少权限 [{}] 拒绝访问 uri={} 持有权限数={}",
            userId, permCode, uri, permissions.size());
        auditLogService.recordDenied(userId, uri, permCode);
        return PermissionCheckResult.fail(403, "无权限访问：" + permCode);
    }
}

/**
 * 访问审计日志服务：记录被拒绝的访问，用于安全审计和异常行为检测
 */
@Service
@Slf4j
public class AccessAuditLogService {

    @Autowired
    private AccessDeniedLogMapper deniedLogMapper;

    /**
     * 异步记录，避免拖慢主链路
     * 审计日志失败不能影响鉴权结果，因此内部吞掉异常仅记录 error
     */
    @Async("auditLogExecutor")
    public void recordDenied(Long userId, String uri, String permCode) {
        try {
            AccessDeniedLog record = new AccessDeniedLog();
            record.setUserId(userId);
            record.setRequestUri(uri);
            record.setRequiredPerm(permCode);
            record.setDeniedTime(LocalDateTime.now());
            record.setTraceId(MDC.get("traceId"));
            deniedLogMapper.insert(record);
        } catch (Exception e) {
            log.error("[审计日志] 记录越权访问失败 userId={} uri={}", userId, uri, e);
        }
    }
}
```

**关于本案例的幂等性**：读请求（`GET /api/order/list`）天然幂等，无需额外幂等控制——同一 Token 重复校验多次结果一致，且校验过程不产生副作用。真正需要幂等控制的是写操作，见案例二。

#### 10.2.2 幂等控制设计

管理员操作最怕"网络抖动导致前端重试"或"用户手抖双击"造成重复创建角色、重复插关联。本系统采用**请求去重表（唯一索引）+ 关联表唯一键**双重保障：

```sql

    id BIGINT PRIMARY KEY AUTO_INCREMENT,

    result_ref VARCHAR(128) COMMENT '首次执行的结果引用（如生成的 roleId）',
```

- `sys_role.uk_role_code`：角色编码唯一，防止同编码角色被重复创建
- `sys_role_permission.uk_role_perm`、`sys_user_role.uk_user_role`：关联表唯一键，重复插入直接被 DB 拦截
- `sys_idempotent_record.uk_request_id`：请求级幂等，重复 requestId 直接返回首次结果

#### 10.2.4 Service 层：事务 + 幂等 + 缓存更新 + 广播

```java
/**
 * 角色配置服务
 *
 * 核心链路：幂等校验 → 创建角色 → 绑定权限 → 分配用户 → 写审计 → 提交事务 → 清缓存 → 广播
 *
 * 关键设计：
 *   1. @Transactional 保证前 5 步的数据库操作原子性，任一失败整体回滚
 *   2. 幂等：requestId 唯一索引兜底，捕获 DuplicateKeyException 转为业务语义
 *   3. 缓存清除与广播放在事务提交之后（TransactionSynchronization），
 *      避免"事务未提交就清缓存 → 其他请求回源读到旧数据 → 又把旧数据写回缓存"的经典时序问题
 *
 * 设计模式：模板方法（固定编排流程）+ 观察者（广播通知其他节点）
 */
@Service
@Slf4j
public class RoleConfigService {

    @Autowired
    private SysRoleMapper roleMapper;
    @Autowired
    private SysRolePermissionMapper rolePermMapper;
    @Autowired
    private SysUserRoleMapper userRoleMapper;
    @Autowired
    private SysIdempotentRecordMapper idempotentMapper;
    @Autowired
    private PermissionChangeLogMapper changeLogMapper;
    @Autowired
    private PermissionCacheService cacheService;
    @Autowired
    private PermissionChangeBroadcaster broadcaster;

    private static final String BIZ_TYPE = "ROLE_CONFIG";

    @Transactional(rollbackFor = Exception.class)
    public Long configRole(RoleConfigRequest req, Long operatorId) {
        // Step 1：幂等校验——先查是否已处理过该 requestId
        SysIdempotentRecord existed = idempotentMapper.selectByRequestId(req.getRequestId());
        if (existed != null) {
            throw new DuplicateRequestException(existed.getResultRef());
        }

        // Step 2：业务前置校验——角色编码是否已存在
        SysRole sameCode = roleMapper.selectByRoleCode(req.getRoleCode());
        if (sameCode != null) {
            throw new RoleConfigException("角色编码已存在：" + req.getRoleCode());
        }

        // Step 3：创建角色
        SysRole role = new SysRole();
        role.setRoleId(IdGenerator.nextId());
        role.setRoleCode(req.getRoleCode());
        role.setRoleName(req.getRoleName());
        role.setDataScope(req.getDataScope());
        role.setStatus(1);
        roleMapper.insert(role);
        Long roleId = role.getRoleId();
        log.info("[角色配置] 创建角色成功 roleId={} code={}", roleId, req.getRoleCode());

        // Step 4：绑定权限（关联表唯一键保证幂等，去重后批量插入）
        if (req.getPermIds() != null && !req.getPermIds().isEmpty()) {
            List<SysRolePermission> rpList = req.getPermIds().stream()
                .distinct()
                .map(permId -> {
                    SysRolePermission rp = new SysRolePermission();
                    rp.setRoleId(roleId);
                    rp.setPermId(permId);
                    return rp;
                })
                .collect(Collectors.toList());
            rolePermMapper.batchInsert(rpList);
            log.info("[角色配置] roleId={} 绑定权限 {} 个", roleId, rpList.size());
        }

        // Step 5：分配用户（同样去重）
        List<Long> affectedUserIds = new ArrayList<>();
        if (req.getUserIds() != null && !req.getUserIds().isEmpty()) {
            for (Long userId : new HashSet<>(req.getUserIds())) {
                SysUserRole ur = new SysUserRole();
                ur.setUserId(userId);
                ur.setRoleId(roleId);
                try {
                    userRoleMapper.insert(ur);
                    affectedUserIds.add(userId);
                } catch (DuplicateKeyException e) {
                    // 用户已拥有该角色，幂等忽略
                    log.info("[角色配置] userId={} 已拥有 roleId={}，跳过", userId, roleId);
                }
            }
        }

        // Step 6：写审计日志（谁在何时做了什么，安全合规必需）
        PermissionChangeLog auditLog = new PermissionChangeLog();
        auditLog.setAction("ROLE_CONFIG");
        auditLog.setTargetId1(roleId);
        auditLog.setTargetId2(null);
        auditLog.setOperatorId(operatorId);
        auditLog.setOperateTime(LocalDateTime.now());
        changeLogMapper.insert(auditLog);

        // Step 7：写幂等记录（唯一索引；并发下靠 DuplicateKeyException 兜底）
        try {
            SysIdempotentRecord record = new SysIdempotentRecord();
            record.setRequestId(req.getRequestId());
            record.setBizType(BIZ_TYPE);
            record.setResultRef(String.valueOf(roleId));
            record.setStatus(1);
            idempotentMapper.insert(record);
        } catch (DuplicateKeyException e) {
            // 并发重复提交：另一个线程已抢先写入，本事务回滚并转为幂等命中
            SysIdempotentRecord winner = idempotentMapper.selectByRequestId(req.getRequestId());
            throw new DuplicateRequestException(winner != null ? winner.getResultRef() : String.valueOf(roleId));
        }

        // Step 8：事务提交后再清缓存 + 广播（避免脏数据回填）
        registerAfterCommit(roleId, affectedUserIds);

        return roleId;
    }

    /**
     * 注册事务提交后的回调：清缓存 + 广播
     * 只有事务真正 COMMIT 后才执行，保证"缓存失效"与"数据落库"的正确时序
     */
    private void registerAfterCommit(Long roleId, List<Long> affectedUserIds) {
        if (!TransactionSynchronizationManager.isSynchronizationActive()) {
            // 兜底：无事务上下文时直接执行
            doInvalidateAndBroadcast(roleId, affectedUserIds);
            return;
        }
        TransactionSynchronizationManager.registerSynchronization(
            new TransactionSynchronization() {
                @Override
                public void afterCommit() {
                    doInvalidateAndBroadcast(roleId, affectedUserIds);
                }
            });
    }

    /**
     * 清缓存 + 广播；本身不能抛异常影响主流程（数据已提交），失败仅记录
     * 即使广播失败，各节点 L1 缓存的短 TTL（5min）也能最终一致
     */
    private void doInvalidateAndBroadcast(Long roleId, List<Long> affectedUserIds) {
        try {
            // 清除本节点：新角色的用户缓存
            for (Long userId : affectedUserIds) {
                cacheService.clearUserPermissionCache(userId);
            }
            // 广播：通知集群其他节点清除本地 L1 缓存
            broadcaster.publishChange("USER_ROLE_CHANGE", roleId);
            log.info("[角色配置] roleId={} 缓存已失效并广播，影响用户 {} 人",
                roleId, affectedUserIds.size());
        } catch (Exception e) {
            log.error("[角色配置] roleId={} 缓存失效/广播失败，将由TTL兜底", roleId, e);
        }
    }
}
```

**实时生效原理**：事务提交后 `clearUserPermissionCache` 清除了张三在本节点的 L1+L2 缓存，`publishChange` 通过 Redis Pub/Sub 通知其他节点清 L1（见 6.3 节的 `PermissionChangeBroadcaster`）。张三下一次请求走到案例一的 `getUserPermissions(1001)` 时，L1/L2 均 miss，回源 DB 加载到含 `order:view`、`order:export` 的新权限集合并回填缓存，新权限即刻生效，全程无需重启服务。

#### 10.2.5 幂等与异常矩阵

| 场景 | 触发条件 | 处理方式 | 结果 |
|------|---------|----------|------|
| 前端重复提交 | 相同 requestId | 幂等表命中，抛 DuplicateRequestException | 返回首次 roleId，无脏数据 |
| 并发双击 | 同 requestId 并发 | 唯一索引 DuplicateKeyException 兜底 | 一个成功，另一个转幂等命中 |
| 角色编码冲突 | roleCode 已存在 | RoleConfigException | 400 提示编码已存在 |
| 用户重复分配 | user_role 已存在 | 捕获 DuplicateKeyException 忽略 | 幂等跳过，不报错 |
| 中途 DB 异常 | 任一 insert 失败 | @Transactional 整体回滚 | 无部分写入，500 |
| 缓存/广播失败 | 事务已提交后 Redis 挂 | 记录 error，TTL 兜底 | 数据已生效，最终一致 |

### 10.3 案例三：数据权限控制全链路

**业务背景**：区域经理李四（userId=2002，dept_id=200，角色数据范围="本部门及以下"）调用 `GET /api/order/list` 查询订单。功能鉴权（案例一）已通过——他有 `order:view` 权限；但"能看订单"不代表"能看所有部门的订单"。数据权限层要保证他**只能看到 200 部门及其子部门的订单**，且这一过滤对业务 SQL 完全无侵入。

#### 10.3.1 全链路时序

#### 10.3.2 行级权限计算（复用 DataScopeService）

数据权限的"计算"部分复用 5.2 节的 `DataScopeService.buildDataScopeSql`，它按角色的 `dataScope` 生成 SQL 片段。本案例李四是"本部门及以下"，`DataScopeService` 会通过 `ancestors` 字段一次查出子部门，生成形如 `t.dept_id IN (200,201,202)` 的条件。校验链的 `DataScopeHandler` 把该片段放入 `DataScopeContextHolder`（ThreadLocal）供拦截器读取——这一段在 7.2 节已给出，此处不再重复。

> 关键点：数据权限**不阻断请求**（返回 pass），它只"收窄"可见数据范围。即便计算结果为空集，也应生成 `1=0` 这类永假条件，让 SQL 查到 0 行，而不是抛异常或放行全部数据。

#### 10.3.3 SQL 自动拼接（实战版拦截器）

5.3 节给出了拦截器骨架。本案例补充生产级细节：**从 ThreadLocal 读取已算好的条件（避免拦截器内重复查库）、多表/子查询的别名安全、写操作跳过、异常降级与日志**。

```java
/**
 * 数据权限拦截器（案例三实战版）
 *
 * 相比 5.3 节，本实现强调：
 *   1. 优先从 DataScopeContextHolder 读取校验链已算好的 SQL 片段，
 *      避免拦截器内再查一次库（校验链 → 拦截器 是同一请求线程，ThreadLocal 可共享）
 *   2. 仅拦截 SELECT，写操作（INSERT/UPDATE/DELETE）不注入行级过滤
 *   3. JSqlParser 改写失败时降级为拒绝（宁可查不到，也不能越权看全量）
 *   4. 完整 traceId 日志，记录原始 SQL 与改写后 SQL 便于排查
 *
 * 设计模式：拦截器模式 + 责任链（MyBatis plugin 链）
 */
@Intercepts({
    @Signature(type = StatementHandler.class, method = "prepare",
               args = {Connection.class, Integer.class})
})
@Component
@Slf4j
public class DataPermissionInterceptorV2 implements Interceptor {

    @Override
    public Object intercept(Invocation invocation) throws Throwable {
        StatementHandler handler = (StatementHandler) invocation.getTarget();
        MetaObject metaObject = SystemMetaObject.forObject(handler);
        MappedStatement ms = (MappedStatement) metaObject.getValue("delegate.mappedStatement");

        // 仅处理 SELECT，写操作不做行级过滤
        if (ms.getSqlCommandType() != SqlCommandType.SELECT) {
            return invocation.proceed();
        }

        // 未标注 @DataPermission 的查询直接放行
        DataPermission annotation = resolveAnnotation(ms);
        if (annotation == null) {
            return invocation.proceed();
        }

        // 优先取校验链算好的条件；ThreadLocal 为空则视为"全部可见"（如超管），放行
        String condition = DataScopeContextHolder.getDataScopeSql();
        if (condition == null) {
            log.debug("[数据权限] traceId={} 无数据范围限制（全部可见），放行", MDC.get("traceId"));
            return invocation.proceed();
        }

        BoundSql boundSql = handler.getBoundSql();
        String originalSql = boundSql.getSql();
        String newSql;
        try {
            newSql = appendCondition(originalSql, condition);
        } catch (Exception e) {
            // 改写失败时降级为永假条件，宁可查 0 行也不能越权返回全量数据
            log.error("[数据权限] traceId={} SQL改写失败，降级为1=0。原SQL={}",
                MDC.get("traceId"), originalSql, e);
            newSql = safeFallback(originalSql);
        }

        log.info("[数据权限] traceId={} mapper={} 注入条件[{}]",
            MDC.get("traceId"), ms.getId(), condition);
        log.debug("[数据权限] 改写后SQL={}", newSql);

        metaObject.setValue("delegate.boundSql.sql", newSql);
        return invocation.proceed();
    }

    /**
     * 用 JSqlParser 将条件 AND 进 WHERE；解析失败抛异常交由上层降级
     */
    private String appendCondition(String originalSql, String condition) throws JSQLParserException {
        Statement statement = CCJSqlParserUtil.parse(originalSql);
        if (!(statement instanceof Select)) {
            return originalSql;
        }
        Select select = (Select) statement;
        PlainSelect plainSelect = (PlainSelect) select.getSelectBody();
        Expression where = plainSelect.getWhere();
        Expression extra = CCJSqlParserUtil.parseCondExpression(condition);
        plainSelect.setWhere(where == null ? extra : new AndExpression(where, extra));
        return select.toString();
    }

    /**
     * 降级：在 SQL 末尾安全追加永假条件
     */
    private String safeFallback(String originalSql) {
        String upper = originalSql.toUpperCase();
        return upper.contains(" WHERE ") ? originalSql + " AND 1=0" : originalSql + " WHERE 1=0";
    }

    private DataPermission resolveAnnotation(MappedStatement ms) {
        try {
            String id = ms.getId();
            String className = id.substring(0, id.lastIndexOf('.'));
            String methodName = id.substring(id.lastIndexOf('.') + 1);
            Class<?> clazz = Class.forName(className);
            for (Method m : clazz.getDeclaredMethods()) {
                if (m.getName().equals(methodName)) {
                    return m.getAnnotation(DataPermission.class);
                }
            }
        } catch (Exception e) {
            log.warn("[数据权限] 解析注解失败 ms={}", ms.getId(), e);
        }
        return null;
    }

    @Override
    public Object plugin(Object target) {
        return Plugin.wrap(target, this);
    }

    @Override
    public void setProperties(Properties properties) {
    }
}
```

#### 10.3.4 数据隔离效果与 ThreadLocal 生命周期

李四的原始业务 SQL 与改写后 SQL 对比：

```sql
-- 业务代码写的原始 SQL（Mapper 中，@DataPermission(deptAlias="t")）
SELECT t.* FROM t_order t WHERE t.status = 1;

-- 拦截器改写后实际执行的 SQL（自动追加数据范围）
SELECT t.* FROM t_order t WHERE t.status = 1 AND t.dept_id IN (200, 201, 202);
```

不同用户查询同一接口，因数据范围不同而看到不同数据，实现了**行级数据隔离**：

| 用户 | 角色数据范围 | 注入的 WHERE 条件 | 可见数据 |
|------|-------------|-------------------|----------|
| 总监（dataScope=全部） | ALL | 不注入条件 | 所有部门订单 |
| 李四（本部门及以下） | DEPT_AND_CHILDREN | `t.dept_id IN (200,201,202)` | 200 及子部门订单 |
| 王五（本部门） | DEPT | `t.dept_id = 300` | 仅 300 部门订单 |
| 张三（仅本人） | SELF | `t.creator = 1001` | 仅自己创建的订单 |
| 无角色用户 | — | `1=0` | 0 行 |

**ThreadLocal 泄漏防护（幂等/安全的关键）**：`DataScopeContextHolder` 基于 ThreadLocal，在 Tomcat/线程池复用线程的环境下，若请求结束不清理，下一个请求可能读到上一个用户的数据范围，造成**越权串数据**这一严重安全事故。因此必须在请求结束时清理（7.3 节 `PermissionInterceptor.afterCompletion` 已调用 `DataScopeContextHolder.clear()`）。

```java
/**
 * 兜底清理：即使业务抛异常，也保证 ThreadLocal 被清空
 * 生产建议同时在 afterCompletion 与 finally 双重清理，防止串数据
 */
public class DataScopeCleanupFilter extends OncePerRequestFilter {
    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain chain) throws ServletException, IOException {
        try {
            chain.doFilter(request, response);
        } finally {
            // 无论正常返回还是异常，都清理，防止线程复用导致越权
            DataScopeContextHolder.clear();
            SecurityContextHolder.clear();
        }
    }
}
```

### 10.4 三案例串联总览

三个案例在一次"李四查订单"的完整请求中依次协作：

```
┌──────────────────────────────────────────────────────────────────────────────┐
│  一次请求 GET /api/order/list （李四，携带 JWT）                              │
├──────────────────────────────────────────────────────────────────────────────┤
│  [案例一] 认证鉴权                                                            │
│    网关解析 Token → 透传 X-User-Id=2002 → RBAC 匹配 order:view → 放行         │
│                              │                                                │
│                              ▼                                                │
│  [案例二] 前置：管理员早已配置好李四的角色与数据范围（本部门及以下）           │
│    角色权限来自缓存（案例二配置后已实时生效）                                  │
│                              │                                                │
│                              ▼                                                │
│  [案例三] 数据权限                                                            │
│    DataScopeHandler 生成 dept_id IN(200,201,202) → 存 ThreadLocal            │
│    → MyBatis 拦截器改写 SQL → DB 仅返回本部门及以下订单                       │
│                              │                                                │
│                              ▼                                                │
│    afterCompletion 清理 ThreadLocal → 返回 200 + 隔离后的数据                 │
└──────────────────────────────────────────────────────────────────────────────┘
```

**三案例的横切关注点小结**：

| 关注点 | 案例一 | 案例二 | 案例三 |
|--------|--------|--------|--------|
| 异常处理 | 401/403/503 分级，验签失败 warn | 事务回滚、业务异常转 4xx | 改写失败降级 1=0 |
| 日志记录 | traceId 贯穿网关+服务，准入/拒绝三态 | operatorId+requestId 审计日志 | traceId + 原始/改写 SQL |
| 幂等控制 | 读请求天然幂等 | requestId 唯一索引 + 关联表唯一键双保险 | 查询无副作用，天然幂等 |
| 实时生效 | 缓存 O(1) 命中 | 事务后清缓存+Pub/Sub 广播 | ThreadLocal 请求级即时生效 |
| 安全兜底 | 缓存异常不放行、回源 DB | 事务后清缓存避免脏回填 | ThreadLocal 强制清理防越权串数据 |
